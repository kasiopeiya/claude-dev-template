`grep`の結果から、各スキルの該当箇所を確認しました。結論から言うと、**これは「不要」ではなく、むしろ意図的に必要な設定**だと考えます。理由を説明します。

## `disable-model-invocation: true` の意味

この設定は「**Claude（モデル）が自動でこのスキルを起動することを禁止する**」というものです。設定すると、ユーザーが `/design` のようにスラッシュコマンドで明示的に呼んだときだけ起動し、Claudeが会話の文脈から勝手に起動することはなくなります。

実際、その証拠が今のセッションにも出ています。設定されている8スキルは、**私（Claude）に見えている「自動起動可能なスキル一覧」に含まれていません**：

| 設定あり（自動起動不可）                                                                | 一覧での扱い                                     |
| --------------------------------------------------------------------------------------- | ------------------------------------------------ |
| design / code-dev / cdk-dev / cdk-imp / tdd / update-design / unit-test / cdk-visualize | 私のスキル一覧に**出てこない**（＝ユーザー専用） |

一方 `check-plan` / `to-plan` / `quick-issue` などは設定がないので一覧に出ており、私が自動起動できます。つまり設定は**正しく機能しています**。

## なぜ必要か

設定されている8スキルの顔ぶれを見ると、すべて以下のいずれかです：

- **自律実行系オーケストレーション**（design / code-dev / cdk-dev / cdk-imp / tdd）
  → descriptionに「人間への確認なしに自律実行」とある通り、実装→レビュー→CI→修正を一気通貫で走らせる重い処理
- **Issue番号を引数に取る、開発フローの特定ステップ専用**（CLAUDE.mdのStep 4〜7に対応）

これらをモデルが自動起動できてしまうと、ユーザーが「ちょっと相談」しただけで**勝手にTDDサイクルやCDKデプロイ準備の全工程が走り出す**リスクがあります。CLAUDE.mdの開発フローは「各段階で人間がスラッシュコマンドを実行する」設計なので、`disable-model-invocation: true` はこの設計思想と一致しており、安全装置として機能しています。

## 矛盾点として残っている問題

ただし1点、**descriptionとの矛盾**があります。たとえば `cdk-imp` の説明は「CDKインフラの実装を依頼されたときに使用すること」と、**自動起動を促す文面**になっているのに、実際は自動起動が禁止されています。これは読み手を混乱させます。

## 推奨

| 対象             | 推奨                                                                                                                                                  |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| **設定そのもの** | **残す**（安全装置として妥当）                                                                                                                        |
| **description**  | 「〜依頼されたときに使用」という自動起動を促す文言を、「`/cdk-imp <Issue番号>` で明示的に実行する」のような**ユーザー起動前提の文面**に直すと整合する |
