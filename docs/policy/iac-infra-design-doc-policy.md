# IaCインフラ設計書作成ポリシー

AWS/IaC で構築したシステムの**インフラ設計書**を書く・レビューする人が、コードから読み取れない全体像・設計判断・組織/運用上の制約を過不足なく書けているかを判断する基準を得たいとき参照するドキュメント。読み終えたとき「良いインフラ設計書とはこういうもの」を自分の言葉で言え、テンプレートに沿って着手できることを目的とする。

> [!IMPORTANT]
> **TL;DR（このポリシーの決定事項）**
> - IaC ではコード自体がインフラの最大の SSoT。ゆえにインフラ設計書は通常の設計書よりさらに**「全体を見渡す図」と「なぜ（設計判断）」に寄る**。具体パラメータ値（インスタンスサイズ・バケット名等）は書かない（コードと二重管理になり腐る）
> - **必須セクション**をテンプレートとして定める。必須とは「セクションを必ず**存在**させる」の意で、該当がなければ**「なし＋理由」で閉じてよい**（不在が「未検討」なのか「対象外」なのかを読者に伝えるため）
> - CDK **コード**の書き方は [cdk-policy](cdk-policy.md) / [.claude/rules/cdk.md](../../.claude/rules/cdk.md) の領分。本ポリシーは**文書**の書き方だけを定める

## 前提：IaC では「コードが最大の SSoT」だから、設計書はいっそう図とWHYに寄る

インフラも、構成そのものはコードを追わないと頭の中で再構築できない。しかも IaC ではコードが宣言的なインフラ定義そのものなので、**コードを読めば分かること（リソースの個別設定・具体値）を設計書に書き写すと、必ずコードと二重管理になって腐る**。

したがってインフラ設計書が担うのは、コードからは読み取れない次の2種類だけである。

| 設計書が担うもの | なぜコードから読めないか |
|---|---|
| **全体を見渡す図**（構成・関係・流れ） | 構成は多数のファイル・スタックに散らばり、一望できない |
| **設計判断とその理由（WHY）** | 「なぜこの構成／このサービスか」「なぜこう分割したか」はコードに現れない |

> [!IMPORTANT]
> **（AI・必須）** 各記述に「これはコードを読めば分かるか？」を問う。分かるなら書かない。設計書の主役は**図**であり、削るときは文章から先に削って図を最後に残す。図は [`/design-doc-mermaid`](../../CLAUDE.md) スキルで作成する（自前で Mermaid を書き起こさない）。

## 書かないもの：具体パラメータ値

インスタンスサイズ・バケット名・タイムアウト値などの**具体パラメータ値は書かない**。設計書とコードの二重管理になり、乖離の原因になる。ここはコードが SSoT のままでよい部分である。

- 例外：**設計判断を支える根拠として値に触れる**のは可（例「同時実行が想定 100 未満のためスケールアウトは不要」）。この場合も値そのものではなく**なぜその値で足りるか**を書く。

## 境界：CDK コードの書き方とは別物

| 対象 | 何を定めるか | どこ |
|---|---|---|
| インフラ設計**書**（文書） | 全体像の図・設計判断・組織/運用の制約をどう書くか | **本ポリシー** |
| CDK **コード**の設計 | リソース名の固定方針・Stack 抽象化など、コードの作り方 | [cdk-policy](cdk-policy.md) |
| CDK **コード**の書式 | import 形式・Construct レベルなど機械的ルール | [.claude/rules/cdk.md](../../.claude/rules/cdk.md) |

例えば「命名規約」は、**規約とその理由**を設計書に記録し（本ポリシー）、その**実装**はコード側ルールに従う——という役割分担になる。

## 必須セクション（テンプレート）

インフラ設計書は次のセクションで構成する。**すべて必須**だが、該当がなければ本文に「**なし**」＋一言理由を書いて閉じてよい（空欄で放置しない）。各図の下には必ず**設計判断とその理由**を添える。

セクションの並び順・番号は縛らない（下表の順序は一例）。

| セクション | 何を書くか（コードから読めないこと） | 該当なし時の書き方 |
|---|---|---|
| **# アーキテクチャ概要** | 下位の図群と、その設計判断 | — |
| 　全体インフラ構成図 | システム全体の主要リソースと接続 | （通常は必ず該当あり） |
| 　アカウントと環境構成図 | AWS アカウントと環境（例：dev/stg/prd）の関係。1アカウントに複数環境を同居させるか、環境間の主要な違い | 単一アカウント単一環境なら、その旨と理由 |
| 　ネットワーク構成図 | VPC・サブネット・VPC Endpoint などネットワーク境界 | VPC を使わない構成なら「なし・理由：フルサーバーレスのため」 |
| 　設計判断とその理由 | 「なぜこの構成か」「なぜこのサービスを選び、どの代替案を却下したか」。各図の直下に配置 | — |
| **# IaC 管理方針** | どこまでを IaC で管理し、手動変更をどう扱うか（例：構築〜運用まで IaC、手動変更は禁止） | — |
| 　手動作成リソース一覧 | IaC 管理外のリソースを表で列挙（手動管理になるため） | 「なし」（全リソースが IaC 管理下であることの確認になる） |
| **# スタック設計** | 下位を含む | — |
| 　スタック関係図 | スタックの依存関係と参照方式（クロススタック参照／Parameter Store による間接参照 など） | 単一スタックなら、その旨 |
| 　スタック命名規約 | プレフィックス設計とその理由（プレフィックスはデプロイ時の対象選択などに使う） | 規約がなければ、その旨 |
| **# 組織の制約** | ガバナンス上のルール | — |
| 　命名／タグ規則 | 組織が課す命名・タグ付けの規約 | 「なし」 |
| 　その他の制約 | 暗号化要件（例：社内ルールで KMS 利用）・リージョン制約 など | 「なし」 |
| **# 設計上の重要ポイント** | 一般的でない・追いにくい・重要な設計の意図と落とし穴（例：なぜ親子構成のステートマシンか／なぜ RDS Proxy か） | 「特になし」 |
| **# コスト** | **コスト構造・主要コストドライバ・コスト観点の設計判断**（例：なぜ常時起動でなくサーバーレスか） | — |
| **# 運用監視（任意）** | 監視・ログ・トレースの**全体像を図**で。項目選定の考え方は監視ポリシーへ委譲 | 別設計書に切り出す場合は参照先を書く |

### スタック命名規約の書き方（補足）

命名スキームはプロジェクトごとに異なる。**規約とその理由**を記録し、スキーム自体は各プロジェクトで定義する（決め打ちしない）。

- スキーム例：`{システム}-{環境}-{ステート}-{任意}-stack`（一例）

### コストの書き方（補足）

月額などの**概算金額は「設計時点のスナップショット」**として補助的に書き、**保守対象にしない**（使用量・価格改定で必ず陳腐化するため）。維持すべきは金額ではなく、コスト構造・ドライバ・設計判断の側である。

### 運用監視の書き方（補足）

インフラ設計書側の役割は「**監視・ログ・トレースの全体像を図で渡す**」ことに限定する。**どういう考え方で監視項目・アラームを選ぶか**（症状ベース・GSM など）は [monitoring-policy](monitoring-policy.md) が単一の基準を持つので、ここで再説明せず参照する。運用監視は独立した設計書に切り出してもよい（その場合は本設計書から参照先を示す）。

## レビュー観点（チェックリスト）

インフラ設計書をレビューするときは次を確認する。

- [ ] 必須セクションがすべて存在するか（該当なしは「なし＋理由」で閉じているか）
- [ ] 主要な構成が**図**で示されているか（文章の壁になっていないか）
- [ ] 各図に**設計判断とその理由**が添えられ、**却下した代替案**にも触れているか
- [ ] コードを読めば分かる具体パラメータ値を**書き写していない**か（二重管理していないか）
- [ ] コストが「金額の羅列」でなく**構造・ドライバ・判断**を語っているか
- [ ] 手動作成リソースが漏れなく列挙されているか（なければ「なし」と明記されているか）

---

> [!IMPORTANT]
> 本ポリシーで最も重要な原則は次の3つ。
> - **コードが最大の SSoT**：設計書には全体像の図と設計判断（WHY）だけを書き、具体パラメータ値は書かない
> - **必須セクションは「存在」を必須にする**：該当なしは「なし＋理由」で閉じ、未検討と対象外を読者が区別できるようにする
> - **文書の話に限定する**：コードの作り方は cdk-policy / cdk.md、監視項目の選び方は monitoring-policy へ委譲する
