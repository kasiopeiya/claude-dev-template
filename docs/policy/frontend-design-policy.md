# フロントエンド設計ポリシー

React コンポーネント・カスタムフックを設計／レビューする人・AIが、UI レイヤー固有の設計判断（責務の分け方・状態のスコープ・Props の形・ビジュアルの作り方）の良し悪しを確かめたいときに参照する。

## 基盤との関係：application-design-policy を継承し、UI 固有の差分だけを足す

フロントエンドのコードも、まずアプリケーション設計の普遍原則（責務の単一性・正しく使うのが一番易しいインターフェース・引数＝責務の鏡）に従う。本ポリシーはそれらを再掲せず、**React／UI コンポーネント領域に固有の差分だけ**を上乗せする。コード内部設計の普遍原則は [application-design-policy.md](application-design-policy.md) を、判断に迷ったときの上位基準は [refined-engineer-judgment-principles.md](refined-engineer-judgment-principles.md) を参照する。実装の機械的ルール（JSX 記法・Hooks・Tailwind・Testing Library 戦術）は [.claude/rules/react.md](../../.claude/rules/react.md) に置く。

> [!IMPORTANT]
> **TL;DR（このポリシーの決定事項）**
>
> - コンポーネントの責務は1つ。表示とロジックを混ぜず、ロジックはカスタムフックへ逃がす。
> - 状態は使う場所に置く（colocation）。持ち上げ（lift up）・グローバル化は必要になってから。
> - ビジュアルデザインは自作しない。Tailwind utility-first で当て、見た目の作り込みは外部に委譲する。

## コンポーネントの責務は1つにする

**1つのコンポーネントは1つの責務だけを持つ。** これは SRP（[refined-engineer-judgment-principles 原則1](refined-engineer-judgment-principles.md)）を、[application-design-policy](application-design-policy.md) のコード内部設計を経由して UI に適用したもの。とりわけ**「見た目（表示）」と「振る舞い（ロジック・データ取得）」を1つのコンポーネントに混ぜない**。データ取得・状態遷移・計算といったロジックは**カスタムフックへ逃がし**、コンポーネント本体は「受け取った値をどう描くか」に集中させる。

| 分け方                     | 担うもの                                           |
| -------------------------- | -------------------------------------------------- |
| カスタムフック（`useXxx`） | データ取得・状態・計算・副作用（＝振る舞い）       |
| コンポーネント             | Props と hook の戻り値をどう描画するか（＝見た目） |

> [!TIP]
> **（AI・推奨）** 判定テスト：そのコンポーネントを「何をする部品か」を**接続詞なしの一文**で言えるか。「データを取得**して**一覧表示**して**フィルタも持つ」のように `して` で連なるなら、責務が割れている兆候。フックとコンポーネントに分ける。

## 状態は最小スコープに置く

**状態は、それを使う場所のできるだけ近くに置く（colocation）。** 持ち上げ（複数コンポーネントの共通祖先へ移す）も、Context やグローバル状態への昇格も、**実際に共有が必要になってから**行う。先回りでトップに置いた状態は、無関係な再レンダーと結合を生む。

これは [refined-engineer-judgment-principles の「決定を遅らせる」](refined-engineer-judgment-principles.md) と YAGNI の UI 適用である。「いずれ共有するかも」で先に持ち上げない——共有要求が**現れた瞬間**に持ち上げる。

| 状態の広がり                              | 既定の置き場所                                               |
| ----------------------------------------- | ------------------------------------------------------------ |
| 1コンポーネント内で完結                   | そのコンポーネントの `useState`（colocation）                |
| 親子・兄弟で共有が必要になった            | 共通祖先へ持ち上げる（lift state up）                        |
| 画面・機能を横断して共有                  | Context                                                      |
| 大規模・高頻度更新で Context が辛くなった | 専用ライブラリ（Redux/Zustand 等）を **ADR で判断** して導入 |

> [!WARNING]
> **（AI・必須）** 専用のグローバル状態ライブラリの導入は、依存とアーキテクチャを左右する後戻りしにくい決定なので、独断で入れない。デフォルトは「ローカル状態＋必要なら Context」とし、それで立ち行かない根拠が出てから ADR 化して判断する。

## Props はインターフェースである

**コンポーネントの Props は、その部品のインターフェースである。** [application-design-policy「正しく使うのが一番易しいインターフェースにせよ」](application-design-policy.md) がそのまま当てはまる——正しい使い方が最も簡単で、誤用が型で困難になっている Props を設計する。

とりわけ**真偽値・フラグ Props で振る舞いを切り替えない**（[ブーリアン引数アンチパターン](application-design-policy.md)）。`<Button primary danger />` のようなフラグの組み合わせは、無効な状態（`primary` かつ `danger`）を表現できてしまう。**見た目の選択肢は判別可能ユニオン（`variant: 'primary' | 'danger'`）にまとめ、無効な組み合わせをそもそも作れないようにする。**

これは application-design-policy が禁じる「振る舞いを切り替えるフラグ」とは別物である——コンポーネントの責務は「ボタンを描く」1つのままで、`variant` は**見た目（presentation）だけ**を選んでいるからだ。逆に、フラグによって**データ取得や処理そのものが変わる**なら、それは責務が割れているサインなのでコンポーネントを分ける。具体的な記法は [react.md](../../.claude/rules/react.md) に従う。

> [!TIP]
> **（AI・推奨）** 確認法は application-design-policy と同じ——**呼び出し側（JSX）を先に書く**。`<UserCard user={user} />` が注釈なしで読めるか、必須・任意が型で導かれるか、無効な Props 組み合わせが型で弾かれるかを、実装前に見る。

## ビジュアルデザインは自作せず委譲する

**見た目の作り込みは、このプロジェクトの設計判断の対象外とし、外部に委譲する。** 理由は明快で、ここでの価値は「動くアプリを成立させること」にあり、独自デザインシステムの構築に投資しないと決めているから。具体的には次を既定とする。

| 対象                   | 方針                                                                           |
| ---------------------- | ------------------------------------------------------------------------------ |
| スタイリング手段       | Tailwind の utility-first。独自 CSS の作り込み・デザインシステムの自作はしない |
| コンポーネントの見た目 | 既製のコンポーネントライブラリ、または AI 生成に委ねる                         |

> [!NOTE]
> どのコンポーネントライブラリ／生成ツールを採用するかは未確定。採用を決める際は、依存に関わる判断として ADR 化する。本ポリシーが定めるのは「**自作しない・Tailwind で当てる・委譲する**」という立場までである。

## テスト戦略は統合テストを厚くする（トロフィー型）

フロントエンドのテストは、コンポーネントを過度に分解した単体テストではなく、**ユーザー操作に対する表示結果を検証する統合寄りのテストを最も厚く**する。この判断軸と理由は [test-strategy-policy.md](test-strategy-policy.md)（テスティングトロフィー型）に、ユーザー視点で振る舞いを検証する具体指針は [unit-test-policy.md §9](unit-test-policy.md) に定義済みのため、ここでは繰り返さない。
