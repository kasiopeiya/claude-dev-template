# PR Review Policy

PRを出す・レビューする開発者が、コメントの書き方・マージ判断・PRのサイズ感について迷ったときに参照する指針。レビューで**何に目を向けるか（観点）**は [code-review-guide](../guide/code-review-guide.md) を参照。

> [!IMPORTANT]
> **TL;DR**
> - PRを出す側は「説明の記載」「CIの全成功」「小さいサイズ（原則200行以内）」を満たす。満たさないPRはレビュワーが拒否できる。
> - レビューコメントは `MUST` / `WANT` / `FYI` のラベルで意図を明確にする。`MUST` のみがマージをブロックする。
> - 変更の種類とレビュイーの経験でケース1〜3を使い分ける。すべてのPRにAIセルフレビューを実施する。

## なぜレビューするのか

コードレビューは、バグを見つけるためだけの作業ではない。

**知識の共有**
レビューを通じて、実装の背景・設計の意図・言語やフレームワークの慣用表現が自然にチーム内へ広まる。
ドキュメントに書かれない暗黙知を伝える場として機能する。

**品質の底上げ**
レビューがなければ、コードのスタイルや設計は自然と個人の習慣に収束する。
他者の目が入ることで、自分では気づかない思い込みや偏りが是正される。
「誰かに見せる」という前提があるだけで、コードを書く側の意識も変わる。

## PRを出す側が満たすべきレビューの前提条件

レビューはレビュワーの時間という有限な資源を消費する。前提が崩れたPRをレビューさせることは、その資源を本来不要な確認に浪費させる。レビュワーが本質的な指摘に集中できるよう、PRを出す側は次の3点を必ず満たす。

| 前提条件 | 内容 |
| --- | --- |
| PR説明の記載 | 何を・なぜ変えたかをシンプルに書く。レビュワーが差分の意図を読み解く負担を減らす配慮として必須 |
| CIの全成功 | 静的解析・テストがpassしていることはレビューの最低条件。レビュワーに機械が検出できる不備を確認させない |
| PRを小さく保つ | 詳細は[PRは小さく保つほどレビュー品質が上がる](#prは小さく保つほどレビュー品質が上がる)を参照 |

> [!IMPORTANT]
> **（AI・必須）** 上記が満たされていないPRは、レビュワーがレビューを拒否し、前提を満たしてからの再依頼を求めることができる。前提を欠いたPRを無理にレビューするより、整えてもらう方がレビュー品質と全体のスループットの双方を守れるためである。

## レビューコメントにはラベルをつけて意図とマージへの影響を明確にする

コメントの冒頭に以下のラベルをつける。ラベルなしのコメントは **LGTM** 扱いとする。

| ラベル | 意味 | マージへの影響 |
| --- | --- | --- |
| `MUST` | 必ず修正が必要 | ブロック |
| `WANT` | 修正を推奨するが必須ではない | ブロックしない（未対応でマージするなら理由を一言残すと親切） |
| `FYI` | 参考情報・知識共有 | ブロックしない |
| `LGTM` | 承認・問題なし | — |

### 各ラベルの使い方

**MUST**
バグ・セキュリティリスク・仕様違反など、このままマージすると問題が生じる指摘に使う。
対応完了まで Approve しない。

```
MUST: このクエリはN+1が発生します。eagerロードを使ってください。
```

**WANT**
設計の改善・可読性の向上・パフォーマンスの最適化など、対応すればより良くなる提案に使う。
対応するかどうかはレビュイーの判断に委ねる。

```
WANT: この条件分岐はStrategy Patternにすると拡張しやすくなりそうです。
```

**FYI**
知っておくと役立つ情報・代替手段の紹介・背景の補足など。
アクション不要。積極的に使ってチームの知識を広げる。

```
FYI: PostgreSQLではこのケースにGINインデックスが有効なことがあります。
FYI: 同様の処理が ○○ モジュールにも存在します。
```

**LGTM**
コード全体に問題がない場合に使う。称賛を添えると場の雰囲気がよくなる。

```
LGTM! エラーハンドリングが丁寧で読みやすいです。
```

## 変更の性質に応じてレビューケースを使い分ける

### 背景

PRレビューは開発のボトルネックになりやすい。レビュー待ちによってスループットが下がる問題を避けるため、すべてのPRに同じレビュープロセスを課すのではなく、変更の性質に応じてレビュー方針を使い分ける。

一方で、レビューには品質保証以外に知識共有や教育的側面もある。そのため、方針上は自己マージが許容されるPRであっても、必要と判断した場合はレビューを依頼できる余地を残す。

### 前提

すべてのPRはAIによるセルフレビューを実施したうえでマージする。

### レビューケース

| ケース | 内容 |
| --- | --- |
| ケース1 | レビュワーの承認を得てからマージ |
| ケース2 | 自己マージ後、レビュワーに非同期レビューを依頼 |
| ケース3 | 自己マージ |

### PRラベルと変更の種類

ラベルは複数付与されることもある

| ラベル | 変更の種類 | 備考 |
| --- | --- | --- |
| `feature` | 新機能・機能変更 | 振る舞いの変更を含む場合はすべてfeatureをつける |
| `cicd` | CI/CD変更 | .githubディレクトリの変更、静的解析、CIなどの変更は全てこのタグを付ける |
| `bug` | バグ修正 |  |
| `refactor` | リファクタリング | 振る舞いの変更を含まない純粋な構造の変更 |
| `chore` | 設定変更・依存関係更新など |  |
| `docs` | ドキュメント変更 |  |

### ラベルとレビュイーの経験でレビューケースを決める

| ラベル | ジュニア | ジュニア以外 | 理由 |
| --- | --- | --- | --- |
| `feature` を含む | ケース1 | ケース2（必要に応じてケース1） | 振る舞いの変更は常に人間がレビューするべき |
| `cicd` | ケース1 | ケース2（必要に応じてケース1） | セキュリティ上リスクが大きいため人間がレビューするべき |
| `bug` | ケース1 | ケース2 | 修正が正しく問題を解消しているか人間が確認するべき |
| `chore`・`docs` | ケース2 | ケース3 | 振る舞いに影響せずリスクが低いため |
| それ以外（`refactor` 等） | ケース3（必要に応じて1or2） | ケース3（必要に応じて1or2） | 開発スピードを優先するため |

### 補足

- 迷ったときはケース3より上のケースを選ぶ

## PRは小さく保つほどレビュー品質が上がる

小さいPRを高速にマージすることを基本方針とする。PRのサイズが大きくなるほどレビューの認知負荷が上がり、見落としが増える。

なお「〜200行」という目安は、書籍『Googleのソフトウェアエンジニアリング』にある「ほとんどのケースでPRは200行以内にすべき」という記述を出典としている。

| 規模 | 差分行数の目安 | 対応 |
| --- | --- | --- |
| 小 | 〜200行 | 理想的なサイズ |
| 中 | 200〜400行 | 許容範囲 |
| 大 | 400行以上 | issueをさらに分割できないか検討する |

差分行数は追加・削除の合計を指す。自動生成ファイル・ロックファイルなど機械的な変更は除外して判断してよい。

「大」以上のPRは、レビュワーの判断でレビューを拒否し、分割を差し戻すことができる。大きなPRを無理にレビューするより、適切な粒度に分割してもらう方が、レビュー品質と全体のスループットの双方を守れるためである。

ただし、分割が常に最善とは限らない。密に関連する変更を無理に切り離すと、各PRが単体では意味をなさず文脈を欠き、かえってレビューしづらくなることがある。まとめて出した方が適切だと判断する場合は、大きいまま出してよい。その際は「なぜ分割せずこのサイズにしたか」をPR説明に添え、レビュワーが背景を理解したうえで判断できるようにする。
