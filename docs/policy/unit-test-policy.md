# 単体テスト方針

テストコードを実装・レビューする開発者が、何をどこまでテストするか・テストダブルの使い方・テストコードの書き方に迷ったときに参照する指針。

> [!IMPORTANT]
> **TL;DR**
>
> - 古典学派スタイルを採用する。隔離対象はテストケース、テストダブルは共有依存と、実行を遅く・非決定的にする依存にのみ使用可。
> - テスト対象はビジネスロジックのみ。明確な価値をもたらさないテストは書かない。
> - 単体テストでは原則モックを使用しない（呼び出し回数・順序が仕様である等、限定的な例外のみ。[インタラクションテストが適切な場合](#インタラクションテストが適切な場合)参照）。スタブも可能な限り避ける。

## 基本思想

本プロジェクトの単体テストは以下の3つの思想を基盤とする。

### コードは資産ではなく負債である

コードが増えればバグが持ち込まれる経路が増え、保守コストが上がる。テストコードも例外なく保守対象。したがって、**明確な価値をもたらさないテストは作成しない**。質の悪いテストを書くくらいなら、書かないほうがマシ。

### 古典学派（Classical School）スタイルの採用

- 隔離対象は**コードではなくテストケース**。各テストケースが互いに影響を与えず独立して実行できることを重視する
- テストダブルの使用は**共有依存（テストケース間で共有される状態）のみ**に限定する
- 「単体」は1つのクラスに限定せず、**同じ振る舞いを達成するクラスのグループ**も含む

### 価値あるテストだけを実装する

最小限のコストで最大の価値を生むテストスイートを目指す。その質は、各テストケースを1つずつ評価してしか測れない。何をテスト対象にするかは[テスト対象の判断基準](#テスト対象の判断基準)に従う。

## 検証原則

### 振る舞いを検証し、実装の詳細を検証しない

テストは**観察可能な振る舞い（出力・戻り値・状態変化）**のみを検証する。内部の実装詳細（privateメソッドの呼び出し順序、内部変数の中間状態など）を検証してはならない。

<details>
<summary>なぜ実装の詳細を検証してはならないのか</summary>

実装の詳細を検証すると、振る舞いを変えないリファクタリング（メソッド名の変更、内部処理の再構成など）でテストが壊れる。これは**偽陽性**（コードは正しく動いているのにテストが失敗する状態）を引き起こす。偽陽性が多いテストスイートは開発者の信頼を失い、テスト失敗が日常的に無視される（**オオカミ少年問題**）。結果としてテストスイートの存在意義が失われ、本物のバグを見逃す原因になる。
</details>

```typescript
// ✅ 良い例：観察可能な振る舞い（出力）を検証
// → 内部でどのような関数を使っていても、このテストは壊れない
const result = await handler(event, context)
expect(result.statusCode).toBe(200)
expect(JSON.parse(result.body).userId).toBe('user-123')

// ❌ 悪い例：実装の詳細（内部メソッドの呼び出し過程）を検証
// → formatName を rename しただけでテストが壊れる（偽陽性）
expect(formatNameSpy).toHaveBeenCalledWith('alice')
expect(validateInputSpy).toHaveBeenCalledTimes(1)
```

### グレーボックステストで書く

テストの**実装**はブラックボックスで行い、テストケースの**設計・過不足確認**にはホワイトボックス的な分析（カバレッジ計測等）を活用する。この2つのアプローチを組み合わせた**グレーボックステスト**を採用する。

| フェーズ                 | アプローチ       | 具体的な活動                                                     |
| ------------------------ | ---------------- | ---------------------------------------------------------------- |
| テストケースの設計       | ブラックボックス | 仕様・振る舞いから必要なテストケースを洗い出す                   |
| テストの実装             | ブラックボックス | 入力と出力のみに着目し、実装の詳細を検証しない                   |
| テストケースの過不足確認 | ホワイトボックス | カバレッジを計測し、未検証の経路からテストケースの漏れを特定する |

**本質的に重要な原則：** 必要なテストだけをやる。ブラックボックスの思想は「実装の詳細を検証するな」の意味であり、「実装を見るな」「テストケースを追加するな」の意味ではない。カバレッジ分析で未検証の重要な経路が見つかった場合は、テストケースを追加すべき。

<details>
<summary>なぜグレーボックステストを採用するのか</summary>

純粋なブラックボックステストは、仕様からのみテストケースを導出するため、実装上の重要な分岐を見落とすリスクがある。一方、純粋なホワイトボックステストは実装の詳細に依存するため、リファクタリングへの耐性が低い。

本プロジェクトでは両者の長所を組み合わせる：

- **リファクタリングへの耐性**を確保するため、テストの実装（assertの対象）はブラックボックスで行う
- **テストケースの網羅性**を確保するため、カバレッジ計測によるホワイトボックス分析でテストケースの過不足を確認する

つまり「何を検証するか」はブラックボックスで決め、「テストケースが足りているか」はホワイトボックスで確認する。
</details>

### 検証手法の優先順位

| 優先度 | 手法                   | 検証内容             | 使用場面                 |
| ------ | ---------------------- | -------------------- | ------------------------ |
| 1      | 出力値ベース           | 戻り値の確認         | 副作用のない処理         |
| 2      | 状態ベース             | 処理後の状態確認     | 状態を変化させる処理     |
| 3      | インタラクションベース | 依存への呼び出し確認 | 単体テストでは原則不使用 |

## テストダブルの使用方針

### 用語の定義

テストダブル（dummy/stub/spy/mock/fake）の厳密な定義は [test-terms.md](../reference/test-terms.md) を参照（単一の真実）。ここでは単体テストで頻出する**スタブとモックの使い分け**だけを再掲する。

> **判断基準：** そのテストダブルに対して `toHaveBeenCalledWith` 等の**検証（assert）を行うかどうか**で区別する。検証しない → スタブ（データ提供）、検証する → モック（呼び出し検証）。

```typescript
// スタブ：固定レスポンスを返すだけ。呼び出し自体は検証しない
const stubRepository = { findById: () => ({ id: 'user-123', name: 'Alice' }) }

// モック：呼び出しの有無・引数を検証する
const mockEmailSender = vi.fn()
// ...（テスト実行後）
expect(mockEmailSender).toHaveBeenCalledWith('alice@example.com', '件名', '本文')
```

### 基本方針：単体テストではモック・スタブを可能な限り使用しない

<details>
<summary>なぜテストダブルの使用を避けるのか</summary>

テストダブルを多用すると、テストが実装の詳細（どのメソッドがどの引数で呼ばれたか）に依存しやすくなり、リファクタリングへの耐性が低下する。また、テストダブルで置き換えた部分は実際のコードが動いていないため、退行（バグ）を検出する能力も下がる。古典学派の原則に基づき、ビジネスロジック（ドメイン）とプロセス外依存の連携（コントローラ）が分離されていれば、単体テストの対象であるドメインは外部依存を持たないはずであり、テストダブルは不要になる。
</details>

| 依存の種別               | テストダブル使用 | 具体例                        |
| ------------------------ | ---------------- | ----------------------------- |
| 共有依存                 | ✅ 使用する      | DB、ファイルシステム、外部API |
| プライベート依存（可変） | ❌ 使用しない    | 内部で利用するクラス・関数    |
| プライベート依存（不変） | ❌ 使用しない    | 値オブジェクト、定数          |

### 速度・決定性を壊す依存も置き換え対象とする

ユニットテストは**速い**ことと**決定的**であることが絶対条件。共有依存でなくても、この2条件を壊す依存はテストダブルで置き換える。

| 軸       | なぜ置き換えるか                                                   | 具体例                     |
| -------- | ------------------------------------------------------------------ | -------------------------- |
| 実行時間 | 実行に時間のかかる依存はテスト全体を遅くし、頻繁な実行を妨げる     | 重い計算、ネットワーク往復 |
| 決定性   | 実行ごとに結果が変わる依存は偽陽性を生み、テストへの不信につながる | システム時刻、乱数         |

### スタブとのやりとりを検証してはならない

スタブは最終的な結果を生成するために必要なデータを提供するだけのもの。

<details>
<summary>なぜスタブとのやりとりを検証してはならないのか</summary>

スタブの呼び出し過程（どの引数で呼ばれたか、何回呼ばれたか）を検証することは**過剰検証（over specification）**にあたる。これはテストを実装の詳細に結びつけることになり、内部のリファクタリングでテストが壊れる原因になる。検証すべきはスタブから得たデータを使って生成される**最終的な結果のみ**。
</details>

### 外部通信の検証は単体テストではなく統合テストで行う

単体／統合の境界は「本物のプロセス外依存に実際に触れるか」で決まる（[test-terms.md](../reference/test-terms.md) 参照）。共有依存（外部API等）をテストダブルで差し替えれば本物に触れていないので、外部依存への**呼び出し自体の検証（モック）**も技術的には単体テストとして書ける。

ただし本プロジェクトでは、**外部通信の検証は単体テストで行わず統合テストに委ねる**。統合テストで本物の依存を使うか境界でダブル化するかは依存の制御可否によって決める（[test-strategy-policy.md](test-strategy-policy.md) の「制御できる依存は本物、できない依存は境界でダブル化」）。単体テストでモックによる呼び出し検証を行うのは、本物もフェイクも使えない等の限定ケース（次項参照）に留める。

<details>
<summary>なぜ単体テストでのモック検証を基本にしないのか</summary>

モックによる呼び出し検証は実装の詳細（どの依存をどう呼んだか）に結びつきやすく、退行（バグ）の検出能力も下がる。連携が「本当に動くか」は本物に近い依存を使う統合テストで初めて保証できるため、単体テストでのモック検証は本物もフェイクも用意できない暫定策に限る。
</details>

### インタラクションテストが適切な場合

原則は **「インタラクションテストよりステートテストを優先する」**。状態や戻り値で振る舞いを検証できるなら、そちらを使う。ただし次の場合に限り、呼び出し自体を検証するインタラクションテストを使ってよい。

| 場面                                 | なぜインタラクションテストになるか                                         | 例                                          |
| ------------------------------------ | -------------------------------------------------------------------------- | ------------------------------------------- |
| 本物の実装もフェイクも使えない       | 状態で検証したくても、検証に使う本物・フェイクが用意できない               | 他チームが開発中で間に合わない依存          |
| 呼び出しの回数・順序自体が仕様である | 回数や順序を誤ること自体が望ましくない挙動であり、状態だけでは検証できない | キャッシュ（2回目は元データを呼ばない、等） |

<details>
<summary>各場面の補足</summary>

- **本物もフェイクも使えない：** 例えば、外部依存を別チームが開発中でリリースが遅れ間に合わない、依存の仕様が未確定でフェイクすら作れない、といったケースが該当する。あくまで暫定策であり、本来は統合テストで本物を使って検証するのが望ましい。依存が使えるようになった時点でそちらへ寄せる。
- **回数・順序が仕様：** キャッシュのように「1回しか呼ばないこと」自体が守るべき振る舞いの場合、最終状態が同じでも内部の呼び出し回数を検証しないと仕様を保証できない。この場合の呼び出し検証は実装の詳細ではなく観察可能な振る舞いの検証にあたる。

</details>

## テストコードの書き方

### テストはDRYよりDAMPであるべき

テストコードは、多少の重複を許容してでも **DAMP（Descriptive And Meaningful Phrases：説明的で意味のわかる記述）** を優先する。各テストケースが、他の箇所を読みに行かなくても単体で「何を・どんな条件で検証しているか」を語れる状態を目指す。

| 観点       | プロダクションコード | テストコード                     |
| ---------- | -------------------- | -------------------------------- |
| 第一優先   | DRY（重複の排除）    | DAMP（説明性・意図の伝達）       |
| 重複の扱い | 原則として排除する   | 説明性を損なわない範囲で許容する |

<details>
<summary>なぜテストではDRYよりDAMPを優先するのか</summary>

- **テストは「失敗したときに読まれる」もの：** テストが読まれる主な場面は失敗時であり、そこで素早く原因を把握できることが最重要
- **共通ヘルパーへの過度な抽象化は、読む箇所を増やす：** Act/Assert のロジックを共通ヘルパーに隠すと、テスト失敗時に「テスト本体」と「ヘルパーの中身」の両方を行き来して読む必要が生じ、原因特定が遅れる
- **各テストが独立して意図を伝えられると、デバッグコストが下がる：** テストケース単体で前提・操作・期待結果が完結していれば、他のテストやヘルパーの文脈を読み込まずに失敗を理解できる

</details>

> [!IMPORTANT]
> **（AI・必須）** **DRYがダメというわけではない。** DRYによるリファクタリングは、あくまでテストが説明的で意味がわかりやすいことを重視した上で行うべきであり、単に繰り返しを減らすという目的だけで行ってはならない。

なお、テストデータ生成のファクトリ（[テストデータの共通化パターン](#テストデータの共通化パターン)の Object Mother / Test Data Builder）は DAMP と矛盾しない。`createUser({ membershipLevel: 'gold' })` のように **入力データの意図が呼び出し側に残る**抽象化は説明性を高める。DAMP が避けたいのは、**Act（実行）や Assert（検証）のロジックそのものをヘルパーに隠してしまう**抽象化。

```typescript
// ❌ 悪い例：DRYを優先しすぎ、テスト単体では何を検証しているか読み取れない
it('割引が適用される', async () => {
  const result = await runDiscountTest('gold', [1000, 2000])
  expect(result).toBe(2700)
})
// → runDiscountTest を読みに行かないと、割引率も会員種別の意味も、なぜ2700なのかもわからない
//   失敗時に「テスト本体」と「ヘルパー」の往復が発生する

// ✅ 良い例：DAMP。多少の重複はあるが、テスト単体で前提・操作・期待結果が完結している
it('ゴールド会員は注文金額が10%割引される', async () => {
  const user = createUser({ membershipLevel: 'gold' }) // データ生成の抽象化は意図が残る
  const event = createOrderEvent({ userId: user.id, items: [1000, 2000] })

  const result = await sut(event, context)

  expect(JSON.parse(result.body).totalAmount).toBe(2700) // 3000 の 10%割引
})
```

### AAAパターン（Arrange-Act-Assert）

すべてのテストケースを3つのフェーズで構成する。

| フェーズ        | 役割                                     | ルール                                                            |
| --------------- | ---------------------------------------- | ----------------------------------------------------------------- |
| Arrange（準備） | テストに必要なオブジェクト・データを用意 | 長すぎる場合はプロダクションコードの設計を見直すシグナル          |
| Act（実行）     | テスト対象の処理を実行                   | **必ず1行**。複数行になる場合はカプセル化の失敗を示す             |
| Assert（確認）  | 結果を検証                               | 1単位の振る舞いの結果が複数になることはあるため、複数assertは許容 |

<details>
<summary>各フェーズのルールの理由</summary>

- **準備が長すぎる場合：** テスト対象が多くの依存を必要としていることを意味し、クラスや関数の責務が大きすぎる可能性がある。テストダブルで隠すのではなく、プロダクションコードの設計を改善すべきシグナルとして扱う
- **実行が1行でない場合：** 1つの目標を達成するために複数の操作が必要になっていることは、公開APIのカプセル化が不十分であることを意味する。1回の呼び出しで目標を達成できるようにAPIを設計すべき
- **確認が複数行の場合：** 古典学派では1単位の振る舞いを検証するため、その結果が複数の値に現れることは自然。ただし、あまりにassertが多い場合はテストの焦点がぼやけていないか見直す

</details>

**フェーズコメントの記載ルール：**

- 準備フェーズまたは確認フェーズのコードが複数行にわたる（空白行での区切りが必要な）場合 → 各フェーズの先頭に `// Arrange` `// Act` `// Assert` コメントを入れる
- それ以外の場合 → コメントは入れず、**空白行のみ**でフェーズを区切る

```typescript
// ✅ 良い例：シンプルなテスト → 空白行のみでフェーズを区切る
it('有効なIDを指定した場合にユーザー情報を返す', async () => {
  const event = createApiGatewayEvent({ pathParameters: { id: 'user-123' } })

  const result = await handler(event, context)

  expect(result.statusCode).toBe(200)
  expect(JSON.parse(result.body).name).toBe('Alice')
})

// ✅ 良い例：準備が複数行 → フェーズコメントを入れる
it('ゴールド会員の場合に割引が適用された注文サマリーを返す', async () => {
  // Arrange
  const user = createUser({ membershipLevel: 'gold' })
  const items = [createItem({ price: 1000 }), createItem({ price: 2000 })]
  const event = createApiGatewayEvent({
    body: JSON.stringify({ userId: user.id, items })
  })

  // Act
  const result = await handler(event, context)

  // Assert
  const body = JSON.parse(result.body)
  expect(result.statusCode).toBe(200)
  expect(body.totalAmount).toBe(2700)
  expect(body.discountApplied).toBe(true)
})
```

### テストケースの命名規則

- **日本語で記述する**（非開発者にも検証内容が伝わるようにする）
- テスト対象の**メソッド名を含めない**（テストはコードではなく振る舞いを検証している）
- 「〜する」「〜される」「〜を返す」のように**事実を示す表現**で書く
- `describe` のグループ名も日本語で記述する

```typescript
// ✅ 良い例：日本語で振る舞いが明確、メソッド名を含まない
'必須パラメータが欠落している場合に400を返す'
'有効期限切れのセッションは拒否される'
'ゴールド会員の場合に割引が適用される'

// ❌ 悪い例：メソッド名を含む
'handleRequestは不正な入力に対して400を返す'

// ❌ 悪い例：検証内容が曖昧
'不正な入力でエラーを返す' // 「不正」が何を指すか不明確
```

### SUT（テスト対象）の明示

テスト対象は変数名 `sut` で明示し、テスト対象と依存の区別をわかりやすくする。

```typescript
const sut = handler // Lambda handlerがテスト対象
const result = await sut(event, context)
```

### テスト内でif文を使わない

テストケース内にif文がある場合、1つのテストケースで多くのことをしようとしている兆候。テストケースを分割すること。

### テストケースの分離基準

テストケースを分けるか1つにまとめるかは、**仕様上の条件が異なるかどうか**で判断する。前提条件（Arrange）の入力値が異なっていても、仕様上同じ条件のバリエーションであれば1つのテストケースにまとめてよい。

**仕様上の条件とは：** テストケース名における条件部分（「〜の場合に」）を指す。この条件部分が同じ文言で自然に表現できるなら同じ条件であり、異なる文言でないと区別できないなら異なる条件と判断する。

| 入力値               | テストケース名の条件部分                     | 判断            |
| -------------------- | -------------------------------------------- | --------------- |
| `email = ''`         | 「メールアドレスの形式が不正な場合に」       | 同じ → まとめる |
| `email = '@foo'`     | 「メールアドレスの形式が不正な場合に」       | 同じ → まとめる |
| トークンなし         | 「未認証の場合に」                           | 異なる → 分ける |
| 他ユーザーのリソース | 「他ユーザーのリソースにアクセスした場合に」 | 異なる → 分ける |

<details>
<summary>なぜ仕様上の条件で分けるのか</summary>

テストケース名は仕様のドキュメントとして機能する。テストが失敗したとき、テストケース名を見て「どの仕様ルールが壊れたか」を特定できる必要がある。仕様上異なる条件を1つのテストケースにまとめてしまうと、失敗時にどのルールが壊れたのか特定できなくなる。
</details>

```typescript
// 仕様：
//   - メールアドレスの形式が不正な場合、400を返す
//   - 未認証の場合、403を返す
//   - 他ユーザーのリソースにアクセスした場合、403を返す

// ✅ まとめてよい：入力値は異なるが、仕様上は同じ条件（メール形式不正）
it.each(['', 'invalid', '@no-local', 'no-domain@'])(
  'メールアドレス「%s」が不正な場合に400を返す',
  async (email) => {
    const event = createEvent({ body: JSON.stringify({ email }) })

    const result = await sut(event, context)

    expect(result.statusCode).toBe(400)
  }
)

// ✅ 分けるべき：出力は同じ403だが、仕様上は別の条件
it('未認証の場合に403を返す', async () => {
  const event = createEvent({ headers: {} })

  const result = await sut(event, context)

  expect(result.statusCode).toBe(403)
})

it('他ユーザーのリソースにアクセスした場合に403を返す', async () => {
  const event = createEvent({
    headers: { Authorization: 'Bearer token-user-A' },
    pathParameters: { id: 'resource-of-user-B' }
  })

  const result = await sut(event, context)

  expect(result.statusCode).toBe(403)
})
```

### パラメータ化テスト

- 仕様上同じ条件のバリエーションに異なる入力を流し込む場合に使用する
- **正常系と異常系は必ず分離する**

```typescript
// ✅ 良い例：正常系と異常系を分けたパラメータ化テスト
describe('配送日バリデーション', () => {
  // 異常系
  it.each([-1, 0, 1])('現在から%d日後の日付は無効である', (days) => {
    const date = addDays(new Date(), days)
    expect(isValidDeliveryDate(date)).toBe(false)
  })

  // 正常系
  it.each([2, 3, 10])('現在から%d日後の日付は有効である', (days) => {
    const date = addDays(new Date(), days)
    expect(isValidDeliveryDate(date)).toBe(true)
  })
})
```

## テストフィクスチャの方針

### beforeEachでの過度な共通化を避ける

`beforeEach` でテストフィクスチャを共通化すると、テストケース間の結合が強くなり、1つのテストの修正が他のテストに影響するリスクが生じる。

**beforeEachに配置してよいもの：**

- テスト実行に最低限必要な土台（DB接続の初期化、ログ出力の初期化など）

**beforeEachに配置してはならないもの：**

- テストケース固有のデータ・オブジェクトの準備

### テストデータの共通化パターン

準備フェーズのコード重複を避けるために、以下のパターンを使用する。

| パターン          | 用途                           | 特徴                              |
| ----------------- | ------------------------------ | --------------------------------- |
| Object Mother     | 典型的なテストデータの生成     | メソッド名でデータの意図が明確    |
| Test Data Builder | 柔軟なカスタマイズが必要な場合 | デフォルト値 + 必要箇所のみ上書き |

```typescript
// Object Motherパターン：ファクトリ関数でデフォルト値を持つテストデータを生成
const createApiGatewayEvent = (overrides?: Partial<APIGatewayProxyEvent>) => ({
  httpMethod: 'GET',
  path: '/',
  headers: {},
  pathParameters: null,
  queryStringParameters: null,
  body: null,
  ...overrides
})

// 使用例：必要な部分だけ上書き
const event = createApiGatewayEvent({ pathParameters: { id: 'user-123' } })
```

## カバレッジの考え方

### カバレッジの位置づけ

カバレッジはテストケースの**過不足を確認するための分析ツール**として積極的に活用する。

### カバレッジで「できること」と「できないこと」

|                 | 説明                                                                     |
| --------------- | ------------------------------------------------------------------------ |
| ✅ できること   | 未検証の経路を特定し、テストケースの**漏れ**を発見する                   |
| ❌ できないこと | テストの**質**を証明する（assertなしでもカバレッジにカウントされるため） |

### 運用方針

- カバレッジ率の**目標値の強要はしない**（100%でなければコミットできない等は不適切）
- カバレッジ計測で未検証の経路が見つかった場合、その経路が[テスト対象の判断基準](#テスト対象の判断基準)に該当するなら**テストケースを追加する**
- カバレッジの数値そのものではなく、**未検証の経路の中身**を見て判断する

## テスト対象の判断基準

「コードは資産ではなく負債である」の原則に従い、すべてのコードにテストを書くのではなく、**テストによるバグ検出の価値がテストの保守コストを上回る部分**のみをテスト対象とする。

### テスト対象とするもの

- ビジネスロジック（ドメイン上重要な処理）
- ドメイン上重要な制約の事前条件チェック（ガード句）
- サポートログ（外部向け：いつ・誰が・どの操作をしたか）の出力

<details>
<summary>テスト対象とする理由</summary>

- **ビジネスロジック：** テストの費用対効果が最も高い部分。バグが混入した場合のビジネスへの影響が大きく、テストによる保護の価値が高い
- **事前条件チェック：** データの整合性を守る不変条件の侵害を防ぐ処理であり、壊れた場合の影響範囲が広い
- **サポートログ：** カスタマーサポートや監査など外部から観察されることを意図したログは、アプリケーションの観察可能な振る舞いの一部

</details>

### テスト対象としないもの

- privateメソッド・private関数
- 単純な委譲のみのコード
- 診断ログ（開発者向けのデバッグ用ログ）の出力
- ライブラリ内部の挙動

<details>
<summary>テスト対象としない理由</summary>

- **privateメソッド・private関数：** 実装の詳細にあたる。直接テストするとリファクタリング時にテストが壊れ（偽陽性）、リファクタリングへの耐性を損ねる。privateな処理はそれを呼び出すpublicなAPIを通じて間接的に検証されるべき
- **単純な委譲のみのコード：** テストを書いても検出できるバグがほぼなく、費用対効果が低い
- **診断ログ：** 開発者しか見ないログは実装の詳細であり、外部から観察される振る舞いではない。テスト対象にすると内部構造の変更でテストが壊れる
- **ライブラリ内部の挙動：** サードパーティの責務であり、自プロジェクトのテストで保証すべき範囲ではない

</details>

## フロントエンド固有のガイドライン

### 基本方針

フロントエンドの単体テストも古典学派の原則に従い、**ユーザー操作に対する振る舞い**を検証する。内部の実装詳細（state変数の値、内部メソッドの呼び出し）ではなく、ユーザーが観察できる結果（画面の表示内容、DOM の変化）を検証する。

### 検証対象

| 対象               | 検証する振る舞い                     |
| ------------------ | ------------------------------------ |
| コンポーネント     | ユーザー操作に対するレンダリング結果 |
| カスタムフック     | 入力に対する戻り値・状態変化         |
| ユーティリティ関数 | 入力に対する出力値                   |

### コンポーネントテストの原則

- **ユーザーの視点**でテストする（Testing Libraryの思想に準拠）
- 要素の取得は `getByRole`、`getByText` など**ユーザーが認識できる属性**で行う
- 内部stateやpropsの直接検証はしない

```typescript
// ✅ 良い例：ユーザー視点で振る舞いを検証
it('フォーム送信失敗時にエラーメッセージを表示する', async () => {
  render(<LoginForm />)

  await userEvent.click(screen.getByRole('button', { name: '送信' }))

  expect(screen.getByText('入力内容を確認してください')).toBeInTheDocument()
})

// ❌ 悪い例：実装の詳細（state）を検証
it('送信時にエラーstateをセットする', () => {
  const { result } = renderHook(() => useLoginForm())
  act(() => result.current.handleSubmit())
  expect(result.current.errorState).toBe(true)  // 内部stateの検証
})
```
