# バグ修正ポリシー

不具合（バグ）を報告・修正する開発者・AI が、何をもって「良いバグ修正」とするかを知りたいときに参照する。バグを起票する前・不具合の修正に着手する前に通すゲート。

> [!IMPORTANT]
> **TL;DR（このポリシーの決定事項）**
>
> - **バグレポートには「再現方法・発生頻度・本来の仕様・実際の動作」の4点を必ず含める。** これが揃わない報告は修正に着手できる状態ではない
> - **修正の前に、まず不具合を再現する自動テストを書く。** 「修正前は落ち、修正後は通る」ことでバグの存在と修正の有効性を証明し、再発（リグレッション）を恒久的に防ぐ
> - **直す前に原因を最小まで絞り込む。** 当てずっぽうの修正を積み重ねない（[原則7](refined-engineer-judgment-principles.md)）

> [!NOTE]
> 本ポリシーは「バグ修正のアプローチ」を定義するものであり、テストの書き方（命名・構造・どの層に書くか等）には触れない。それらは [unit-test-policy.md](unit-test-policy.md) / [test-strategy-policy.md](test-strategy-policy.md) に従う。

## 1. 良いバグレポート——直せる状態で起票する

バグを起票する Issue には、最低限**次の4点**を含める。1つでも欠ければ、修正担当（人間・AI）は再現も検証もできず、推測で直すことになる。推測で直したバグは「直ったように見えるが直っていない」状態を生む。

| 項目           | なぜ必要か                                                                         | 書き方の要点                                                                                             |
| -------------- | ---------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **再現方法**   | 再現できない不具合は、原因特定も修正の検証もできない                               | できるだけ詳しく。前提条件・操作手順・入力データ・環境（バージョン等）を、第三者がそのままなぞれる粒度で |
| **発生頻度**   | 常時か散発かで原因の当たり（競合・タイミング依存・特定データ依存）と優先度が変わる | 「毎回」「N回に1回」「特定条件下でのみ」など観測した頻度を具体的に                                       |
| **本来の仕様** | 何を「正しい」とするかの基準。これが無いと「直った」を判定できない                 | バグが無い場合の望ましい動作。仕様書・設計書に根拠があればリンクする                                     |
| **実際の動作** | 本来の仕様との差分が、修正すべき対象そのものになる                                 | 観測された結果。エラーメッセージ・ログ・スクリーンショット等の証拠を添える                               |

> [!TIP]
> **（AI・推奨）** **「本来の仕様」と「実際の動作」を必ず分けて書く。** バグとは「本来の仕様」と「実際の動作」の差分である。この2つを並べて初めて、何を直すのかが一意に定まる。差分が言語化できないなら、それはまだバグとして起票できる状態にない（仕様の確認が先）。

## 2. 不具合修正の手順——再現テストを先に書いてから直す

不具合の修正は、必ず**先に再現テストを書いてから**着手する。手順は以下に従う。

> [!NOTE]
> 本節のアプローチは『プログラマが知るべき97のこと』所収の和田卓人（t-wada）氏のコラムに基づく。

1. **手元で不具合を再現させる。** 再現できなければ、原因特定も修正の検証もできない。まず Issue の再現方法どおりに自分の手元で再現させる
2. **コードを注意深く調べ、不具合を発生させている最小の部分を絞り込む。** 当てずっぽうで直さない。原因箇所を最小まで特定する
3. **最小レベルで不具合を再現させ、修正されたら通る自動テストを書く。** バグを最小単位で捉えた、回帰を見張り続けるテストを書く
4. **3 で書いたテストを実行し、落ちることを確認する。** 落ちないテストは、バグを捉えていないか、原因の絞り込み（手順2）が誤っている。**ここで必ず落とす**ことが、テストがバグを正しく捕まえている証明になる
5. **不具合を修正する。** ここで初めてプロダクトコードに手を入れる
6. **3 で書いたテストが通ることを確認する。** これで修正の有効性が証明される
7. **既存の全テストを実行し、修正が他を壊していないかを確認する。** 1つのバグを直して別の機能を壊しては意味がない。全テストの緑をもって完了とする

### なぜ「再現テストを先に書く」のか

バグ修正は本質的に**「当て物」**である。どこが原因かを推測してから直すしかなく、**「ここが原因だと思って修正したら、全然違う場所が原因だった」**という事故が起こりやすい。再現テストを先に書くことは、この推測（当て）を修正前に検証可能にし、的外れな修正を炙り出すためのアプローチである。

## 上位原則との関係

本ポリシーは横断原則の具体化である。判断に迷ったら上位の原則に立ち返る。

- [原則6「目的を見失えば、速く進むほど遠ざかる」](refined-engineer-judgment-principles.md)：バグ修正の目的は「本来の仕様に戻す」こと。本来の仕様（レポートの必須項目）を握らずに直すと、違うものを直す
- [原則3「シフトレフト——検証を前倒しする」](refined-engineer-judgment-principles.md)：再現テストは回帰検証をテスト層へ前倒しする仕組み
- [原則7「ハードワークは報われない——難航は『やり方が違う』の信号」](refined-engineer-judgment-principles.md)：原因を絞り込まず当てずっぽうの修正を積み増すのは「やり方が違う」のサイン。再現で原因を特定してから直す
- [原則8「理解は委譲できない」](refined-engineer-judgment-principles.md)：最小部分への絞り込みは、原因を理解する行為そのもの
