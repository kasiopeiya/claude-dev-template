# 構成管理ポリシー

構成値（設定値）を **コードに直書きするか・外部の設定へ追い出すか**、そして **アプリがそれをどう受け取るか** で判断が分かれる事柄に、我々の既定の立場を declare する。[application-architecture-policy「詳細と距離をとる」](application-architecture-policy.md) を継承し、その構成値版の差分だけを書く。

教科書的なテクニック解説は書かない。載せるのは「**判断が分かれるが、我々はこうする**」という既定の立場だけである。いずれの立場も既定であり、文脈が強ければ覆る（[原則1](refined-engineer-judgment-principles.md)）。

## 対象読者

構成値を「直書きするか設定に出すか」「アプリでどう読むか」で迷う開発者・AIエージェント。設定ファイル（`config.ts`・`.env`・パラメータ定義など）を編集する前に参照する。

> [!IMPORTANT]
> **TL;DR（このポリシーの決定事項）**
> - **環境で変わる値・外部依存・認証情報の参照先・変更が予想される値**は、コードに直書きせず設定へ追い出す。
> - **機密の実値・ビジネスロジック・不変の定数**は設定に書かない（設定に置くのは機密の*参照先*だけ）。
> - アプリは構成値を **境界の単一 config モジュールで読み・検証・型付け**し、他所で `process.env` を直接触らない（起動時に fail-fast）。

## 1. 何を設定へ追い出すか——判断基準

**既定：次のいずれかに当てはまる値はコードに直書きせず、設定（外部の構成）へ追い出す。**

| 区分 | 何が該当するか |
|---|---|
| 環境によって異なる値 | dev/stg/prod などデプロイ先ごとに変わる値（接続先・有効化フラグ・上限値など） |
| 外部依存パラメータ | 外部 API のエンドポイント・リージョン・バケット名など、自分の管理外の世界を指す値 |
| 認証情報の「参照先」 | シークレットの所在を示す識別子（例：Secrets Manager のキー名／ARN、環境変数名）。**実値ではなく参照先だけ**を扱う（§2） |
| 変更が予想される値 | タイムアウト・リトライ回数・閾値など、コード変更なしに調整したくなる運用パラメータ |

判断が分かれる点：何でも設定に出すべきか。我々は**上記に当てはまるものだけ**を出す——設定への外出しは「変わりうる詳細をコードから切り離す」ための手段であり（[application-architecture-policy 方針と詳細の分離](application-architecture-policy.md)）、目的のない外出しは値をいたずらに分散させ、追跡を難しくするだけだからだ。

> [!NOTE]
> 逆テスト：**環境やデプロイ先によらず一定で、コードと一体で初めて意味を持つ値**は設定に出さない（§2）。「外に出すと却って読みにくくなるか？」を一度問う。

## 2. 設定に書いてはいけないもの

**既定：次のものは設定（`config.ts`・`.env`・パラメータ定義など）に置かない。**

| 置かないもの | 理由・代わりにどうするか |
|---|---|
| 機密の実値 | パスワード・API キー・トークンの中身そのもの。設定に置くのは**参照先**（§1）だけで、実値は実行時にその参照先から解決する。設定ファイルや環境変数に焼き込まない |
| ビジネスロジック・分岐 | 「設定で振る舞いを切り替える」は、しばしば[ブーリアン引数アンチパターン](application-design-policy.md)の構成版になる。振る舞いの違いはコードで表現する |
| 不変の定数・派生値 | 絶対に変わらない値はコード内の名前付き定数に、他の構成値から計算できる値は受け取り後に算出する（設定として二重に持たない） |

## 3. アプリでの受け取り方——境界の単一 config モジュール

**既定：env など外部から来る構成値は、境界に置いた1つの config モジュールで「読み・検証・型付け」を済ませ、アプリの他の部分はそこから型付き値を import する。コード各所で `process.env` を直接触らない。**

判断が分かれる点：`process.env.X` を使う場所で都度読むか、1か所に集約するか。我々は**集約**する——理由は複数ある。

| 理由 | 内容 |
|---|---|
| fail-fast | 欠損・不正な構成を、リクエスト処理の奥ではなく**起動時に**検出して落とす |
| 型安全 | `process.env` は全て string。型変換・パースをここで一度だけ行う |
| ポータビリティ | 「構成が env 経由で来る」という*詳細*にアプリ本体が依存しなくなる。供給元を替えても差し替えはこのモジュールだけ（[application-architecture-policy 薄いアダプタ](application-architecture-policy.md)） |
| 凝集 | このアプリが何に依存するかが config モジュールを見れば一覧できる（SRP） |

```typescript
// config.ts — 責務: 外部構成（環境変数）を境界で読み込み・検証・型付けする唯一の場所
function required(key: string): string {
  const value = process.env[key]
  if (!value) throw new Error(`Missing required config: ${key}`) // 起動時に fail-fast
  return value
}

const config = {
  apiEndpoint: required('API_ENDPOINT'),
  dbSecretRef: required('DB_SECRET_REF'), // ← 参照先。実値ではない（§2）
  timeoutMs: Number(required('REQUEST_TIMEOUT_MS')),
} as const

export default config
```

```typescript
// ❌ 各所で直接読む：依存が散り、検証も型付けもバラバラになる
const endpoint = process.env.API_ENDPOINT! // undefined のまま奥へ流れる

// ✅ 境界の config から型付きで受け取る
import config from './config'
const endpoint = config.apiEndpoint
```

## 4. 値の「決定」と「受け取り」を分ける

**既定：構成値の決定（SSOT）はデプロイ層が持ち、アプリはそれを受け取って使うだけにする。**

環境差分の値をどこで決めるかと、アプリがどう受け取るかは別レイヤーの関心である。アプリ側は §3 の config モジュールで受け取り、**値そのものの決定には関与しない**。

| 関心 | 担当 | 本ポリシーの扱い |
|---|---|---|
| 値の決定（SSOT） | デプロイ層（供給元） | 供給元の選択は範囲外 |
| 値の受け取り | アプリの config モジュール（§3） | declare する |
| 何を外部化するか | 共通の判断基準（§1・§2） | declare する |

供給元（値を決め、アプリへ渡す側）は、IaC（インフラ定義）・デプロイ環境の設定・`.env` ファイルなどプロジェクトによって異なる。**どれを供給元にするかは本ポリシーの範囲外**であり、本ポリシーは「何を外部化するか（§1・§2）」と「アプリでどう受け取るか（§3）」だけを declare する。

> [!IMPORTANT]
> 本ポリシーは特定の供給元（CDK 等）を前提にしない——IaC を使わないプロジェクトもあるためだ。IaC 固有の事柄（パラメータ定義ファイルの構造、決定論的な構築との関係など）は、それを採用するプロジェクトの設計・規約（例：[cdk-design-policy](cdk-design-policy.md) / `.claude/rules/cdk.md`）に委ね、ここには書かない。
