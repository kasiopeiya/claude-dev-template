# 仕様書作成ポリシー

仕様書（システムが *何をするか* ＝外から観測できる振る舞いを規定する文書）を書く・レビューする人が、良い仕様書の基準を知りたいときに参照するドキュメント。

## 想定読者には開発者だけでなくビジネスサイドも入りうる

仕様書の想定読者は開発者に限らない。**プロダクトオーナーやビジネスサイドの関係者も、仕様を読み・承認することがある。** 仕様＝外から観測できる振る舞いは、ビジネスが理解し検証できる唯一の層だからである。

そのため、ビジネス読者を含む仕様書は平易に書く（ドメイン用語で書き、実装寄りの専門用語は避け、開発者向けの補足は `<details>` や用語集リンクへ逃がす）。ただし読者は文書ごとに定める——内部技術仕様（サービス間 API 契約など）は開発者限定でよい（documentation-policy「対象読者と目的が全てを決定する」）。

## 仕様書と設計書の書き分け

何を仕様書に書き、何を設計書に書くかは、[design-doc-policy「迷ったら：仕様書か設計書か」](design-doc-policy.md) に従う。迷う項目は「振る舞い（What）＝仕様／構造・なぜ＝設計書／具体値＝コード」の3層に分解して振り分ける。
