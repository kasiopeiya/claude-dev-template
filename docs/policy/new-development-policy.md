# 新規開発ポリシー（グリーンフィールド）

新規開発——システム・モジュール・境界づけられたコンテキストを **0 から立ち上げる**とき——の「進め方」で判断が分かれる事柄に、我々の既定の立場を宣言する。判断の上位原則 [refined-engineer-judgment-principles](refined-engineer-judgment-principles.md)（特に**原則2 決定を遅らせる**・**原則4 YAGNI**）を継承し、マクロ構造は [application-architecture-policy](application-architecture-policy.md) を継承する。本ポリシーは**それらに含まれない新規開発固有の差分だけ**を declare する（重複は再掲せず参照する）。

教科書的なテクニック解説は書かない。載せるのは「**判断が分かれるが、我々はこうする**」という既定の立場だけである。いずれの立場も既定であり、文脈が強ければ覆る（[原則1](refined-engineer-judgment-principles.md)）。

## 対象読者

新しいシステム・モジュールを 0 から設計／実装する人・AIエージェントが、立ち上げの進め方で迷ったとき。

## 1. 曳光弾（walking skeleton）を既定にする——薄い縦スライスを先に通す

**既定：レイヤーごとの水平な積み上げ・big-design-up-front ではなく、入口から出口まで動く薄い縦スライス（曳光弾＝tracer bullet）を最初に1本通し、以降そこへ肉付けする。**

判断が分かれる点：水平（各レイヤーを層ごとに作り込んでから繋ぐ）か、垂直（薄い1経路を端から端まで先に通す）か。我々は**垂直を既定**にする——早期に統合してフィードバックを得られ、「全部作ったが繋がらない」を避けられるため。

- 曳光弾は**本番品質で残す**コードである（次節のスパイクと違い捨てない）。
- 最初に通すのは1本の細い経路（入口 → ユースケース → 永続化スタブ → 出力）。詳細（DB・外部サービス）はスタブで代用し、その決定は[原則2](refined-engineer-judgment-principles.md)に従って遅らせる（再掲しない）。

## 2. スパイクは使い捨て——本番に昇格させない

**既定：不確実な技術的問いは、使い捨てのスパイク（throwaway prototype）で答えだけを得て破棄し、本実装はやり直す。スパイクのコードを「もったいない」と育てて本番にしない。**

判断が分かれる点：学習のために書いたコードを育てるか、捨てるか。我々は**捨てる**——学習目的のコードはテスト・エラー処理・設計という品質の前提を欠いており、育てると負債の土台になる（[原則4](refined-engineer-judgment-principles.md) コードは負債）。

- 曳光弾（残す）とスパイク（捨てる）は**目的で区別**する：到達経路を作るのが曳光弾、特定の不確実性を1つ潰すのがスパイク。

## 3. ドメイン／コアから着手する——FW・DB から始めない

**既定：新規開発は FW 選定・DB スキーマ・画面から書き始めず、ドメインモデル（コアのビジネスロジック）から書き始める。**

判断が分かれる点：何から書くか。我々は**コアから**——詳細を先に決めるとドメインが詳細に従属し、[原則2](refined-engineer-judgment-principles.md)に反する。詳細は境界の外に置き決定を遅らせる（[application-architecture-policy §1](application-architecture-policy.md)）。

## 4. ディレクトリ構成の方針を先を見越して最初に決める

**既定：新規開発の初期に、ディレクトリ構成の*方針*を場当たりに育てるのではなく、プロジェクトの成長を見越して意図的に選ぶ。**

判断が分かれる点：構成は必要に応じて育つに任せるか、先に方針を決めるか。我々は**先に決める**——ディレクトリ構成は後から変えるほど高くつき（広範なファイル移動と import 修正、レビュー差分の肥大）、最初の方針が以後「どこに置くか」の基準になるからだ。

ただし最適な構成はプロジェクトによって大きく異なり、アプローチも複数ある（技術レイヤー型／機能・ドメイン型／その混成 …）。**どのアプローチをどう選ぶかの具体は [directory-structure-guide](../guide/directory-structure-guide.md) に委ね、本ポリシーは「先を見越して意図的に決める」立場だけを declare する。** 方向性の既定（FW でなくドメインを語らせる＝Screaming Architecture）は [application-architecture-policy §5](application-architecture-policy.md) に従う。

> [!NOTE]
> このプロジェクトの**具体的な**ディレクトリ構成は設計の領分であり、[design-hub](../design-hub.md) 配下の設計書が SSOT。本ポリシーは方針（先を見越して意図的に決める）だけを declare し、具体パスは持たない。
