# アプリケーションアーキテクチャポリシー

アプリケーション（コードの内部構造）のアーキテクチャに関する、特定パターン（Clean / Hexagonal / Layered …）に依存しない普遍原則を宣言する。**どのアーキテクチャ特性を重視し、どう均衡させるか**の立場を示すのが目的。具体的なトポロジー（採用パターン・レイヤー名・ディレクトリ対応）の選択は、各プロジェクトの設計書に委ねる。

設計判断・レビューの北極星として常時参照する。具体的な適用手順・レビュー観点は [clean-architecture-guide](../guide/clean-architecture-guide.md) を参照。

---

## 0. メタ原則：アーキテクチャ特性は複数あり、競合する

アーキテクチャの本質は構造そのものではなく、**どのアーキテクチャ特性（-ility）を重視するかの選択と均衡**である（『ソフトウェアアーキテクチャの基礎』）。特性は複数あり、互いに競合する（例：シンプルさ ↔ 拡張性）。すべてを最大化することはできない。

> [!IMPORTANT]
> **（AI・必須）** **このシステムで何を優先するかを決めること自体が、アーキテクチャの第一の仕事**である。これは [refined-engineer-judgment-principles 原則1（すべてはトレードオフ）](refined-engineer-judgment-principles.md) の、アーキテクチャ領域での具体化にあたる。

業務アプリケーションでは **変更容易性を既定の最優先** とする。ただし唯一の特性ではない。特性を無条件に追い求めること（過剰な抽象化・先回りした拡張点）は、[原則4（コードは負債）](refined-engineer-judgment-principles.md) に反する。**いま必要な特性に絞って投資する。**

## 1. 変更容易性

変更の波及を局所に閉じ、詳細の決定を遅らせ続けられる構造を保つ。

- **依存性のルール（The Dependency Rule）**：ソースコードの依存は、常に「安定した方針（policy）」へ内向きに向ける。制御フローの向きとソースコード依存の向きは独立であり、必要なら DI（依存性逆転）で反転させる。
- **方針(policy)と詳細(detail)の分離**：DB・フレームワーク・IO・外部サービスは「詳細」。これらを境界の外側へ追い出し、差し替え可能に保つ。
- **SOLID**：とくに DIP（依存方向を制御する）・SRP（変更理由ごとに分ける）・OCP（拡張で対応し既存を壊さない）がアーキテクチャ層で効く。
- **境界の引き方・決定の遅延**は [原則2（境界を引いて選択肢を残す）](refined-engineer-judgment-principles.md) に従う。境界もコストなので、揺れうる重要な詳細に絞って引く。

### 詳細と距離をとる（ポータビリティ）

**フレームワークもプラットフォームも「詳細」である。これらに自分のコアロジックを寄せきらず、外から呼び出す関係に保つ**（Clean Architecture「フレームワークは詳細」）。上の「方針と詳細の分離」を、コードの書き方へ具体化したのが本項である。フレームワーク（Web FW・ORM・CDK 等）とプラットフォーム（Lambda・コンテナ・特定ランタイム等）は、システムの中で最も移ろいやすい詳細であり、ここへコアを密結合させると**乗り換え・単体テスト・再利用のいずれもできなくなる**。

| 対象 | 守る姿勢 |
|---|---|
| フレームワーク | 機能を「使う」のはよい。ただし**コアロジックの構造を FW の流儀へ作り変えないと使えない機能**は避ける。FW は便利な道具として境界の外から呼び、コアは FW を知らないままに保つ |
| プラットフォーム | エントリポイント固有の作法（Lambda の `handler` 等）は**薄いアダプタに閉じ込め**、本処理はプラットフォーム非依存の素の関数（`main` 等）に書く。本処理が動く場所を選ばない＝そのままテストでき、別の実行基盤へも移せる |

```typescript
// ❌ ハンドラに本処理が癒着：Lambda の外では再利用も単体テストもできない
export const handler = async (event: APIGatewayProxyEvent) => {
  const body = JSON.parse(event.body ?? '{}')
  // …数十行の業務ロジックが handler に直書き…
}

// ✅ ハンドラは薄いアダプタ。本処理はプラットフォームを知らない関数へ押し下げる
export const handler = async (event: APIGatewayProxyEvent) => {
  const input = parseHttpInput(event)        // 入力をドメインの形へ変換
  const result = await registerOrder(input)  // 本処理：Lambda も HTTP も知らない
  return toHttpResponse(result)              // 出力を HTTP の形へ変換
}
```

> [!WARNING]
> **（AI・必須）** ある FW 機能を使うか迷ったら判定する：**その機能を使うために、自分のコアロジックの構造を FW の流儀へ作り変える必要があるか？** Yes なら使わない（境界の内側へ FW を侵入させる）。No（薄いアダプタから呼ぶだけ）なら使ってよい。

> [!IMPORTANT]
> **（AI・必須）** ポータビリティの目的は「いつでも乗り換える」ことではなく、**詳細の決定を遅らせ変更を局所に閉じる**こと。その副産物としてテスト容易性と移植性が手に入る。ゆえに**完全な FW 非依存を目的化して過剰に抽象化しない**（先回りした隠蔽層は [原則4（コードは負債）](refined-engineer-judgment-principles.md)・§0 に反する）。投資先は、現に揺れうる重要な詳細に絞る。

## 2. テスト容易性

境界による分離は、詳細をテストダブルで差し替え可能にし、ドメインを隔離してテストできる状態を生む。**テスト容易な構造は良いアーキテクチャの結果であり、その指標**でもある。テストの方針・配分は [test-strategy-policy](test-strategy-policy.md) / [unit-test-policy](unit-test-policy.md) に従う。

## 3. 凝集と結合（根の力）

変更容易性・テスト容易性・理解容易性のすべてを支える根本の力。**高凝集・低結合**を保つ。

- **関心の分離**：変わる理由が同じものを集め、違うものを分ける（SRP の構造版）。
- **複雑性の管理**：本質的複雑性（ドメイン由来）と偶有的複雑性（手段由来）を分け、後者を増やさない。シンプルな構造を志向する（[原則5](refined-engineer-judgment-principles.md)）。

## 5. 意図を表出する構造（理解容易性）

ディレクトリ・レイヤー構成が、フレームワークやインフラではなく **ドメイン（何のためのシステムか）を語る** こと（Screaming Architecture）。新規参入者・AIが構造を見ただけで意図を読み取れる状態を目指す。

## 6. 射程外（他ポリシーへ委譲）

本ポリシーは **アプリケーションの内部構造** に閉じる。実行時・運用系の特性は以下に委譲し、ここには書かない。

| 特性・関心 | 委譲先 |
|---|---|
| パフォーマンス・可用性・スケーラビリティ | インフラ / CDK 側の設計 |
| 可観測性（監視・アラーム） | [monitoring-policy](monitoring-policy.md) |
| エラー処理 | error-handling-policy |
| セキュリティ | セキュリティポリシー |
