# CDK設計ポリシー

AWS CDK で**コードを書く・レビューする人**が、環境差分の出し方・リソース命名・Stack 構造など「人により判断が割れる場面」で「私たちの考える良い CDK コードとはこれ」を共有し、実装とレビューのチェックリストとして使うためのドキュメント。

> [!IMPORTANT]
> **TL;DR（このポリシーの決定事項）**
> - **環境差分は Stack 内で分岐させない**。値の差分は `parameter.ts` へ、構造・振る舞いの差分は Builder 層（環境別 Builder／Stack の public メソッド）へ寄せる
> - **外部依存は `parameter.ts` に集約**し、**リソース物理名はなるべく固定しない**（固定するなら prefix ＋ 理由コメント）
> - **Stack は鳥瞰できる可読性に保ち**、**公開インターフェースは最小の型に絞る**（ISP）
> - すべての根拠は **決定論的な構築（正確な `cdk diff`）を最優先する**ことに帰する。判断軸（なぜ）を定め、機械的な実装ルールは [.claude/rules/cdk.md](../../.claude/rules/cdk.md) の領分

## 境界：コードの書式・機械ルール・設計書とは別物

| 対象 | 何を定めるか | どこ |
|---|---|---|
| CDK **コード設計の判断軸** | 環境差分の出し方・命名方針・Stack 抽象化・インターフェース設計（なぜ） | **本ポリシー** |
| CDK **コードの機械ルール** | import 形式・Construct レベル・差分検知の具体 DON'T/MUST | [.claude/rules/cdk.md](../../.claude/rules/cdk.md) |
| インフラ設計**書**（文書） | 全体像の図・設計判断・組織/運用の制約をどう書くか | [iac-infra-design-doc-policy](iac-infra-design-doc-policy.md) |

`.claude/rules/cdk.md` は毎 CDK タスクで自動ロードされる。判断の根拠（なぜ）を本ポリシーに置き、守るべき機械ルールを cdk.md に残すことで、常時ロードの強制力を保ったまま「なぜ」を必要時に引ける。

## 貫く原則：決定論的な構築を最優先する

CDK コード設計のすべての判断は、**`cdk diff` による正確な差分検知と、決定論的（予測可能）なインフラ構築を最優先する**ことに帰する。デプロイ時評価（動的参照・テンプレート内条件分岐）や実行のたびに変わる値に依存すると、差分が正しく出ず、意図しない再作成・データ消失を招く。

この原則を守るための具体的な DON'T/MUST（動的参照・`CfnParameter`・物理名の動的生成の禁止など）は [.claude/rules/cdk.md](../../.claude/rules/cdk.md) の「差分検知ルール」に従う。以下の設計判断は、いずれもこの原則の現れである。

## 1. 環境差分は Stack 内で分岐させない

環境（dev/stg/prd 等）の違いは、Stack の中に条件分岐として撒かず、**差分の種類に応じて集約点へ寄せる**。

| 差分の種類 | どこで表現するか |
|---|---|
| **値**の差分（インスタンスサイズ・prefix など） | `parameter.ts`（環境別パラメータ）に持たせ、Stack へ注入する |
| **構造・振る舞い**の差分（あるリソースの有無・prd のみアラーム等） | Builder 層で表現する。環境別の Builder クラスに分ける／Stack に public メソッドを生やして環境別 Builder から呼び分ける |

**なぜ**：Stack 内に `if (env === 'prd')` を撒くと、その Stack が「どの環境で何を作るか」を一望できなくなり、`cdk diff` の追跡も難しくなる。分岐を Builder へ寄せれば、各 Stack は「1 環境ぶんの宣言」に保て、環境ごとの違いは Builder を見れば分かる。

> [!NOTE]
> TypeScript の `if` 文そのものは禁止ではない（テンプレートに分岐を残さないための正攻法。[.claude/rules/cdk.md](../../.claude/rules/cdk.md) 差分検知ルール参照）。本ポリシーが定めるのは、その分岐を **Stack 内ではなく Builder 層に置く**という位置づけである。

## 2. 外部依存パラメータは `parameter.ts` に集約する

環境ごとに異なる値と、外部からインポートするリソース（手動作成した VPC の ID など）は、**`parameter.ts` を見れば全部わかる**状態にする。

**なぜ**：外部依存が各 Stack に散らばると、「このシステムが何に依存しているか」と「環境を跨ぐ差分」が追えなくなる。窓口を 1 つにすることで、依存の全体像を 1 ファイルで棚卸しできる。

これは [configuration-policy](configuration-policy.md)（構成値を外部化し、供給元を前提にしない）を CDK に落とした具体化である。

## 3. リソース物理名はなるべく固定しない

物理名は既定で **CDK の論理 ID 管理に委ねる**。固定が必要なとき（外部から名前で参照される等）は、Stack 名などを prefix にし、**固定する理由をコメントに書く**。

**なぜ**：物理名の固定は、名前の衝突や「削除→再作成」による事故（ダウンタイム・データ消失）のリスクを生む。決定論的な差分検知を最優先する立場から、名前は CDK に任せるのが既定。固定はその例外なので、理由を残して次に読む人が判断できるようにする。

## 4. Stack は抽象的に保つ

**constructor は全体を鳥瞰できる可読性**にする。リソースが増えたら、L1 Construct などは自作 Construct で抽象化し、まとまった処理はメソッドに切り出す。Stack のレイヤーに全リソースをベタ書きすると見通しが失われる。

**なぜ**：constructor を読めば「この Stack が何を作るか」を追える状態に保つ。これは環境差分を Builder に寄せる（トピック 1）ための前提でもある。

## 5. 公開インターフェースは最小の型に絞る（ISP）

**外部公開インターフェース（props と public メンバー）は、利用者が必要とする最小限の型に絞る**（Interface Segregation）。型によって利用者に許可する操作範囲を制御する。Construct 内部では具体的な L2 Construct を使ってよい。

**なぜ**：公開する型を広く取るほど利用者に余計な操作を許し、結合が強くなる。最小の型だけ公開すれば、内部実装を差し替えても利用者に影響しない。

> 型の選び方（`IBucketRef` / `IBucket` / `Bucket` の使い分け）など具体的な選択基準と実装例は [.claude/rules/cdk.md](../../.claude/rules/cdk.md) の「Interface Segregation」に従う。

## レビュー観点（チェックリスト）

CDK コードをレビューするときは次を確認する。

- [ ] Stack 内に環境の条件分岐が撒かれていないか（値の差分は `parameter.ts`、構造・振る舞いの差分は Builder へ寄っているか）
- [ ] 外部依存・環境固有値が `parameter.ts` に集約されているか
- [ ] リソース物理名を不要に固定していないか（固定するなら prefix ＋ 理由コメントがあるか）
- [ ] 外部公開インターフェースが最小の型に絞られているか（ISP）
- [ ] Stack の constructor が全体を鳥瞰できる可読性か（自作 Construct・メソッドで抽象化されているか）

---

> [!IMPORTANT]
> 本ポリシーで最も重要な原則は次の 3 つ。
> - **決定論を最優先する**：正確な `cdk diff` を壊さない設計を選ぶ。命名・分岐・参照はすべてここに帰する
> - **環境差分は Stack から追い出す**：値は `parameter.ts`、構造・振る舞いは Builder 層へ
> - **判断軸はここ、機械ルールは cdk.md**：なぜを本ポリシー、守るべき具体ルールを cdk.md に置く
