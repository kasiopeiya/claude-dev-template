# アプリケーションロギングポリシー

ロギング設計・実装の「良い／悪い」を判断するための基準を示すドキュメント。**ログをいつ・何を・どのレベルで出力するかを判断する開発者・AIエージェントが、統一された基準を知りたいとき**に参照する。

> [!IMPORTANT]
> **TL;DR（このポリシーの決定事項）**
> - ログは**少なく・鋭く**保つ。多すぎるログはないのと同じくらい役に立たない
> - **エラー**と**外部依存との通信**は必ずログに出す。ERRORは詳細を、INFOはキー識別子のみに絞る
> - 異常ケースには**エラーコード**とキー識別子を必ず入れる。個人情報・認証情報は絶対に含めない

---

## いつログを出すか

### ログの多さは「ログがない」と同じくらい役立たない

「プログラマが知るべき97のこと」が指摘するとおり、**ログの情報があまりに大量だと、ログがないのと同じくらい役に立たない**。大量のログはノイズとなり、本当に必要な情報を見つけるコストを増大させる。ログ基盤（CloudWatch Logs 等）の課金コストも増える。

ログを出す前に「**このログは誰がいつ何を調べるために見るか**」を問い、答えられないなら出さない。

### 必ず出力すべき2カテゴリ

| カテゴリ | 理由 |
|---|---|
| **エラー** | 問題が発生したことを認識するため。重要でないエラーはエラーが起きたことが認識できれば十分で、必ずしも詳細を書く必要はない（「プログラマが知るべき97のこと」より） |
| **外部依存との通信** | 外部サービス・DB・APIへの呼び出しは問題が発生する可能性が高く、トラブルシューティングで必ず参照される（同書より） |

### 出力を検討する場面

- システムの状態が変化する重要なビジネスイベント（注文確定・決済完了・ユーザー登録など）
- 非自明な条件分岐（「なぜこの分岐に入ったか」が後から分からなくなりそうな箇所）

### 出力しない場面

- ループ内の繰り返し処理（1件ごとにログを出すと件数×ログになる）
- 正常ルートの単純な進捗（「〜を開始しました」「〜が完了しました」の連発）
- 上流ですでに出力済みのエラーの再出力（次節「アンチパターン」参照）

## ログレベルの使い分け

ログレベルは「重大度」ではなく「**誰がどの状況で見るか**」で使い分ける。

| レベル | いつ使うか | 例 |
|---|---|---|
| **FATAL** | システムが継続不能な致命的障害（プロセスが落ちるレベル） | DBへの接続が完全に失敗、必須設定ファイルが存在しない |
| **ERROR** | 処理が失敗し、呼び出し元への影響がある | 外部API呼び出し失敗、ビジネスルール違反による処理中断 |
| **WARN** | 処理は続けられたが注意が必要な事態 | リトライで回復した、閾値に近づいている |
| **INFO** | 正常動作の要所（障害調査時のタイムラインになる） | 外部API呼び出し（リクエスト概要・レスポンスステータス）、ビジネスイベント完了 |
| **DEBUG** | 開発・調査時にのみ必要な詳細（本番では無効化） | 入力値の詳細、中間計算結果 |

> [!WARNING]
> **（AI・必須）** ERROR と WARN の境界で迷ったら「処理が失敗して呼び出し元にエラーを返すか」で判断する。返すなら ERROR。処理を続けているなら WARN。
>
> DEBUG レベルのログを本番で有効にしてはならない（コストとノイズの両面で）。

---

## ログに含めるべき内容

### ERRORは詳細を出す。INFOはキー情報のみに絞る

ログに含める情報量はレベルによって変える。

| レベル | 含める内容 |
|---|---|
| **ERROR / FATAL** | キー識別子・エラーコード・エラーメッセージ・スタックトレース・原因究明に必要なコンテキスト（ステータスコード・パラメータ等）。調査に必要な情報を出し惜しみしない |
| **WARN** | キー識別子・何が起きたか・どう回復したか |
| **INFO** | キー識別子のみ。詳細（ペイロード・パラメータ等）は含めない |
| **DEBUG** | 制限なし（本番では無効） |

```typescript
// ✅ ERROR：原因究明に必要な情報を詳細に出す
logger.error('外部API呼び出しが失敗しました', {
  errorCode: 'EXT_API_001',
  orderId,
  userId,
  endpoint: '/payments',
  statusCode: response.status,
  responseBody: response.data,
  error: err.stack,
})

// ✅ INFO：キー識別子のみ（詳細は不要）
logger.info('注文処理が完了しました', { orderId, userId })

// ❌ INFO に詳細を詰め込む（コスト大・ノイズになる）
logger.info('注文処理が完了しました', {
  orderId, userId, orderItems, shippingAddress, totalAmount, ...
})
```

### キー識別子を必ず入れる

**ログが役立つのは「何について・いつ起きたか」が追跡できるときだけ**。処理対象データのキーとなる識別子が含まれていなければ、障害調査でそのログは使えない。

```typescript
// ❌ 誰の・どの注文の話か分からない
logger.error('決済処理が失敗しました')

// ✅ 追跡できる
logger.error('決済処理が失敗しました', { errorCode: 'PAY_001', orderId, userId })
```

含める識別子は「このログを見たとき、何で検索するか」で決める（例：リクエストID・ユーザーID・リソースID）。

### 異常ケースにはエラーコードを必ず入れる

エラーコードを付与すると、同じ種類のエラーの集計・検索・アラート設定が可能になる。エラーメッセージ（人間が読む文章）は変わっても、コード（システムが識別する値）は変わらないため追跡が安定する。

> [!IMPORTANT]
> **（AI・必須）** エラーコードの命名体系はプロジェクトの設計書で定義する。このポリシーでは「異常ケースには必ずエラーコードを付ける」原則のみを定める。

### 入れてはいけない情報

| カテゴリ | 理由 |
|---|---|
| 個人情報（氏名・メールアドレス・電話番号・住所） | 法令・規約上の義務。漏洩リスク |
| 認証情報（パスワード・トークン・APIキー・セッションID） | セキュリティリスク |
| 大きなペイロード全体（リクエスト・レスポンス全文） | コスト大、かつ重要情報が埋もれる（INFOレベルでは特に厳守） |

> [!WARNING]
> **（AI・必須）** ログは本番環境でも出力される。個人情報・機密情報を含むログはコードレビュー時も確認する。IDに置き換えられないか（氏名→ユーザーIDなど）を先に検討する。

## 構造化ログ（JSON形式）を使う

ログは構造化（JSON）形式で出力する。人間が読むフリーテキストよりも、CloudWatch Logs Insights 等での検索・集計・フィルタが容易になる。

```typescript
// ❌ フリーテキスト（検索・集計が困難）
logger.error(`orderId=${orderId} の決済処理が失敗しました: ${err.message}`)

// ✅ 構造化（フィールドで検索・集計できる）
logger.error('決済処理が失敗しました', {
  errorCode: 'PAY_001',
  orderId,
  userId,
  error: err.message,
})
```

> [!TIP]
> **（AI・推奨）** メッセージ文字列はイベントの種類を表す固定文言にする（`'決済処理が失敗しました'`）。IDや可変値はフィールドに分離する。固定文言で絞り込み、フィールドで詳細を確認できる構造にする。

## アンチパターン

| アンチパターン | 何が悪いか | 対処 |
|---|---|---|
| エラーを飲み込んでログだけ出す | 呼び出し元にエラーが伝わらず、壊れた状態で処理が続く | エラーはログ＋throwをセットで行う（[application-design-policy「失敗は投げて伝えよ」](application-design-policy.md)参照） |
| スタックを伝わるたびに同じエラーをログ出力 | 1つのエラーが何倍にも膨れてノイズになる | エラーは発生箇所か最終ハンドラのどちらか1箇所だけで出す |
| ループ内で1件ごとにINFOを出す | 件数×ログになりコストが膨れ、重要な情報が埋もれる | 開始時・完了時（件数サマリ）のみに絞る |
| メッセージに可変値を文字列結合する | ログの集計・検索が困難になる（メッセージが毎回変わる） | 可変値は構造化フィールドに分離する |

## レビューチェックリスト

| 観点 | 確認する問い |
|---|---|
| 量 | 不必要に多くのログを出していないか。「誰がいつ何を調べるために見るか」が答えられるか |
| エラー出力 | エラーは必ずログに残っているか（飲み込まれていないか） |
| 外部依存 | 外部サービス・DB・APIとの通信はログに記録されているか |
| ログレベル | 適切なレベルが選ばれているか。ERRORとWARNの境界は正しいか |
| 情報量 | ERRORは詳細を出しているか。INFOはキー識別子のみに絞られているか |
| キー識別子 | 「何についての処理か」が追跡できる識別子が含まれているか |
| エラーコード | 異常ケースにはエラーコードが付与されているか |
| 機密情報 | 個人情報・認証情報がログに含まれていないか |
| 構造化 | JSON形式で出力されているか。可変値がメッセージではなくフィールドに分離されているか |
| 重複出力 | 同じエラーがスタックを伝わるたびに重複出力されていないか |
