# 手順書作成ポリシー

手順書（セットアップ・デプロイ・運用・障害対応などの実行手順書、いわゆる Runbook）を書く・レビューする開発者／運用者が、**良い手順書の統一基準を知りたいとき**に参照するドキュメント。

> [!IMPORTANT]
> **TL;DR（このポリシーの決定事項）**
> - **最良の手順書は「書かずに済む手順書」**。書く前に「自動化できないか？」を必ず問い、できるなら自動化する（手順書は次善策）
> - 良い手順書とは、著者不在でも読者が目的の結果へ到達できる文書（＝**再現性**が北極星）。再現性は7条件の総和で、一つ欠けても再現できない
> - 安定した手段（CLI最優先・画面ショット回避・実行環境を一つに固定）で書き、通し検証と鮮度管理で腐らせない

## まず問う：その手順書は本当に必要か？（自動化できないか？）

**最良の手順書は、書かずに済む手順書である。** 手順書は「再現性を人間の手作業に依存させる」手段であり、本質的に腐り（持続性の宿命）、人為ミスを招く。同じ再現性を**自動化**（スクリプト・IaC・CI/CD）で達成できるなら、そちらが上位互換である——コードは実行可能で、常に最新で、差分が取れ、人手を介さない。**自動化されたものに手順書は要らない（コードが手順そのものになる）。**

だから手順書を書き始める前に、必ず次を問う。

| 問い | Yes なら |
|---|---|
| この手順は自動化できないか？（スクリプト・IaC・CI/CD） | 手順書ではなく自動化を作る |
| なぜ手作業のまま残すのか、説明できるか？ | 説明できないなら自動化に倒す |

手順書を書いてよいのは、自動化が**できない・割に合わない**ときの次善策としてである（例：頻度が低く自動化コストが見合わない／外部都合で手作業が避けられない不可逆操作）。

> [!IMPORTANT]
> **（AI・必須）** 手順書のレビューでも、まず「これは自動化すべきではないか？」を問う。書く前提・残す前提で読み始めない。以降の原則は、*それでも手順書が必要だと判断したとき*にのみ適用される。

## 基本原則

### 1. 良い手順書とは、著者がそばにいなくても読者が確実に「目的の結果」へ到達できる文書である

設計書が*理解*を渡すのに対し、手順書は*行動と再現*を渡す。決定的な制約は「**著者不在で成立しなければならない**」こと——読者の疑問にその場で答えられない。だから**再現性**が北極星になる。

再現性は単独では成り立たず、次の条件の総和で決まる。どれか一つでも欠ければ、手順は間違い、再現できない。

| 条件 | 一言 | 欠けると起きる失敗 |
|---|---|---|
| 完全性 | 漏れがない | 前提・手順が抜け、途中で詰まる |
| 一意性 | 迷いがない | 「適宜」「必要に応じて」で解釈が割れる |
| 正確性 | 本当に動く | 机上で書かれ、通すと動かない |
| 検証可能性 | 合っているか分かる | 誤った状態のまま先へ進む／不安で止まる |
| 自己完結性 | それだけで足りる | 文書外の暗黙知に依存し、当人しか実行できない |
| 持続性 | 腐りにくい・保守され続ける | 画面ショットやバージョンが陳腐化し、再現不能に |
| 適応可能性 | なぜが分かる | 想定外が起きたとき、判断・回復できない |

> [!IMPORTANT]
> **（AI・必須）** 以降の原則は、すべてこの条件のいずれかを担保するために存在する。**メンテナンスし続けること（持続性）もこの一部**——腐った手順は間違った手順であり、再現性を失う。すべては再現性へ一本に繋がる。

### 2. 対象読者と前提条件を定める（→自己完結性）

誰が・何が揃った状態で実行するのかを定めなければ、自己完結しない。対象読者は役割名だけ（例：開発者）では不十分。「◯◯な人が△△したいとき」という想定シーンを一文で書く。

| | 例 |
|---|---|
| ❌ 弱い | インフラ担当者向け |
| ✅ 強い | 既定の実行環境を初めて使う開発者が、dev環境に初期データを投入したいとき |

読者像が定まると「何を前提として書き、何を補足すべきか」が決まる。読者が当然知っていることは書かず、知らないことは前提条件（原則7）に明示する。

### 3. なぜその手順が必要かを書く（→適応可能性）

各手順に**なぜそのステップが要るか**（飛ばすと何が壊れるか）を添える。機械的な操作列だけでは、想定外が起きたとき読者は判断も回復もできない。

> [!TIP]
> **（AI・推奨）** 設計書の「なぜ」が*なぜこの判断をしたか*なのに対し、手順書の「なぜ」は*なぜこのステップが要るか*。粒度が違う。

```bash
# 旧プロセスが残ると二重起動でロックするため、先に停止する
<service-cli> stop --name worker
```

### 4. シンプルで短く、迷いなく辿れる（→一意性）

長い手順は読まれず、飛ばされ、再現性を失う。各ステップ・各語は、その箇所の目的に奉仕するものだけを残す。

- 「適宜」「必要に応じて」「よしなに」など、解釈が読者に委ねられる語を排除する（一意性を壊す）
- 専門用語は使ってよいが、言い回しは平易に。硬い動詞を和語に（「実施する」→「やる」）

> [!TIP]
> **（AI・推奨）** **削除テスト**：その語・行を消したとき、読者は実行に困るか？ 困らないなら消す。

## 手順の書き方

### 5. 安定した手段で手順を書く（→一意性・持続性）

手順書の再現性は「どの手段で書くか」で大きく決まる。**安定した手段＝時間が経っても変わりにくく、コピペで再現できる手段**を選ぶ。

| 手段 | 安定性 | 手順での扱い |
|---|---|---|
| CLIコマンド | 高（コピペで再現・差分が取れる） | ✅ 最優先 |
| 管理コンソール等のGUI画面ショット | 低（UIが頻繁に変わり陳腐化） | ❌ 主軸にしない |

> [!WARNING]
> **（AI・必須）** クラウドの管理コンソール等の画面ショットは頻繁に変わり、すぐ陳腐化する。手順の主軸に据えてはならない。GUI操作がどうしても必要な場合のみ、文章で要点を簡潔に書く。

具体的な書き方:

- **実行環境を一つに固定し、それを既定とする。** 指示がない限り、その環境のコマンドで書く（既定が固定されれば、OS別の場合分けが原則不要になる）。本プロジェクトの既定は Linux / CloudShell
- **コピペでそのまま動く形にする。** 可変部は `<...>` で明示し、固定部と混ぜない
- **具体コマンドはハードコードしてよい**（実行可能性が最優先）。抽象化して実行できなくなっては本末転倒。**変わりやすい値だけ**を可変部 `<...>` や変数に外出しし、バージョン等の陳腐化は鮮度管理（原則12）で担保する

```bash
# ✅ 可変部を明示。固定部はそのまま動く
<cli> upload ./asset/init.sql <destination>/init.sql
```

### 6. 1ステップ＝1アクション、番号付きで順序を固定する（→一意性）

手順書は順序が命。1ステップに複数操作を詰めると、どこで失敗したか分からず再現性が崩れる。番号付きリストで順序を固定し、1ステップ＝1アクションに保つ。

| | 例 |
|---|---|
| ❌ NG | バケットを作成し、ポリシーを付与してからファイルを置く |
| ✅ OK | 1. バケットを作成する → 2. ポリシーを付与する → 3. ファイルを置く |

### 7. 前提条件を冒頭に明示する（→完全性）

実行前に揃っていなければならないもの（権限・ツール・事前状態・対象環境）を冒頭にまとめる。前提が抜けると読者は途中で詰まる。

<details>
<summary>前提条件に書く項目の例</summary>

- 必要な権限・ロール（例：dev環境への書き込み権限）
- 必要なツールとバージョン（例：プロジェクト指定のCLI）
- 事前に完了していること（例：基盤リソースが構築済み）
- 対象環境（例：dev のみ。本番では実行しない）

</details>

### 8. 各ステップに「期待される結果・確認方法」を添える（→検証可能性）

各ステップの後、正しく進んでいるかを読者自身が判定できるようにする。これが無いと、読者は誤った状態のまま先へ進むか、不安で止まる。

```bash
<service-cli> get-status --name worker
# 期待される結果: STOPPED が返れば停止完了
```

### 9. 予期される失敗と対応策を添える（→適応可能性）

手順どおりに進めても失敗は起きる。**よくある失敗**（権限不足・リソース未作成・バージョン不一致など）と、その見分け方・対応策を表で添える。これが無いと、読者は失敗した瞬間に手が止まり、著者不在では回復できない（再現性が切れる）。

原則8が「正常に進んだ確認」なら、本原則は「異常時の回復」——両者は対になる。

| 症状（返るエラー等） | 原因 | 対応 |
|---|---|---|
| `AccessDenied` が返る | 実行ロールの権限不足 | 前提条件（原則7）の権限を確認し、不足していれば付与を依頼する |
| `NoSuchBucket` が返る | 基盤リソースが未作成 | 事前手順（基盤構築）が完了しているか確認する |

書き方の注意:

- **想像で網羅しない。** 通し検証（原則11）や実運用で**実際に遭遇した失敗**を蓄積する。机上で並べた失敗リストは腐りやすく、当たらない
- **高頻度・高影響に絞る。** すべての失敗を書くと読まれない（原則4のシンプルさ）

### 10. 見出し・トピックセンテンスに要点を置く（→一意性）

読者は見出しと冒頭1〜2文しか読まない。見出しだけで「このセクションで何を実行するか」が分かるようにする。

## 手順書の品質保証

### 11. 通しで1回実行して検証してから公開する（→正確性）

手順書は机上で書かず、**書いた手順を実際に最初から最後まで通して実行し、目的の結果に到達することを確認してから公開する**。書いた本人が動かしていない手順は、ほぼ必ずどこかで動かない。

### 12. 最終動作確認日・確認した環境を残す（→持続性）

手順書は書いた瞬間から腐る。**いつ・どの環境で通しの動作確認をしたか**を残し、読者が鮮度を判断できるようにする。

```markdown
> 最終動作確認: 2026-06-23 / dev環境 / <ツール> <バージョン>
```

## 視覚表現

### 13. 構造・関係・フローは図表にする

手順全体の流れ・分岐・登場リソースの関係は、文章で説明するより図表にする。画面ショット（原則5で回避）とは別物で、**変わりにくい構造の可視化は推奨**される。

- 全体の流れ → フローチャート
- 条件による分岐 → 表 or フローチャート
- 登場するリソースの関係 → 関係図

> [!NOTE]
> **（AI・必須）** 図は `/design-doc-mermaid` スキルで作成する（自前でMermaid記法を書き起こさない）。

### 14. 視覚的に文字を強調する

`コマンド・パス・リソース名` はバッククォートで囲む。注意・警告は Admonition 記法（`> [!WARNING]` 等）で示す。補足・背景は本文を伸ばさず `<details>` トグルに畳む。

---

> [!IMPORTANT]
> 本ポリシーで最も重要な原則は次の3つ。
> - **再現性が北極星**：著者がそばにいなくても、読者が確実に目的の結果へ到達できること
> - **安定した手段で書く**：CLI最優先・画面ショット回避・実行環境を既定に固定・コピペ可動
> - **腐らせない**：通し検証してから公開し、最終動作確認日で鮮度を担保する
