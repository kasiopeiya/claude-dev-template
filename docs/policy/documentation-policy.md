# ドキュメント作成ポリシー

抽象的な方針を示すドキュメント。仕様駆動開発において、AIおよび人間がドキュメントを作成・編集する際の指針として参照する。

## 基本原則

### ドキュメンテーションとプログラミングは本質的に同じである

両者は多くの本質を共有する。プログラミングで重要な原則は、ドキュメンテーションにもそのまま当てはまる。

| 観点       | プログラミング                 | ドキュメンテーション         |
| ---------- | ------------------------------ | ---------------------------- |
| 目的       | SRP原則、アクターと目的が重要  | 誰に何を伝えるかが全て       |
| ネーミング | 変数名・関数名が可読性を決める | 見出し・用語が可読性を決める |
| 保守性     | 変更容易性を考慮した設計       | 更新容易性を考慮した構造     |
| 重複       | DRY原則                        | DRY原則                      |
| 抽象化     | 適切な粒度での分割             | Progressive Disclosure       |
| シンプルさ | 短く明快なコードが良い         | 短く明快な文章が良い         |

> [!TIP]
> **（AI・推奨）** 迷ったら「これがコードならどう書くか？」と自問する。

### 対象読者と目的が全てを決定する

対象読者と目的が異なれば、書く内容も書き方も異なる。ドキュメント作成において最も重要な本質は「対象」と「目的」。

#### 対象読者は「想定シーン」として具体化する

対象読者は役割名だけ（例：開発者）では不十分。「◯◯な人が△△したいとき」という想定シーンを一文で書く。読者の前提知識と目的をまとめて表現することで、捜索者が「自分のための文書か」を即座に判断できる。

|         | 例                                                                            |
| ------- | ----------------------------------------------------------------------------- |
| ❌ 弱い | アプリケーション/インフラ開発者                                               |
| ✅ 強い | RDB設計の経験はあるがAuroraは初見の開発者が、設計思想と選定理由を知りたいとき |

> [!TIP]
> **（AI・推奨）** 具体化の判断基準は「その読者像によって書き方が変わるか？」である。用語解説の要否や説明の深さが変わるなら書く。

#### レビュー：設定した対象読者と表現が合っているか確認する

文中の専門用語・前提知識を、設定した読者像が「知っている」前提と照合する。ズレは双方向に起きるので両方見る。読者像が明示されていなければ、まずその不在を指摘する。

| ズレの方向 | 症状                                     | 直し方               |
| ---------- | ---------------------------------------- | -------------------- |
| 説明不足   | 読者が知らないはずの用語を補足なしで使用 | 補足レイヤーに逃がす |
| 過剰説明   | 読者が当然知っている用語をくどく説明     | 削る                 |
| 深さのズレ | 目的に対し浅すぎる／細かすぎる           | 必要な深さに合わせる |

### 良いドキュメントとは読者目線のドキュメントである

読者が抱くであろう疑問を先回りして提示する。

例：「外部接続用クラスは単体テストを実施しない」と書いたら、次の見出しを `なぜ外部接続用クラスは単体テストを実施しないのか？` とする。

### "工事中"のドキュメントは永遠に工事中である

**「まだ作成中」「TODO」「整備途上」「暫定」といった経過・状態の記述を、ドキュメント本文に直接書いてはならない**（人間が明示的に指示したときを除く）。過去の経緯（旧仕様・改定の履歴）も同様に書かない——本文は常に「現在の仕様」だけを言い切る。

理由は単純で、**こうした記述はほぼ必ず、修正されないまま残り続ける**から。「工事中」と書かれたページが永遠に工事中なのは、どこにでもある光景。いつ外すべきかは書いた本人すら分からなくなり、注記だけが腐って残る。

未完了の作業が本当にあるなら、追跡できる場所（Issue）に逃がし、本文には**Issue番号だけ**を残す。

| 場面                       | ❌ 本文に直接書く              | ✅ 推奨                                              |
| -------------------------- | ------------------------------ | ---------------------------------------------------- |
| まだ書けていない節がある   | 「※この節は作成中」            | Issue化し `TODO(#300)` のように番号で参照する        |
| 仕様が暫定・今後変わりうる | 「暫定。今後変更の可能性あり」 | 現在の仕様として言い切る。変わったらそのとき更新する |

> [!IMPORTANT]
> **（AI・必須）** 経過・状態を本文に書いてよいのは、人間から明示の指示があったときだけ。指示がなければ書かない。「親切のつもりの注記」を、腐ったまま残る負債にしない。

## 構造設計

### Progressive Disclosure な設計にする

AIおよび人間が、必要な時に必要なドキュメントだけを読み込める構造にする。1つのドキュメントに情報を詰め込みすぎると、コンテキストを圧迫するため避ける。

|       | 例                                                                                            |
| ----- | --------------------------------------------------------------------------------------------- |
| ❌ NG | テスト設計書を1つ作り、全ての情報をまとめる                                                   |
| ✅ OK | 単体テスト・統合テスト・E2Eテストそれぞれの設計書を作り、ハブとなるテスト仕様書からリンクする |

分割検討の目安：600行

1ドキュメントが600行を超えたら、分割を検討する。600行は閾値ではなく目安であり、超えたから必ず分割するわけではない。

分割を検討する際の判断基準
・独立した関心事が複数含まれているか
・読者が一度に全てを読む必要があるか
・特定セクションだけを参照するユースケースがあるか

分割する場合は、ハブとなるドキュメントから個別ドキュメントへリンクする構成にする。

> [!TIP]
> **（AI・推奨）** 600行を超えたドキュメントを編集・作成する際は、分割の要否をユーザーに提案すること。

### 詳細な記載よりシンプルで短い方が良い

いくらわかりやすい文章でも、長い文章は読まれない。プログラミングと同じで、シンプルで短い記載が良いドキュメント。

> [!TIP]
> Blaise Pascalは「もし私にもっと時間があるなら、あなたにもっと短い手紙を書いただろう」と記した。短く書くことは、長く書くことより難しい。

AIは「正しい情報」「網羅した情報」を良いものと誤解し、読者が今必要としない情報まで盛り込みがち。**各文・各行・各語は、その箇所の目的に奉仕するものだけを残す。** 正しいだけの情報・読者の判断を変えない修飾・他の箇所で説明済みの属性は、文章を伸ばすノイズでしかない。

| ❌ NG（冗長）                                                               | ✅ OK（必要十分）                                          |
| --------------------------------------------------------------------------- | ---------------------------------------------------------- |
| DB 初期化 Lambda（DataInitializer・SharedResourceStack 配下）を Invoke する | DB 初期化 Lambda を Invoke する                            |
| ある属性（所属スタック・分類など）を登場のたびに繰り返す                    | その属性を主題とする箇所で一度説明し、他所は名前で参照する |
| 表に該当する全項目を機械的に列挙する                                        | 図や本文で伝わらない、判断に効く項目だけに絞る             |

> [!TIP]
> **（AI・推奨）** **削除テスト**：その語・行・項目を消したとき、この箇所の読者は困るか？ 困らないなら消す。「具体的な方が親切そう」は残す理由にならない。

### 短さと丁寧さが衝突したら、本文を伸ばさず補足レイヤーに逃がす

専門用語の補足を入れたいが本文が長くなる——この衝突は「本文に書く or 省く」の二択で考えるから起きる。第3の道として、本文は短いまま、補足を別レイヤー（トグル `<details>`・用語集へのリンク・脚注）に逃がす。熟練者は読み飛ばし、初学者は開く。両者の要求を同時に満たせる。

そもそも補足の要否は対象読者が決める（「対象読者と目的が全てを決定する」参照）。文書ごとに読者像を定め、その読者が知らない語だけ補足する。全文書を初学者向けに揃える必要はない。

補足する語は次の2条件のAND（両方満たす）で絞る。片方でも欠けたら省く。

- その文書の対象読者が知らない可能性が高い
- その語を理解しないと読者が判断・行動できない

### 「どうやって」より「なぜ」の方がずっと大事

`なぜ` が書かれていないドキュメントほど不親切なものはない。「なぜ」のない設計書では、変更して良いのか判断できず読者が安心できない。

> [!IMPORTANT]
> **（AI・必須）** 設計判断には必ず理由を添えること。

### DRY原則を遵守する、ただし最重要事項は繰り返す

重複記述はメンテナンス性を損なうためDRY原則を遵守する。ただし、最も重要な事項についてはあえて繰り返すことで重要性を強調する。

DRYは1文書内だけでなく**文書間でも守る**。AIは他文書の存在を意識せず同じ説明を書きがちなので、書く前に「この説明はどの文書の主題か」を判断する。その主題の文書に1度だけ書き、他文書からはリンクする（Single Source of Truth）。既に他文書で説明済みの事項は、書かずにリンクする。

### ドキュメントもメンテナンス性が重要であると理解する

ドキュメントもコードと同様に保守対象。関数名・ファイル名・クラス名などの直接記述（ハードコード）は、リネーム時の修正コストになる。一方で、抽象化しすぎるとかえって読者の理解を妨げる。

判断基準は「メンテナンスコストを上回る読者価値があるか」。あるなら書く、ないなら書かない。

変更が予想されるものはハードコードしない
・ファイルパス、ファイル名
・関数名、クラス名、変数名
・モジュール構成、ディレクトリ構造の詳細
・**件数・列挙の転記**（「5つの問い」のような個数、別セクション／別文書にある項目リストの再掲）。項目が増減するたび全転記先の修正が必要になるため、件数を書かず、本体（SSOT）への参照に置き換える（例：「5つの問い」→「早見表の問い」）

ただし、他ファイルへのリンク（相対パス参照）は、ナビゲーション目的のため許容する。

| 状況                                                 | 判断                                       |
| ---------------------------------------------------- | ------------------------------------------ |
| 抽象表現で十分に意図が伝わる                         | ハードコードしない                         |
| 具体名がないと読者が実装を追えない                   | ハードコードする                           |
| 具体名そのものが議論の対象（API仕様・ADR等）         | ハードコードする                           |
| 他のドキュメントやファイルへのリンク                 | 許容する                                   |
| 単に「具体的な方が親切そう」という理由               | ハードコードしない                         |
| 件数・他所の列挙を本文に転記する（増減で更新が要る） | 転記せず、本体（SSOT）への参照に置き換える |

> [!TIP]
> **（AI・推奨）** 判断に迷ったら「この名前が変わったとき、このドキュメントを更新する価値があるか？」と自問する。Yesなら書く、Noなら抽象化する。

> [!IMPORTANT]
> **（AI・必須）** 個別判断に迷う場合は、勝手に決めず、ユーザーにハードコードするかどうか確認すること。

## 視覚表現

### 可視化されていないドキュメントは怠慢である

文章はほとんど読まれないものと考える。人間はほとんど図や表しか見ていない。図やテーブルを使ってなるべく可視化する。

> [!IMPORTANT]
> **（AI・必須）** ドキュメント作成・レビュー時は、必ず「図や表で可視化できる箇所はないか」を検討すること。検討した上で不要と判断した場合はその旨を明示する。

> [!IMPORTANT]
> **（AI・必須）** **良い図とは、文章を読まずに構造（関係・流れ・階層・分岐）が一目で掴める図である。** 文章をそのまま箱に詰め替えただけの図は、可視化ではなくノイズなので作らない。

### 図にする前に「逆変換テスト」を通す

可視化を増やす圧力（前項）に対する質のブレーキ。**削除テスト・表化テストと対をなす、図のための捨てる基準**。

> **逆変換テスト**：その図を箇条書き（単文の羅列）に戻したとき、情報が失われないなら、それは図ではなく箇条書き。図の価値は**2次元配置でしか表せない構造**——分岐・合流・ループ・並行・多対多・階層・状態遷移——にある。一直線（A→B→C）に箱を並べただけの図は文章の言い換えにすぎないので、箇条書きか表に戻す。

| 観点   | 図にする価値あり               | 価値なし（文章の言い換え） |
| ------ | ------------------------------ | -------------------------- |
| 線の形 | 分岐・合流・ループ・並行がある | 一直線（A→B→C）            |
| 関係   | 多対多・階層・状態遷移         | 1対1の列挙                 |
| 逆変換 | 箇条書きに戻すと情報が落ちる   | 完全に復元できる           |

### 行と列に収まる構造はデフォルトで表にする（シンプルでも）

可視化の既定値を反転する。**行と列に収まる構造は、まず表にする。** 文章で書くのは表にできない／不自然なときだけで、その場合は文章を選んだ理由を一言残す。「文章で書いて、表も検討する」ではなく「表で書いて、表にできないものだけ文章にする」という順序にすることで、ラクな方（文章）に倒れる癖を断つ。

> [!WARNING]
> **（AI・必須）** 「表にする」は可視化の唯一形ではない。**関係・時系列・手順の流れ・階層・状態遷移は、表ではなく図にする**（後述「可視化を検討すべきトリガー」で形を選ぶ）。順序・分岐・流れがあるものを表に押し込めない。表化の対象は、行と列に収まる静的な構造に限る。

表が向く「行と列に収まる構造」＝次のどれかに当てはまる（**2項目でも該当する**）：

| 構造のパターン          | 例                                         |
| ----------------------- | ------------------------------------------ |
| 項目 × 属性             | 「AはX、BはY」の形                         |
| 2つ以上の対比・分類     | 旧方式と新方式、許可と禁止                 |
| 静的な条件 → 結果の対応 | 入力Xなら結果Y（分岐・順序を伴わないもの） |

> [!IMPORTANT]
> **（AI・必須）** シンプルさ・短さは表にしない理由にならない。むしろ2行2列の小さな構造こそ、表なら一瞬で読める。「短いから文章で十分」と感じたら、それは表にすべきサインである。

> [!TIP]
> **（AI・推奨）** **表化テスト**（削除テストと対をなす）：「この内容を『項目 → 属性』に分解できるか？」分解できるなら文章にせず表にする。逆に、単一の主張・理由・流れ（narrative）は分解できないので文章のままでよい。これが文章と表の線引きになる。

可視化を検討すべきトリガー
・複数の要素が関係し合う説明（→ 関係図、シーケンス図）
・時系列・手順の説明（→ フローチャート、シーケンス図）
・分類・比較・対比（→ 表）
・階層構造、ディレクトリ構造（→ ツリー図、Mermaid）
・状態遷移（→ 状態遷移図）
・2項目以上の対比・属性列挙（→ 表。2項目でも表にする）

レビュー時の手順

1. 既存ドキュメントの各セクションについて、上記トリガーに該当する箇所を全て洗い出す
2. 該当箇所ごとに、追加すべき図・表の種類と内容を提案する
3. ユーザーの承認を得てから追加する

レビュー報告のフォーマット

| セクション名   | 可視化提案の有無 | 提案内容                                         |
| -------------- | ---------------- | ------------------------------------------------ |
| 例：認証フロー | あり             | シーケンス図でリクエスト・レスポンスの流れを表現 |
| 例：用語定義   | なし             | 既に表形式のため不要                             |

> [!TIP]
> **（AI・推奨）** 「文章で書けているからOK」ではなく「図にした方がもっと伝わるのでは？」を基準に判断する。

### 図の作成手段の優先順位

| 優先度 | 手段        | 見やすさ | AI編集 | 備考                    |
| ------ | ----------- | -------- | ------ | ----------------------- |
| 1      | Mermaid     | ◎        | ◎      | 第一選択                |
| 2      | ASCIIアート | △        | ◎      | Mermaidで表現困難な場合 |
| 3      | drawio      | ◎        | ○      | 手動でのPNG変換が必要   |

### 視覚的に文字を強調する

Markdown記法の `**太字**` は実際のViewerではわかりにくい。特に重要な箇所は次の手段で強調する。

- バッククォート： `重要キーワード`
- Admonition記法（GitHub Flavored）
- HTMLタグ

> [!NOTE]
> ViewerはGitHub Flavored Markdownを前提とする。

### Admonition（callout）の先頭に「宛先・強制力」ラベルを置く

`[!IMPORTANT]` / `[!TIP]` などの callout は、人間向けの強調とAIへの行動指示が混ざりやすい。**行動を促す callout は、本文の先頭を `**（宛先・強制力）**` で始め、誰がどの強さで従うべきかを一目で示す。** callout の種別（IMPORTANT/TIP）はAIにとって描画ではなくテキストの意味として効くため、宛先をラベルで言い切ることで取り違えと指示の素通りを防げる。

| 要素   | 取りうる値                    |
| ------ | ----------------------------- |
| 宛先   | AI / 人間 / AI・人間          |
| 強制力 | 必須（MUST） / 推奨（SHOULD） |

両対象なら `**（AI・必須 / 人間にも）**` のように併記する。単なる補足・前提（`[!NOTE]` 等で行動を促さないもの）にはラベル不要。

```markdown
> [!IMPORTANT]
> **（AI・必須）** 仕様にない依頼には即着手せず、まず仕様にあるか確認する。
```

## 文章の書き方

### 見出しとトピックセンテンスに全てを注ぐ

人間は目次の見出しと、各見出し直後の1〜2文までしか読まないというデータがある。見出しの内容と冒頭1〜2文に、全てのエッセンスを詰め込む。

見出しは多少長く説明的でも構わない。良いドキュメントとは見出しだけで内容を想像できるドキュメント。

例（書き出しは違っても、どちらも結論を冒頭に置いている）：

```markdown
## なぜクリーンアーキテクチャを採用したのか？

ひとことで言うと、変更に強い設計にすることを、ほかの何よりも優先したからです。...
```

```markdown
## 外部接続用クラスはなぜ単体テストしないのか？

応答をこちらで制御できず、本物の不具合ではなく接続状態でテストが落ちてしまうからです。...
```

### 見出しに連番を振らない

見出しを `## 1. 目的` のように番号付きで書かない。項目を1つ挿入・削除するだけで以降の全番号と、他文書からの参照（`§3` など）を書き換える羽目になるからだ。番号が伝えるのは並び順だけで、その情報量は書き換えコストに見合わない。

| 例                                   | 評価                                |
| ------------------------------------ | ----------------------------------- |
| `## なぜ番号を振らないのか？`        | ✅                                  |
| `## 3. なぜ番号を振らないのか？`     | ❌ 節が増減するたび全体を直す       |
| `[命名規則](typescript.md#命名規則)` | ✅ 参照は見出し名かアンカーリンクで |
| `typescript.md §4 を参照`            | ❌ 番号がずれると参照先が壊れる     |

**例外は、実行順序そのものが情報である手順書**（runbook・セットアップ手順・移行手順など）。「上から順に実行する」ことを読者に伝える必要があるなら番号を振ってよい。順序に意味がない列挙は、手順書の中でも番号を振らない。

### 専門用語は使ってよいが、文章はシンプルで短くする

AIは「賢そうに見える文章＝良い文章」と誤解し、硬い言い回し・名詞化・入れ子の文構造で無駄に飾りがち。だが価値を生むのは読者の理解だけ。目指すのは「**専門用語を別の記号に置き換えたとき、残りの文を中学生が一度で追える**短さと、シンプルな構造・論理」。

専門用語そのものは避けなくてよい。むしろ専門用語で一言にできるなら、その方が短くて良い。難しさの原因は用語ではなく、言い回しと文の組み立てにある。

| 種類               | 例                                                                            | 評価                          |
| ------------------ | ----------------------------------------------------------------------------- | ----------------------------- |
| 専門用語でシンプル | GSMフレームワークを使って設計した                                             | ✅ そのままで良い             |
| 専門用語でシンプル | flakyなテストの「オオカミ少年問題」を避けるため、タグで自動テストから隔離した | ✅ そのままで良い             |
| 硬い言い回し       | 変更耐性の最大化を企図したため                                                | ❌ → 変更に強くしたかったから |
| 名詞化・硬い動詞   | 〜が最優先事項であると判断した                                                | ❌ → 〜を何よりも大事にした   |

直すのは用語ではなく言い回しと構造：

- 漢語の硬い動詞を和語に：「企図する」→「ねらう」、「担保する」→「保つ」
- 名詞化をほどく：「変更への耐性を担保する」→「変更に強くする」
- 一文の主語・述語は1組まで：長い一文は切って、論理を一直線にする

> [!TIP]
> **（AI・推奨）** 自己判定テスト。**専門用語は数に入れない**。
>
> - **中学生テスト**：専門用語を別の記号に置き換えても、文の構造と論理を中学生が追えるか。追えないなら言い回しか構造が難しすぎる。
> - **音読テスト**：声に出して言わない硬い言い回し（「〜を企図する」等）は、書き言葉としても難しすぎる。

### 冒頭に「決定事項」を3行で先出しする（TL;DR）

捜索者は自分知りたい情報を探すために来る。設計書の冒頭に「何を・なぜそう決めたか」を**最大3行**でまとめる。目次は「何について書いてあるか（話題）」しか示さないが、TL;DRは「で、どう決めたのか（結論や決定事項）」を示す。両者は別物なので重複しない。

> [!IMPORTANT]
> **（AI・必須）** 絶対に3行まで。4行になったら、それはTL;DRではなく本文である。短くまとめられないなら、まだ設計判断が定まっていない兆候。

これは「最重要事項はあえて繰り返す」（DRY例外）に該当する、許可された重複。

ただし**決定を含む文書に限る**。事実を列挙するだけの文書（用語集・API一覧・テーブル定義）には決定がないため、TL;DRは不要で索引・目次で足りる。

| 文書の種類                                  | TL;DR                       |
| ------------------------------------------- | --------------------------- |
| アーキテクチャ設計、ADR（なぜその構成か）   | ✅ 決定事項を3行で先出し    |
| 用語集、API一覧、テーブル定義（事実の列挙） | ❌ 不要、索引・目次で足りる |

### トグル記法を積極的に活用する

長いドキュメントは読者の読むモチベーションを奪う。トグルで一部を隠すと、これを防げる。

<details>
<summary>トグル化すべき内容の例</summary>

- 補足説明
- 設計判断の詳細な理由
- 参考情報・背景知識
- 長いコード例

</details>

> [!IMPORTANT]
> 本ポリシーで最も重要な原則は次の3つ。
>
> - 対象読者と目的が全てを決定する
> - 「なぜ」を必ず書く
> - 図表で可視化する
