# Git Policy

commit・branch・PR・merge操作の前に参照する、コミットメッセージの書き方・粒度・ブランチ命名規則・rebase禁止の指針。

> [!IMPORTANT]
> **TL;DR**
>
> - コミットメッセージには「なぜ変えたか」を書く。「何を変えたか」はコードが語る。
> - issueに紐づく作業は件名末尾に `(#issue番号)` を付与する。
> - `git rebase` は使用しない。履歴の書き換えがバグ特定を困難にするためである。

## コミットの書き方

コミットメッセージには変更の**理由・背景**を書く。

「何を変えたか」はコードを読めばわかる。「なぜ変えたか」はコードからは読み取れない。その情報をメッセージに残す。

```
# Bad: 何をしたかしか書かれていない
パスワードバリデーションを追加

# Good: なぜ必要だったかが書かれている
パスワードバリデーションを追加

短いパスワードによる不正アクセスのリスクを下げるため、
8文字以上を必須とするバリデーションを実装した。
```

## issue対応時のコミットメッセージ

GitHub Issueに紐づく作業のコミットは、**件名の末尾に `(#issue番号)` を付与する**。

形式は `type: 説明 (#issue番号)` とする。

```
# Bad: どのissue対応か追跡できない
fix: ログイン失敗時のエラーメッセージを修正

# Good: 件名にissue番号があり、対応関係を追跡できる
fix: ログイン失敗時のエラーメッセージを修正 (#42)
```

issue番号を記録するのは、コミットとissue（背景・議論・受け入れ条件）を双方向に辿れるようにするためである。GitHubは `#42` を自動でissueへリンクするため、後からコミット履歴を見たときに「なぜこの変更が必要だったか」へ即座に到達できる。

なお、`Closes #42` のようなクローズ用キーワードは原則として件名・本文に書かない。本プロジェクトの開発フローではissueのタスク一覧（チェックリスト）を1つずつ進める運用であり、コミット単位でissueを自動クローズすると作業途中で閉じてしまうためである。

## 「なぜ」をどこに書くか（本文とissueの使い分け）

「なぜ変えたか」を残す場所は、**コミット本文** と **issue** の2つがある。どちらに書くかは作業の種類で決める。件名（`type: 説明`）はどの場合も必須である。

| 作業の種類                       | 「なぜ」の置き場所 | コミット本文               |
| -------------------------------- | ------------------ | -------------------------- |
| issueあり・背景がissueに十分     | issue              | 省略可（`(#番号)` で辿る） |
| issueあり・issueに無い判断をした | issue＋コミット    | その判断の理由を書く       |
| issueなし（アドホック対応）      | コミットのみ       | 必ず理由を書く             |

### issueに紐づく作業

issueに背景・受け入れ条件が記述されているなら、コミット本文は**省略してよい**。件名末尾の `(#issue番号)` から背景へ辿れるためである。

ただし、**issueに書かれていない判断をコミット時に行った場合は、その理由を本文に書く**。issueに無い情報は番号では辿れないためである。

```
# Good: issueに背景があり、本文は省略
fix: ログイン失敗時のエラーメッセージを修正 (#42)

# Good: issueに無い「実装上の判断」を本文に残す
fix: ログイン失敗時のエラーメッセージを修正 (#42)

ライブラリXのバリデーションは多言語対応が不十分なため、
issueの方針とは別に自前のメッセージ生成へ差し替えた。
```

この使い分けが成立する前提は、**issueが「なぜ」を十分に記述していること**である。issueの記述が薄い場合は、本方式は理由がどこにも残らない状態になるため、本文に書く。

### issueに紐づかないアドホックな対応

issueを起票しない単発の対応（軽微な修正・設定変更など）は、辿れる先がないため、**変更理由をコミット本文に書く**。冒頭「コミットの書き方」の原則どおり、コードから読み取れない背景を残すこと。

```
# Good: issue番号が無い分、本文に理由を残す
chore: タイムアウトを30秒から60秒へ変更

外部APIの応答が稀に30秒を超えタイムアウトしていたため、
暫定的に上限を引き上げた。
```

## コミットの粒度

粒度は基本的に任意とする。ただし、コミットメッセージと無関係なファイルを含めてはならない。

```
# Bad: ログイン修正とは無関係なCSSの変更が混入している
fix: ログイン失敗時のエラーメッセージを修正
- src/auth/login.ts
- src/styles/global.css   ← 関係ない

# Good: メッセージに関係するファイルだけが含まれている
fix: ログイン失敗時のエラーメッセージを修正
- src/auth/login.ts
```

## rebaseの禁止

`git rebase` は使用しない。マージには **3-way merge** または **fast-forward** を使う。

rebaseは履歴を書き換えるため、バグの混入箇所の特定（`git bisect`）が困難になる。また、共有ブランチでrebaseを行うと他者の履歴を破壊するリスクがある。

## ブランチ命名規則

ブランチ名は `<prefix>/<説明>` の形式とする。

| プレフィックス | 用途                         |
| -------------- | ---------------------------- |
| `feat/`        | 新機能・機能変更             |
| `fix/`         | バグ修正                     |
| `refactor/`    | 振る舞いを変えない構造の変更 |
| `chore/`       | 設定変更・依存関係の更新など |
| `ci/`          | CI/CD・GitHub Actionsの変更  |
| `test/`        | テストの追加・修正           |
| `docs/`        | ドキュメントの変更           |
