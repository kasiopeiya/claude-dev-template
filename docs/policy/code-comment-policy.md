# コードコメント作成ポリシー

コードを書く・レビューする開発者が、どの種類のコメントに何を書くべきか迷ったときに参照する指針。

> [!IMPORTANT]
> **TL;DR**
> - コメントは種類で書くべき内容が真逆になる。**実装コメントはWHY、ドキュメンテーションコメントはWHAT（契約）**を書く。読者が違うからである。
> - コードが語れること（HOW）は書かない。名前で消せるコメントは、コメントより**名前を直す**。
> - コメントは腐る。だから変わりやすいHOWでなく、変わりにくい**契約・なぜ**を書く。

## 基本原則

### コードコメントもドキュメンテーションの一種である

コメントもドキュメントであり、[documentation-policy.md](documentation-policy.md) の全原則が適用される。とりわけ「**対象読者と目的が全てを決定する**」「**HOWよりWHY**」「**シンプルで短く**」がそのまま効く。本ポリシーはその原則をコメントに特化させたものである。

### コメントは種類で書くべき内容が真逆になる

「コメントには何を書くべきか」に唯一の答えはない。**読者が違えば書くべき内容も反転する**からである。まず種類を見分けることが全ての出発点になる。

| コメントの種類 | 場所 | 読者（想定シーン） | 書くべき内容 |
|---|---|---|---|
| 実装コメント | 関数本体内・インライン | 実装を**読む**保守者 | **WHY**（なぜ） |
| ドキュメンテーションコメント（doc comment） | 関数・クラス宣言の上 | 実装を**読まない**呼び出し側 | **WHAT**（契約） |
| ファイル冒頭コメント | ファイル先頭 | ファイルを**開いた**開発者 | ファイルの責務＋前提 |

> [!TIP]
> **（AI・推奨）** 判断に迷ったら、まず「このコメントの読者はコードを読む人か、読まない人か」を問う。それで書くべき内容が決まる。

## 種類別の指針

### 実装コメントはWHYを書く（WHATは書かない）

読者は実装を読んでいる保守者である。WHAT（何をするか）はコードが語るので、再掲はノイズになり、しかも腐る。書く価値があるのは、コードを読んでも分からない**WHY**——なぜこの方式にしたか、なぜこの順序か、何の罠を避けているか——だけである。

| | 例 |
|---|---|
| ❌ WHAT（不要） | `// iをインクリメントする` |
| ✅ WHY（必要） | `// 外部APIが1始まりのため+1で補正している` |

### ドキュメンテーションコメント（doc comment）はWHAT（契約）を書く

宣言の上に書く `doc comment`（Javadoc・docstring・JSDoc等）を指す。読者は実装を読まずにその関数を使いたい呼び出し側である。中身を読ませてWHATを推測させるのは**カプセル化の放棄**にあたる。だから「コードを見れば分かる」WHATでも、ここでは書く。IDEのホバーや生成ドキュメントに現れるのもこのコメントで、**実装を開かずに使うための情報**である。

ただし、ここでのWHATは実装の言い換えではなく**契約（仕様）**である。一目では分からない以下を明示する。

- 受け付ける入力と暗黙の変換（例：数値も受け取り文字列化する）
- 事前条件・事後条件、戻り値、エッジケース（空入力時など）
- やらないこと、null許容性、単位

> [!NOTE]
> ドキュメンテーションコメントは「実装」ではなく「契約」を書くため、実装を変えても契約が同じなら更新不要。実装コメントのWHATより腐りにくい。

### ファイル冒頭コメントはファイルのTL;DRである

読者はそのファイルを変更・利用しに開いた開発者である。書くべきは「このファイルを触る前に、**コードを読んでも分からない知っておくべきこと**」——責務を一言、そして存在理由・不変条件・罠。行ごとの説明は書かない。

| | 書くか |
|---|---|
| このファイルの責務（一言で） | ✅ 書く |
| なぜ存在するか／なぜこの方式か | ✅ 書く |
| 守るべき不変条件・前提・罠 | ✅ 書く |
| 何をするか（コードが語るHOW） | ❌ 書かない |

> [!TIP]
> **（AI・推奨）** 責務が名前で自明で、前提も罠もないなら、冒頭コメントは不要である。

## 共通原則

### 名前で消せるコメントは、コメントより名前を直す

変数名・関数名が可読性を決める。WHATを補うコメントを書きたくなったら、まず**名前の改善**を検討する。名前は「だいたい何か」を、コメントは名前に載らない「契約・なぜ・罠」を担う。

### 詳細な記載よりシンプルで短い方が良い（最小限に絞る）

コメントは経験的に長くなりがちである。だが長いコメントは読まれず、腐る面積も広い。[documentation-policy.md](documentation-policy.md) の「詳細な記載よりシンプルで短い方が良い」がそのまま効く。書いた後、本当に必要最小限か必ず一度見直す。

> [!TIP]
> **（AI・推奨）** **削除テスト**：そのコメントを消したとき、読者は何を失うか。失うものがない、またはコードや名前で補えるなら消す。残すなら、削っても意味が変わらない語を削る。

| 絞り込みの問い | Yesなら |
|---|---|
| コードや名前を読めば分かるか？ | 消す |
| 同じことを別の場所でも書いていないか（DRY）？ | 一方に集約 |
| この一文がなくても読者は困らないか？ | 消す |
| 半分の長さで同じ意味を伝えられるか？ | 削る |

### コメントは腐る前提で、変わりにくいものを書く

コンパイラはコメントを検証しないため、コメントはコードより速く腐る。変わりやすいHOWを書くと嘘になりやすい。変わりにくい**WHY・契約・不変条件**を書けば腐りにくい。これは「変更が予想されるものはハードコードしない」のコメント版である。

> [!IMPORTANT]
> **（AI・必須）** 全てに共通する判断基準：**コード自身が語れることは書かない。語れないことだけ書く。** ただし「呼び出し側がコードを開かないための契約」は、コードが語れても書く。
