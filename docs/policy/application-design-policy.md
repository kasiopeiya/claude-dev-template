# アプリケーション設計ポリシー

「良い設計とは何か」を、特定の技術・パターンに依存しない普遍的な指針として宣言する。アプリケーションの**コード内部の設計（クラス・関数・モジュールの作り方）**を対象とし、マクロ構造（レイヤー・境界・依存方向）は [application-architecture-policy](application-architecture-policy.md) に委ねる。

## 対象読者

クラス・関数・モジュールを設計／実装／レビューする人 / AIエージェントが、設計判断の良し悪しを確かめたいとき。

## 正しく使うのが一番易しいインターフェースにせよ

**良いインターフェースとは、正しい使い方が「一番易しい使い方」で、誤った使い方が困難なものである。** 設計が良ければ利用者（呼び出し側）はほぼ間違いなく正しく使える——正解が一番自然で見つけやすい道になっているからだ。これは本文書の多くの節が仕える上位の目標であり、§引数は責務を映す鏡である・§関数とクラスの使い分け・§失敗は投げて伝えよ は、いずれもこの原則の具体的な適用例である。対象は関数・クラス・モジュールのインターフェース（原則自体はUIにも通じる普遍則だが、UIは本ポリシーの対象外）。

### 正しい使用を「最も簡単な道」にする

利用者が最短・最自然に選ぶ道が、そのまま正解になっているようにする。一番「自然な値」が正しい値であるよう既定やシグネチャを設計し、名前だけで正解が一目で分かるようにする（[命名規約](../../.claude/rules/typescript.md)）。正しく使うために仕様書を読み込まねばならない時点で、インターフェースが負けている。

### 誤用を構造的に困難にする

利用者がしそうな間違いを**先回りして予期し、そもそも起こせない形**にする。「規約で禁止する」より「型・構造で不可能にする」を上に置く（暗黙より明示。[refined-engineer-judgment-principles 原則1](refined-engineer-judgment-principles.md)）。

| 誤りの種 | 構造で封じる手 |
|---|---|
| 引数の順序ミス | 順序に依存しない設計にする（どの順でも同じ結果／引数オブジェクトで名前付きにする） |
| 無意味な状態・操作の実行 | その文脈で無意味な操作はそもそも表現・実行できないようにする（不正な状態に「できない」＝カプセル化） |
| 同型の引数の取り違え | 取り違えやすい同型値（複数の `string` 等）は型で区別する |

> [!WARNING]
> **（AI・必須）** 「ドキュメントに書いたから大丈夫」は誤用対策ではない。利用者は注意書きを読まずに最短の道を通る。誤用を**型・シグネチャ・構造**で起こせなくして初めて対策になる。

### 確認法：呼び出し側のコードを先に書く

インターフェースの良し悪しは、定義側を眺めても分からない。**使う側のコードを先に書く**と一発で見える——呼び出しが自然に読めるか、引数の順序や値に迷わないか、誤用が型で弾かれるか。実装を書き切ってから使いにくさに気づくと作り直しのコストが高いので、設計段階で呼び出し例を先に書き、使い勝手を実装前に検証する（[refined-engineer-judgment-principles 原則3 シフトレフト](refined-engineer-judgment-principles.md)）。テストファースト（呼び出し例＝テスト）は、この確認法の自然な実践形でもある。

> [!TIP]
> **（AI・推奨）** 判定テスト：その関数・クラスの「最初の呼び出し例」を数行書いてみて、コメントなしで意図が読めるか。読めなければ呼び出し側で辻褄を合わせず、**インターフェース側を直す**。

## 関数とクラスの使い分け

判断軸は1本：**「状態と不変条件を、外部から守る必要があるか」**。**既定は関数 + モジュール**で、クラスは下表のいずれかに当たるときだけ選ぶ（クラスは `this` という暗黙で守るべき表面積を増やすため。[refined-engineer-judgment-principles 原則1・4・5](refined-engineer-judgment-principles.md)）。

| クラスを選ぶ理由 | 内容 | 備考 |
|---|---|---|
| 状態と不変条件のカプセル化 | 外部から不正な状態に「できない」保証が要る。private + メソッドで不変条件を内部に閉じる | 不変条件を持つエンティティ・値オブジェクトが典型 |
| ポリモーフィズム / 差し替え（DIP） | 同一インターフェースの実装を注入・差し替える | TS は関数注入でも代替可。**メソッドが複数揃って初めて**クラスが優位 |
| フレームワーク要求 | 選択の余地がない | CDK Construct、`Error` 継承など |

これに当たらないもの（状態を持たない変換・計算・IO・手続き）はすべて関数にする。ドメインの中核はクラス、それを呼ぶユースケース・変換・IO は関数、クラスのメソッド内部のロジックは純粋関数に切り出す——というように**併用**する。

> [!TIP]
> **（AI・推奨）** **迷ったときのテスト**：「この内部状態は、外から守る必要があるか？」——No なら関数。

## コードを共通化すべきか

**共通化とは、結合を許容することである。** 重複を見つけたら即座にまとめる——これは初心者の本能だが、罠でもある。共通化は目に見える重複を消す代わりに、目に見えない結合を生む。そして消した重複より、生んだ結合のほうが高くつくことは珍しくない。だから「重複していて気持ち悪い」だけでは理由にならない。**① 本当に同じ概念か → ② 結合コストに見合うか** の両方が Yes のときだけ共通化する。

### ① それは「同じ概念」か

形が同じでも概念が違えば共通化しない。DRY が禁じるのはコードの重複ではなく**知識（意図）の重複**だから。

| 問い | 共通化する側 |
|---|---|
| 将来「同じ理由で・同時に」変わるか？（＝SRP。DDDではコンテキストの一致） | Yes |
| 形ではなく "知識・意図" が同じか？（偶然の一致でないか） | Yes |

> [!TIP]
> **（AI・推奨）** 判定テスト：片方の仕様変更で、もう片方も必ず変えたくなるか？ No なら別概念——たまたま今だけ同形なので、共通化せず放置する。

### ② 結合コストに見合うか

① が Yes でも、共通化が生む結合が重複の痛みを上回るなら共通化しない。**間違った抽象は重複より高くつく**（重複は局所的だが、間違った抽象は全利用者を巻き込む）。

| 問い | 共通化する側 |
|---|---|
| 重複を放置するコスト > 結合を生むコストか？ | Yes |
| 同一コンテキスト内に収まるか？（境界・デプロイ単位・チームをまたがないか） | Yes |
| 抽象は安定したか？（仕様が固まり、3回以上現れたか＝Rule of Three） | Yes |

> [!WARNING]
> **（AI・必須）** 早すぎる共通化は負債になる。仕様が流動的なうちは重複を許容し、抽象が固まるのを待つ。

DRY と SRP が競合したときの既定の重み付け（SRP 優先）は [refined-engineer-judgment-principles 原則1](refined-engineer-judgment-principles.md) に従う。本節はそれを「共通化」の文脈に具体化したものである。

## レイアウトで重要箇所を語らせよ

**最重要のロジックは、読まなくても「ここが重要だ」と分かるレイアウトにする。** 非エンジニアがファイルを開いても心臓部を指させるのが理想。手段は2段構え。

- **① 新聞構造**：上位の流れほど上、詳細ほど下に置く。`main` や公開関数を上から読めば重要な処理が見出しのように並ぶ。重要さは第一に「名前」と「位置」で語らせる。
- **② バナー**：新聞構造で並べた中の**心臓部1箇所だけ**に、読まずに目へ飛び込むバナーを打つ。具体記法は [typescript.md コメント節](../../.claude/rules/typescript.md)。

| バナー様式の原則 | 理由 |
|---|---|
| 1ファイルに最大1個 | 希少だから信号になる。全部に付ければ何も目立たない |
| 右枠は付けない／区切り線は固定長でコピペ | 全角を含む右端合わせは保守地獄。左の線だけで十分目立つ |
| 意味タグ（grep 用）ではなく視覚装飾と割り切る | 読ませる情報は命名・新聞構造側に置く。バナーは"指差し"専用 |

## 上位関数は目次であれ

**主要な処理の流れを書いた関数は、それ単体を読むだけで全体像が想像できる「目次」にする。** システムを知らない読者でも `readCsv → validate → saveRecords → deleteCsv` を見れば大筋を当てられる。条件は2つ。

| 条件 | なぜ |
|---|---|
| ひとつの関数の中は**抽象度を一段に揃える**（SLAP） | 生のIO・ループと高レベル呼び出しが混ざると、流れが詳細に埋もれて読めない |
| 各ステップは**ドメインの動詞で命名**する（[命名規約](../../.claude/rules/typescript.md)） | 名前が語れば中身を読まずに流れが分かる。詳細は下位関数へ押し下げる |

```typescript
// ❌ 抽象度が混在：流れが詳細に埋もれる
function main() {
  const text = fs.readFileSync('in.csv', 'utf-8')        // 低レベル（IO）
  const rows = text.split('\n').map((l) => l.split(',')) // 低レベル（パース）
  validate(rows)                                          // 高レベル
  for (const r of rows) db.insert(r)                      // 低レベル
  fs.unlinkSync('in.csv')
}

// ✅ 抽象度を揃える：main を読めば流れが想像できる
function main() {
  const records = readCsv('in.csv')
  validate(records)
  saveRecords(records)
  deleteCsv('in.csv')
}
```

> [!TIP]
> **（AI・推奨）** 本節は**書く高さ（抽象度）**、§レイアウトで重要箇所を語らせよ の新聞構造は**並べる順（位置）**の話。別の軸だが、両方そろうと上位関数が真の「目次」になる。

## 引数は責務を映す鏡である

**引数の数と形は、その関数の責務をそのまま映す。** 引数が増えるのは関数が多くを抱えすぎたサインであり、引数で振る舞いを切り替えるのは1つの関数に複数の責務が同居しているサインである。だから引数は小手先で「減らす」のではなく、**まず責務を疑う**——本節は [refined-engineer-judgment-principles 原則1（SRP）](refined-engineer-judgment-principles.md) を引数設計に具体化したものである。

### 数は4つまでを基本に（ただし絶対ではない）

引数は最大4つを目安とする。これは機械的な上限ではなく、**それを超えたら「この関数は責務を持ちすぎていないか」と立ち止まって問うための閾値**である。引数が多いほど呼び出し側での順序ミスや可読性低下も招く。超えそうなときの対処は2段階：

| 手順 | 内容 |
|---|---|
| ① まず責務を疑う | 引数の多さは凝集度低下のシグナル。関数を分割できないか先に検討する |
| ② 引数オブジェクト化（Parameter Object） | 分割しきれないなら、**関連する引数をまとめて1つのオブジェクト**にする |

> [!WARNING]
> **（AI・必須）** ②で**無関係な引数を袋詰めして数を誤魔化さない**。オブジェクトにまとめてよいのは「同じ概念としてまとまっている＝凝集している」ときだけ（判断軸は §コードを共通化すべきか の "同じ概念か" と同じ）。数合わせの袋詰めは、低い凝集度を隠してむしろ悪化させる。

### ブーリアン引数アンチパターンを禁ずる

**呼び出し側が渡す値で関数の処理が分岐するなら、その関数は責務が割れている。** その代表が**ブーリアン引数アンチパターン**——真偽値で振る舞いを切り替える引数である。

```typescript
// ❌ ブーリアン引数アンチパターン：呼び出し側で true が何か読めず、関数は2つの責務を抱える
saveUser(user, true)   // ← この true は何を意味する？
function saveUser(user: User, isNew: boolean) {
  if (isNew) { /* INSERT */ } else { /* UPDATE */ }
}

// ✅ 責務ごとに関数を分ける：名前そのものが振る舞いを語る
createUser(user)
updateUser(user)
```

| 原則 | なぜ |
|---|---|
| boolean に限らず、振る舞いを切り替える**フラグ引数全般を避ける**（`mode: 'a' \| 'b'` 等の enum 分岐も同類） | boolean だけ禁じても enum で抜け道が残る。根本は「引数で分岐させない＝単一責務」 |
| 呼び出し側に**マジック値を渡さない**（`null` / `3` / `true`） | 呼び出し箇所だけ読んで意図が分かること。意味が要るなら named な形（オブジェクト）にする |

> [!TIP]
> **（AI・推奨）** 判定テスト：その引数を消すために関数を2つに分けたら、それぞれの名前で振る舞いを言い切れるか？ Yes なら分割が正解。

## 多重ネストは設計を見直すシグナルである

**深いネスト（多重の if / try / for / while）は、その関数が責務を抱えすぎ／制御フローが複雑すぎるという code smell である。** §引数は責務を映す鏡である と同じく、表面的な指標が責務過多を映している。だからネストは見た目だけ「数を減らす」のではなく、**まず責務を疑う**——責務の抽出・早期 return・分岐の単純化で複雑さそのものを下げる。複雑さを温存したまま `.catch` チェーン化などで見た目だけ平坦化する小手先の回避は、smell を隠すだけなので禁止する。

> [!TIP]
> **（AI・推奨）** ネスト数の上限・許される対処・禁止する回避の具体は [typescript.md「ネスト制限は『設計を見直せ』というシグナル」](../../.claude/rules/typescript.md) を参照。

## 失敗は投げて伝えよ

**エラーは例外スロー（`throw`）で伝える。失敗を戻り値の型に畳み込む `Result` 型・`Either` 型は使わない。** `throw` は標準機構だけで完結し、合成のたびに失敗を取り出す（unwrap する）儀式が要らない——呼び出しコードは本筋だけが並ぶ。`Result` の長所「失敗が型に出る」を手放す代償は、後述「型に出ない失敗をどう補うか」で埋める。

```typescript
// ❌ 失敗を戻り値型に畳み込む（Result 型）：呼び出しのたびに unwrap が要る
function validateEmail(email: string): Result<string> { ... }

// ✅ 失敗は throw、戻り値は成功値だけ
function validateEmail(email: string): string {
  if (!EMAIL_REGEX.test(email)) throw new ValidationError('Invalid email')
  return email
}
```

### 既定は throw、戻り値でエラーを表すのは「狭い例外」だけ

判断軸は「異常か想定内か」ではない。その軸だと想定内の失敗がすべて戻り値表現に倒れ、本末転倒になる。軸は **「呼び出し側にとって `throw` が役に立つか」**。既定は `throw` とし、戻り値でエラーを表すのは下の場面に限る。

| 戻り値表現が勝つ場面 | 例 |
|---|---|
| 失敗を**複数まとめて**返したい（`throw` は最初の1件で止まる） | フォーム全項目のバリデーション |
| 失敗が**日常的な結果の一種**で、呼び出し側が毎回必ず分岐する | `findUser`（不在）・`tryParse` |
| ホットループで**頻発**し、例外を制御フローに使うことになる | 大量データの逐次判定 |

これ以外（単発のバリデーション失敗・ビジネスルール違反・外部IO失敗など）はすべて `throw`。クエリメソッドもエラー時は `throw` し、戻り値型にエラー表現を含めない。

> [!WARNING]
> **（AI・必須）** 戻り値でエラーを表す数少ない場面でも、**`Result`／`Either` 型は使わない**。ネイティブの `T | null` または判別可能ユニオンで表す。「とりあえず Result で包む」癖が、本来 `throw` でよい大半の関数まで巻き込むのを防ぐため。

### 型に出ない失敗をどう補うか

`throw` の弱点は、TS に検査例外がなく**何を投げるかが型シグネチャに現れない**こと。`Result` を捨てた分、次の3点で補う。

| 補い方 | 内容 |
|---|---|
| `@throws` を明記 | 関数が投げるエラーを JSDoc に書く（[typescript.md](../../.claude/rules/typescript.md)） |
| エラー型を分類する | カスタムエラーの基底クラスを継承し、`throw new Error('msg')` の直書きを避ける。catch（TS では `unknown`）は `instanceof` で絞る。分類の柱は §例外は2種に分ける |
| 境界で集約 catch | ドメイン層は投げっぱなしにし、エントリポイント（ハンドラ等）1箇所で catch・ログ・レスポンス化する。握りつぶしを禁ずる |

> [!TIP]
> **（AI・推奨）** 判定テスト：その失敗を、呼び出し側は**毎回必ず分岐して処理するか**？ No（基本は通すだけ・たまに上位で拾う）なら `throw`。Yes（結果の一種として常に扱う）なら戻り値表現（ただし `Result` 型ではなくネイティブ union）。

### 例外は2種に分ける——技術的例外とビジネス例外

`throw` する失敗は、さらに**技術的例外**（technical / system exception）と**ビジネス例外**（business / domain exception）に分類する。これは失敗の**表し方**（`throw` か戻り値か＝前項）とは独立した、失敗の**性質**の軸である。分類の柱は **「その失敗を誰が解消するか」**——文脈に依らず常に成立する問いに置く。

| 種類 | 何か | 誰が解消するか | 例 |
|---|---|---|---|
| 技術的例外 | これが起きると**アプリの実行そのものを継続できない**。システムが壊れている | 開発者・運用者（コード修正・インフラ復旧） | DB接続断・ネットワーク断・メモリ枯渇・到達不能パス・プログラミングバグ |
| ビジネス例外 | ユーザーにアプリの誤用をさせないため、設計上**わざと**発生させる。システムは正常で、要求が不正 | ユーザー（入力・操作の修正） | バリデーション違反・ビジネスルール違反・権限不足・不正な状態遷移 |

**なぜ分けるか**：同じ「例外」でも、発生後にすべき扱いが正反対だから。ビジネス例外（＝日常茶飯事のユーザー操作）を技術的例外と一緒くたに扱うと、本当に壊れた技術的例外がユーザー操作のノイズに埋もれ、対応すべき障害を取り逃がす。

**実装**：基底クラス `TechnicalError` / `BusinessError` を常に用意し、すべての例外をどちらかに属させる。境界の集約 catch（前項）で `instanceof` により振り分ける。**どちらにも分類されない未知のエラーは技術的例外として扱う**——分類漏れ＝想定外＝壊れているはずだから、安全側に倒す。

個別のサブクラス（`ValidationError` 等）は、(a) 呼び出し側が個別に捕捉して分岐したい、(b) 固有の構造化データ（どのフィールドが不正か等）を持たせたい、のいずれかの理由があるときにだけ作る。理由がなければ `throw new BusinessError('...')` でよい——**理由なく例外ごとにクラスを増やさない**（早すぎる抽象は負債。§コードを共通化すべきか）。基底2クラスは素の `Error` ではないので、直接投げれば「`throw new Error` 直書きを避ける」は満たす（＝サブクラス必須という意味ではない）。

```typescript
class TechnicalError extends Error {}   // 継続不能：開発者・運用者が直す
class BusinessError extends Error {}    // 設計上の失敗：ユーザーが直す

class ValidationError extends BusinessError {}  // 個別捕捉・固有データが要るので作る

// 境界（ハンドラ）での集約 catch
catch (e) {
  if (e instanceof BusinessError) {
    return toClientError(e)   // 何を直せば解決するか、理由を具体的に返す（握りつぶさない）
  }
  // 未知も含め技術的例外：内部詳細（スタックトレース・SQL等）はユーザーに出さない
  return toInternalError(e)
}
```

> [!TIP]
> **（AI・推奨）** 判定テスト：**その失敗は、ユーザーが自分の入力・操作を直せば解消するか？** Yes → ビジネス例外。No（直すのは開発者・運用者）→ 技術的例外。判断がつかない／想定外は技術的例外に倒す。

> [!NOTE]
> HTTP API ならビジネス例外→4xx、技術的例外→5xx が素直な対応だが、これは**境界での写像**であって例外そのものの性質ではない（バッチ・キュー消費など非HTTP文脈に status code は無い）。だから分類の柱は「誰が解消するか」に置き、4xx/5xx は定義に含めない。
