# 設計書作成ポリシー

設計書を書く・レビューする開発者／AIが、**良い設計書と悪い設計書を見分ける基準を知りたいとき**に参照するドキュメント。読み終えたとき「良い設計書とはこういうもの・悪い設計書とはこういうもの」を自分の言葉で言えるようになることを、この文書の目的とする。

> [!IMPORTANT]
> **TL;DR（このポリシーの決定事項）**
>
> - **良い設計書**＝コードから読み取れないこと（全体の構造・流れ・なぜ）だけを直感的に示した文書。**悪い設計書**＝コードを読めば分かること（実装詳細・手順・仕様）を書き写した文書（保守段階で二重管理になり腐る）
> - 大局的な構成・関係はコードから読み取れず、図表で表すのが最も直感的。ゆえに**設計書の主役は図表**——削るなら文章から削り、図表を最後まで残す
> - 「なぜ」（採用理由・**却下案**・変更可能境界・スコープ外）に必ず答える。なぜは図の注記としても書ける

## 基盤との関係：documentation-policy を継承し、差分だけを足す

設計書も文書なので、**まず [documentation-policy](documentation-policy.md) の全原則に従う**（対象読者と目的・Progressive Disclosure・DRY・可視化・短さ・TL;DR・「なぜ」を書く）。本ポリシーはそれを再掲せず、**設計書という種別に固有の指針だけ**を上乗せする。runbook-policy が documentation-policy に対して持つ関係と同じ、独立・単体完結の専用ポリシーである。

## 前提：本ポリシーが定義するのは「保守され続ける設計書」である

「設計書」と聞くと、設定値まで盛り込んだ文書を先に書き、それを書き写すように実装する——というウォーターフォール的なイメージがあるかもしれない。だが**具体値だらけの設計書は、保守段階でコードと同じ情報を二重に抱える「二重管理」になり、必ず腐る。**

ただし一次資料として作ること自体は否定しない（初期設計の検討・合意のスナップショットなど）。本ポリシーが定義するのは、その先の**コードと共に生き続け、保守され続ける設計書**である。

| 設計書の種類                               | 具体値だらけでよいか | 本ポリシーの対象    |
| ------------------------------------------ | -------------------- | ------------------- |
| 一次資料（保守しない、作った時点の記録）   | ✅ よい              | — 対象外            |
| 保守され続ける設計書（コードと共に生きる） | ❌ 二重管理で腐る    | ✅ 本ポリシーが定義 |

## 定義：良い設計書とは「コードから読み取れないこと」を直感的に示した文書である

良い設計書か悪い設計書かは、次の理屈で決まる。

1. **良い設計書とは、コードからは読み取れないことが書かれた文書である**（容易に読み取れないことを含む）。コードが SSoT（Single Source of Truth）だから、コードを読めば分かることの書き写しは、前述の二重管理＝悪い設計書にすぎない
2. 大局的な**構成や関係性**は、コードからは容易に読み取れない（何ファイルも追って初めて頭の中で再構築できる）
3. 構成や関係は、**図表で示すのが最も直感的でわかりやすい**（documentation-policy「可視化されていないドキュメントは怠慢」）
4. ゆえに、**良い設計書は構成・関係・流れを図表で直感的に示す**

> [!IMPORTANT]
> **（AI・必須）** これは「図表**だけ**書け」という意味ではない。**設計書を削っていったとき、最後に残すべきは図表である** という意味である。人は文章より図表を見る。WHY（なぜ）も図の注記・吹き出しに書き込めるので、図表を最優先にしても「なぜ」は失われない。**削るときは文章から先に削り、図表を本体として残す。**

良い設計書と悪い設計書は、次の軸で見分けられる。

| 観点 | ✅ 良い設計書                                                | ❌ 悪い設計書                                                    |
| ---- | ------------------------------------------------------------ | ---------------------------------------------------------------- |
| 中身 | コードに書いていない／容易に読めないこと（構造・流れ・なぜ） | コードを読めば分かること（実装詳細・手順・仕様）を書き写している |
| 主役 | 図表が主役、文章は補足                                       | 文章の壁。図がない、または文章を箱に詰め替えただけの図           |
| なぜ | 採用理由・却下案まで書いてある                               | 結論だけ。なぜ・却下案がない                                     |
| 寿命 | コードと二重管理にならず腐りにくい                           | コードと重複し、変更のたびに片方が陳腐化する                     |

## 悪い設計書の典型：コードに書いてあることを混入させる

悪い設計書の最大の症状は **混入**——コードや他文書に居場所がある情報を、設計書に書き写してしまうことである。次の3つは設計書に**混入させない**。それぞれ別の居場所がある。

| 混ぜないもの | 性質                                                 | 正しい居場所                                 |
| ------------ | ---------------------------------------------------- | -------------------------------------------- |
| 手順         | 行動と再現（セットアップ・デプロイ・運用・障害対応） | 手順書 → [runbook-policy](runbook-policy.md) |
| 仕様         | 振る舞いの契約・入出力の網羅（WHATの詳細）           | 業務What→requirements.md／契約→型＋テスト／コード |
| 実装詳細     | 具体ロジック・命名・行レベルの中身                   | コード（SSoT）                               |

> [!IMPORTANT]
> **（AI・必須）** 設計書を書く・レビューするとき、各記述に **「これはコードを読めば分かるか？」** を問う。分かるなら設計書に書かない（書けば二重管理で腐る＝悪い設計書化する）。手順は手順書へ、仕様の網羅は業務Whatなら requirements.md・技術契約なら型＋テストへ、実装詳細はコードへ逃がす。

## 迷ったら：What（振る舞い）か設計か（項目を分解して振り分ける）

「これは What（振る舞い）か設計か」で迷う項目は多い。迷う原因は、**項目名が「振る舞い（What）」「構造・なぜ（How/Why）」「具体値」の3層を混ぜて含んでいる**こと。項目を丸ごと振り分けようとするから割れない。**まず層に分解すれば、各層の行き先は一意に決まる。**（ここで決めるのは内容の行き先）

What はさらに2つの居場所に分かれる——**業務用語で書けるビジネスの振る舞いは requirements.md、コード読者向けの入出力・エラー契約は型＋テスト**。専用の仕様書 markdown は作らない（型・テストが実行可能な SSoT であり、markdown に書き写すと二重管理で腐る）。

| 行き先                    | 何を書くか                             | ひとことテスト                           |
| ------------------------- | -------------------------------------- | ---------------------------------------- |
| requirements.md（業務What） | 正しい振る舞い・業務ルール（業務用語） | 外から観測でき、業務で検証できるか？     |
| 型＋テスト（技術契約）    | 入出力・エラーの契約（コード読者向け） | 型・テストで表現・検証できるか？         |
| 設計書                    | 構造・流れ・なぜ（How/Why）            | コードを読んでも分からない作り・理由か？ |
| コード/設定               | 具体値・実装そのもの（SSoT）           | 値が変わるたびに直る具体物か？           |

同じ項目名でも、層ごとに行き先が分かれる：

| 項目             | What：振る舞い・契約（業務→requirements.md／契約→型＋テスト） | 設計書（How/Why・構造）                                          | コード/設定（具体値）            |
| ---------------- | ----------------------------------------------------------- | ---------------------------------------------------------------- | -------------------------------- |
| 環境変数         | （振る舞いを定義する変数なら）その契約                      | 何を設定軸にし、なぜ外出しするか                                 | 変数名と値（`.env`・パラメータ） |
| ビジネスロジック | 業務ルール＝正しい結果（例：ポイント1%・有効期限1年）       | 実現の流れ（シーケンス図）・なぜその方式か                       | 計算・判定の実装                 |
| API              | 入力／出力／エラーの契約                                    | 認証・認可の流れ、コンポーネント連携                             | ハンドラ実装                     |
| データモデル     | 各項目の制約・整合性ルール                                  | エンティティ関係（ER図）・なぜこのモデルか                       | スキーマ定義（DDL）              |
| 画面／UI         | 表示・入力・バリデーション・画面遷移のルール（操作→遷移先） | 画面の状態管理・コンポーネント構造・なぜこの画面分割／遷移設計か | コンポーネント・ルーティング実装 |
| 非機能           | 可用性・性能の目標値（要件）                                | 目標をどう達成するかの方式・トレードオフ                         | タイムアウト等の設定値           |
| エラー定義       | 呼び出し側が受け取るエラー契約                              | 失敗モードの俯瞰・なぜこの分類か                                 | エラーコード値・実装             |

**図かどうかは行き先を決めない。** 画面遷移図・業務フロー図は「仕様（外から観測できる振る舞い）を図にしたもの」で、図でも仕様である。「設計書の主役は図表」は*設計書の中での話*にすぎない。振り分けは形式（図か文か）ではなく **「外から観測・検証できる振る舞いか／内部の構造・なぜか」** で行う。

> [!IMPORTANT]
> **（AI・必須）** 迷ったら項目を丸ごとどちらかに入れようとせず、まず3層に割る。設計書に残すのは「構造・流れ・なぜ」だけ——振る舞い（What）は業務なら requirements.md・契約なら型／テストへ、具体値はコードへ逃がす（混入させない）。

## 良い設計書が書くこと（1）：構造・流れ・全体像を図表で

コードは正確だが粒度が細かすぎて、**全体像が見えない**。「どのサービスがどう繋がるか」「処理がどう流れるか」は、コードを何ファイルも追って初めて頭の中で再構築できる——その再構築をあらかじめ済ませて図表で渡すのが、良い設計書である。

| 書くべきもの            | いつ書くか（コードからは見えない理由）                                                                                                                         |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| アーキテクチャ構成図    | 全体構成はコードに散らばり一望できない。AWSサービス構成（物理）／論理アーキテクチャ（例：Clean Architecture の同心円）                                         |
| シーケンス図            | 複数コンポーネントをまたぐ処理の流れ。呼び出し順序・往復はコードを追わないと見えない                                                                           |
| 状態遷移図              | 状態と遷移条件が複数箇所に散らばり、取りうる状態の全体像がコードから読めない                                                                                   |
| データモデル図（ER 等） | エンティティ間の関係・整合性制約は、テーブル定義やクラスを個別に見ても「関係」が見えない                                                                       |
| 依存・境界図            | レイヤ／モジュール間の依存方向・責務境界（どこがどこに依存してよいか）はコードに明示されない                                                                   |
| 要注意箇所の解説＋図解  | **ややこしい**（追いにくい）／**重要**（セキュリティ・バグを生みやすい）／**特殊**（一般的でないアプローチ）な設計は、コードだけでは意図と落とし穴が伝わらない |

> [!TIP]
> **（AI・推奨）** 図は `/design-doc-mermaid` スキルで作成する（CLAUDE.md の方針。自前で Mermaid 記法を書き起こさない）。

## 良い設計書が書くこと（2）：なぜ（4つの問い）

（1）が _構造・流れ_ なら、こちらは _判断・理由_。どちらもコードに現れないが、性質が違うので分けて渡す（なぜは図の注記としても渡せる）。設計書の章立て（目次）は縛らない——構成は対象読者と目的から導くものだからである（documentation-policy「対象読者と目的が全てを決定する」）。**縛るのは目次ではなく、答えているべき問い**。次の4つに答えていない文書は、設計の*記録*ではあっても設計*書*ではない。

| 問い                                                     | なぜ要るか                           | ❌ 欠けると起きること（悪い設計書）                  |
| -------------------------------------------------------- | ------------------------------------ | ---------------------------------------------------- |
| なぜこの構造・方式を選んだか（**採用理由**）             | 設計書の存在理由そのもの             | 変えてよいか判断できず、読者が塩漬けにする           |
| どの代替案を、なぜ却下したか（**却下案**）               | 検討の深さと「選ばなかった道」を残す | 同じ検討が蒸し返される／一度否定した道へ再び迷い込む |
| どこまで変えてよく、何が不変の前提か（**変更可能境界**） | 変更可能性を渡す核                   | 触ってはいけない前提を壊す／逆に怖くて何も触れない   |
| 何を意図的に対象外としたか（**スコープ外**）             | 「書いていない＝未検討」の誤読を防ぐ | 未対応を見落としと取り違える／際限なく要求が膨らむ   |

> [!IMPORTANT]
> **（AI・必須）** とりわけ **却下案を省かない**。documentation-policy は「なぜを書く」を求めるが、設計書の「なぜ」は*採用した理由*だけでは半分。**却下した案とその理由**を書いて初めて、読者は設計者と同じ判断を再現できる。採用案だけの設計書は、結論だけ書いた数学の答案に等しい。

## 経緯（なぜ変えたか）は設計書でなく ADR に逃がす

設計書が言い切るのは **現在の設計の「なぜ」**（＝なぜ今この構造か）。これに対し、**設計を変えた経緯の「なぜ」**（＝なぜ旧設計から変えたか）は ADR（`docs/adr/`）の領分である。両者を混ぜると、設計書に改定履歴が溜まって「現在の仕様だけ読みたい読者」が困る（documentation-policy「"工事中"のドキュメントは永遠に工事中」と同根）。

| 「なぜ」の種類                             | 置き場所                  |
| ------------------------------------------ | ------------------------- |
| なぜ今この構造・方式か（現在の設計の根拠） | 設計書（本書の4つの問い） |
| なぜ旧設計から変えたか（意思決定の経緯）   | ADR（`docs/adr/`）        |

---

> [!IMPORTANT]
> 本ポリシーで最も重要な原則は次の3つ。
>
> - **良い設計書はコードから読み取れないことだけを書く**：コードを読めば分かること（実装詳細・手順・仕様）の混入が悪い設計書（保守段階で二重管理になり腐る）。ただし保守しない一次資料は対象外
> - **図表が主役**：構造・流れ・全体像は図で渡す。削るなら文章から削り、図表を最後まで残す
> - **「なぜ」を4つの問いで渡す**：却下案を省かない
