# 非機能要件の項目カタログ

要件定義書（`docs/requirements.md`）を書く・レビューする人が、**非機能要件として分野別に何を書くべきか**を知りたいときに参照するカタログ。IPA「非機能要求グレード」の6大項目をベースに、要件に書く **What** だけを抽出した（設計に踏み込む How は各表の「除外」に回す）。

## 使い方

- 要件段階で全項目を具体化しきる必要はない。IPA のグレード（重要度・目標水準のレベル）を**議論のたたき台**にし、顧客と合わせながら**段階的に**埋める。
- 埋まらない項目は無理に数値を置かず（[requirements-doc-policy 原則2](../policy/requirements-doc-policy.md)）、**オープンクエスチョンへ逃がす**（黙って落とさない）。
- 継続的に監視する目標値（稼働率・応答時間 等）は **SLA/SLO/SLI で定量化**し、分野別テーブルは「何を保証するか」の項目だけを持つ（同じ事実を二重管理しない）。
- 全項目に共通の線引きは **How に降りないこと**——業務・サービスの水準で書き、製品名・構成・実現手段は書かない（[原則1](../policy/requirements-doc-policy.md)）。

## A. 可用性

| 決める項目 | 要件定義に書く内容（What・例） |
|---|---|
| サービス提供時間 | いつ使えるべきか（例：平日8–20時／24時間365日）。計画停止を許すか |
| 稼働率の目標 | 目標値は **SLO（可用性）で定量化**。ここでは対象範囲と除外条件（計画停止を含む/含まない）だけ決める |
| 目標復旧時間（RTO） | 障害発生から復旧までの許容時間（例：2時間以内） |
| 目標復旧時点（RPO） | 許容できるデータ損失の幅（例：直近5分まで／前日まで） |
| 縮退運転の許容 | 障害時に機能を限定して継続してよいか、全面停止か |
| 大規模災害時の目標 | 被災時に復旧を目指すか（要否）、目指すなら目標時間。マルチリージョン等の要否判断の前提 |

除外（How・設計へ）：冗長化構成・クラスタ方式、RAID/レプリケーション、DR 構成、バックアップ取得方式。

## B. 性能・拡張性

| 決める項目 | 要件定義に書く内容（What・例） |
|---|---|
| 通常時の業務量 | 想定ユーザ数・同時接続数・リクエスト/データ件数（数値で。原則3） |
| ピーク時の業務量 | ピークの倍率と発生タイミング（例：月末に平常の5倍） |
| 将来の増加見込み | いつ・どれだけ増えるか（容量計画の前提。例：3年で2倍） |
| 応答時間の目標 | 目標値は **SLO（レイテンシ）で定量化**。ここでは対象トランザクションと測定条件 |
| 処理量・締切の目標 | バッチ等の完了締切・スループット（例：日次バッチを6時までに完了） |

除外（How・設計へ）：CPU/メモリ/ディスクの拡張方式、オートスケール設定、負荷分散・帯域保証の実現手段。

## C. 運用・保守性

| 決める項目 | 要件定義に書く内容（What・例） |
|---|---|
| 運用時間・体制 | 運用対応の時間帯、有人/無人、問い合わせ窓口の要否 |
| 監視の要求 | 何の異常を検知し誰に通知すべきか（指標は SLI、詳細は[監視ポリシー](../policy/monitoring-policy.md)） |
| バックアップの要求 | 何を・何世代/何日保持するか（RPO と整合。取得方式は How） |
| 計画停止の許容 | 定期メンテのための停止可否・頻度・時間帯 |
| 更新（パッチ/EOL）方針 | OS・ミドルの更新をどう扱うか、サポート切れ(EOL)の前提（適用手順は How） |
| ログ保持の要求 | 運用ログ・監査ログの保持期間（法令・監査要件と整合） |
| 保守・サポート契約 | 保守窓口・対応時間・製品ライフサイクルの前提 |
| 教育・手順書の要求 | 運用者への教育・マニュアル提供の要否 |

除外（How・設計へ）：監視ツール構成・アラート実装、バックアップ/パッチの具体手順、リモート運用の仕組み。

## D. 移行性

| 決める項目 | 要件定義に書く内容（What・例） |
|---|---|
| 移行の要否・対象 | 既存システム/データから移行するか、対象範囲 |
| 移行データ量 | 移行対象データの規模（切替時間の前提） |
| 移行時期・許容停止 | いつまでに移行するか、切替に許される停止時間（可用性と整合） |
| 移行方式の制約 | 一括/段階、並行稼働の要否など業務都合の制約（具体手順は How） |
| 移行の受入基準 | 移行完了と判断できる条件（件数一致・検証内容） |
| リハーサル・切戻し要求 | リハーサルの要否、切戻し可能性への要求 |

除外（How・設計へ）：移行ツール/スクリプト、ETL 設計、具体的な移行手順。

## E. セキュリティ

| 決める項目 | 要件定義に書く内容（What・例） |
|---|---|
| 準拠すべき規程・法令 | 従うべき基準（個人情報保護法・業界ガイドライン・社内規程等） |
| 情報の格付け | 扱う情報の機密度分類と要求される保護レベル |
| 認証・認可の要求 | 誰がどう認証するか（多要素の要否等の業務要求）、権限区分（方式実装は How） |
| アクセス制限の要求 | 利用できる範囲・場所・端末の制限（業務要求） |
| データ保護の要求 | 暗号化の要求（保存/通信）と秘匿対象（アルゴリズム・鍵管理は How） |
| 追跡・監査ログの要求 | 何を追跡可能にするか、証跡の保持期間 |
| 攻撃対策の要求 | マルウェア/不正アクセス/Web 攻撃への対策要求の水準（製品・構成は How） |
| インシデント対応の要求 | 検知・対応・復旧の体制と目標、届出義務の有無 |
| セキュリティ診断の要求 | 脆弱性診断・ペネトレーションテストの実施要否 |

除外（How・設計へ）：WAF/IDS 等の製品選定、暗号方式・鍵管理設計、FW/ネットワーク構成。

## システム環境・エコロジー（対象外）

クラウド利用を前提とするため対象外。機材設置環境（電源・空調・耐震・重量）や環境負荷は、クラウド事業者の責務であり要件定義に書くことがほぼない。

唯一書く価値があるのは **データの保存場所（リージョン）に関する法令・データ主権の制約**だが、性質はセキュリティ／制約に近いため、独立分野を立てず **E. セキュリティ**または要件定義書の「制約と前提」に寄せる。
