# GSMによるモニタリング項目選定ガイド

GSMフレームワークを使って、監視・アラーム項目を**目的から逆算して**選ぶための手法ガイド。**何を監視すべきか迷っている設計者が、根拠のある項目を体系立てて選びたいとき**に参照する。

> [!NOTE]
> これは [モニタリング・アラームポリシー](../policy/monitoring-policy.md) の「項目は根拠を持って選定し、文書化する」を**実践するための推奨手法の一つ**である。GSMの使用は必須ではない。重要なのは「GSMを使ったか」ではなく「**選定根拠が目的に紐づき、文書化されているか**（設計書またはコード内）」である。

## GSMとは — 目的（Goal）から測る指標（Metric）を逆算する

GSM は Google が提唱した、**Goal → Signal → Metric** の順に考えるフレームワーク。「測れるから測る」ではなく「**目的に紐づくから測る**」を強制するため、ノリで項目を選ぶ事故を防げる。

| 段階 | 問い | 内容 |
|---|---|---|
| **Goal** | 何を達成したいか | システム／機能が果たすべき目的。ユーザー視点の成功状態 |
| **Signal** | 達成／失敗はどう現れるか | Goalの達成・失敗を示す観測可能な兆候（まだ数値ではない） |
| **Metric** | それをどう数値で測るか | Signalを測る具体的な指標・数値 |

> [!TIP]
> **（AI・推奨）** 順番が肝心。**必ず Goal から始める**。Metric から考え始めると「取れるデータ」に引きずられ、目的とずれた指標を量産する（＝ノリの選定）。

## 手順

1. **Goal を一文で書く** — そのシステム／機能の目的を、ユーザー視点の成功状態で書く。
2. **Signal を洗い出す** — その Goal が「達成できている／失敗している」と分かる兆候を挙げる。
3. **Metric に落とす** — 各 Signal を測れる数値指標にする。**なぜその Metric か（根拠）も併せて書く**。
4. **役割を割り当てる** — 各 Metric を「アラート（症状）」「ダッシュボード（原因究明）」のどちらに使うか決める。

## ワークスルー例：決済機能

**Goal**：ユーザーが決済を最後まで完了できる

| Signal（兆候） | Metric（指標） | 選定根拠 | 役割 |
|---|---|---|---|
| 決済が成功して完了する | 決済完了率（成功数 / 試行数） | Goalの失敗を最も直接的に示す症状だから | **アラート**（症状） |
| 決済が許容時間内に終わる | 決済API の p95 レイテンシ | 遅延による離脱の先行指標だから | アラート or ダッシュボード |
| どこで・なぜ失敗したか | 決済失敗のエラー種別内訳 | 原因究明に使う内部情報（ホワイトボックス）だから | ダッシュボード（原因究明） |

> [!TIP]
> **（AI・推奨）** Metric を「アラート」と「ダッシュボード」に振り分ける段階で、ポリシーの **症状ベース** 原則が効く。ユーザー影響を示す症状（完了率・レイテンシ）はアラート、原因を辿るための内訳はダッシュボード、と分ける。

## 出力：選定根拠を文書化して残す

GSM で洗い出した表（Goal / Signal / Metric / 根拠 / 役割）を、そのまま残す。残す場所は設計書が基本だが、CDK等のコードでアラームを定義していて設計書を作らない場合はコード内（コメント等）でもよい。これがポリシーの言う「**根拠を文書化する**」の具体的な成果物になり、レビューの対象になる。
