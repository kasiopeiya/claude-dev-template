# コードレビューガイド

## 対象読者

コードをレビューしていて、**何にコメントすべきか・限られた時間をどこに使うべきか**を判断したい人 / AI エージェント。**観点3〜5（後述の Part B）は `code-review` Skill からも参照される**、人間と AI 共通のレビュー観点である。

## 目的

CI・静的解析・AI レビューが機械的な検査を肩代わりする時代に、**人間レビュワーが限られた時間をどこに向けるべきか**を示す。レビューはスループットのボトルネックになりやすい。機械が担える検査を人間がやり直さず、機械には任せきれない判断に集中することで、レビューの価値を最大化する。

## 役割分担（本ガイドが扱わないこと）

重複を避けるため、次は本ガイドでは**定義し直さない**。中身はリンク先を正典とする。

- レビューコメントのラベル（MUST / WANT / FYI / LGTM）・レビューケースの使い分け・PR サイズ・「なぜレビューするのか」 → [pr-review-policy](../policy/pr-review-policy.md)
- 良いコメントの定義（実装コメントは WHY・doc comment は契約・名前で消せるコメントは名前を直す） → [code-comment-policy](../policy/code-comment-policy.md)
- 良い命名の定義 → [typescript.md「命名規則」](../../.claude/rules/typescript.md)

本ガイドは「**レビューの観点＝何に目を向けるか**」だけに集中する。

レビューの分業を一枚で示すと次のとおり。各担い手が守備範囲を重複なく分担することで、人間の時間を価値の高い判断に振り向ける。

| 担い手 | 守備範囲 |
| --- | --- |
| CI / ESLint・静的解析 | 自動修正・自動検知できるもの（整形・Import 順序・未使用変数など） |
| AI レビュー（`code-review` Skill） | CI が機械化済みの領域（整形・Import 順序・未使用など）はやり直さず、**機械で測れない判断項目**（命名の意味・重複の文脈判断・単一責任・コメント品質・エラー処理・設計）を nit レベルまで網羅的に照合する |
| 人間レビュワー | 機械がやったことをやり直さない。**Part B の観点＋重要なビジネスロジック・抽象レイヤ**に集中する |

## Part A：人間レビュワーの心構え

> Part A は**人間レビュワー向け**である。網羅的・機械的な検査は CI と AI レビューが担う前提で、人間が何をやめ、何に集中するかを示す。

### 1. 自動修正可能な問題をレビュー対象にしない

CI で検査されない領域を中心に見る。ESLint・Prettier・型チェックなど**機械が検知・修正できることは、レビュー前に自動修正で解消しておく**。整形やスタイル規約の違反を人間が指摘するのは、レビュワーの時間の浪費である。

もしルール違反のコードを見つけたら、その場で直して終わりにせず、**仕組みとして自動修正・自動検知できないかを検討し、必要なら issue 化する**（割れ窓の可視化）。コードスタイルの規約違反がレビューで上がってくること自体が、仕組みの不備のシグナルである。レビューで毎回同じ指摘をするより、ゲートを一度作る方が安い。

### 2. すべてのコードをレビューしない

すべての private 関数やメソッドの細部までレビューしていると、時間がいくらあっても足りない。レビューはスループットのボトルネックになりやすいため、**重要度の低い詳細はレビュー対象から外す**勇気を持つ。

優先して見るべきは次の箇所。

| 優先して見る箇所 | なぜ |
| --- | --- |
| 抽象レイヤ（handler・main など、外部との境界・エントリポイント） | 全体の振る舞いを規定し、誤りの影響が広い |
| 重要なビジネスロジック | 壊れたときの影響が大きい中核 |
| テストコード | 仕様の意図が表れ、回帰を守る砦 |

## Part B：レビューの観点

> Part B は**人間レビュワーと AI レビューの共通観点**である。`code-review` Skill からも参照される。

### 3. インターフェースを確認する

ここで言うインターフェースとは、型としての `interface` に限らず、**関数・モジュールが外部に見せる機能の境界**を指す。引数・戻り値の型と意味、責務の境界が適切かという「仕様」を確認することは、レビューで最も価値の高いポイントの一つである。実装の詳細は後で変えられるが、インターフェースの誤りは利用側すべてに波及する。

- 引数・戻り値は、その関数が表す意味に対して過不足ないか
- 利用側が誤用しにくい形になっているか（不正な状態を型で排除できているか）
- 必要以上に optional（`?`）にしていないか。常に渡される引数・必ず存在するプロパティは required にする（未使用の optional は削除する）
- 公開すべきものだけが公開されているか（不要な `export` / `public` を広げていないか）

### 4. コメントの適切さ

コメントが**必要なのに無い**、あるいは**必要以上にある**、もしくは**意味のないコメント**（コードの直訳）になっていないかを見る。コメントは腐る前提で、メンテナンス性を考慮して「変わりにくいもの・WHY」を書けているかを確認する。

良いコメントの定義は [code-comment-policy](../policy/code-comment-policy.md) を正典とする。本観点はそれを**レビューの現場でどう当てるか**に集中する。

### 5. 名前の適切さ

『リーダブルコード』などで繰り返し語られるとおり、**プログラマの仕事は名前を決めることだ**と言われるほど命名は重要である。変数名・関数名が適切なら、コメントはほとんど要らなくなる。レビューでは、名前から意図が読み取れるか、命名に妥協していないかを確認する。

良い命名の定義は [typescript.md「命名規則」](../../.claude/rules/typescript.md) を正典とする。本観点はそれを**レビューの現場でどう当てるか**に集中する。
