# クリーンアーキテクチャ・レビューガイド

クリーンアーキテクチャの思想で書かれた設計書・実装を **レビューするためのガイド**。判断基準（何が良いアーキテクチャか）は [application-architecture-policy](../policy/application-architecture-policy.md) に従い、本ガイドはそれを **レビューの現場でどう当てるか** の手順・観点に集中する。

- **対象読者**：設計書・実装をレビューする人 / AIエージェント
- レビューの進め方・コメントのラベル（MUST/WANT/FYI/LGTM）は [pr-review-policy](../policy/pr-review-policy.md) に従う（本ガイドは「中身」に集中）

---

## レビュー手順（この順で当てる）

1. **依存方向の確認**：依存は内向き（詳細 → 方針）か。逆流していないか。
2. **境界の妥当性**：揺れうる詳細にインターフェースが置かれているか。逆に、安定していて変わらないものへ過剰な境界を引いていないか。
3. **詳細の遅延**：DB・フレームワーク・外部サービスがドメインに侵入していないか。
4. **レイヤー責務の純度**：各層が単一の変更理由に閉じているか。

## チェックリスト（機械的に当てられる粒度）

- [ ] ドメイン層がフレームワーク / DB / 外部SDK を import していない
- [ ] ユースケースに IO・HTTP・SQL の詳細が漏れていない
- [ ] エンティティが永続化アノテーション等の詳細を持たない
- [ ] 依存は抽象（interface）経由で、具象への直接依存がない
- [ ] 「詳細」がテストダブルで差し替え可能になっている
- [ ] ディレクトリ / レイヤー構成がドメインを語っている（フレームワーク名が前面に出ていない）

## 検出すべきアンチパターン（シグナルと例）

- **依存方向の逆流**：ドメイン / ユースケースがインフラ層を import している
- **フレームワークへの直結**：ユースケースがリクエスト型・ORMエンティティをそのまま受け渡している
- **貧血ドメイン + 業務ロジックの漏れ**：エンティティが getter/setter だけで、業務ルールが詳細層に散っている
- **過剰な境界**：変わらないものにまでインターフェースを噛ませ、間接層だけが増えている（[原則4](../policy/refined-engineer-judgment-principles.md) 違反）

## 設計書に貼れる確認テンプレ

レビュー時に上記チェックリストを設計書 / PR に転記して使う。

```text
## アーキテクチャレビュー確認
- [ ] ドメイン層がFW/DB/外部SDKをimportしていない
- [ ] ユースケースにIO・HTTP・SQLの詳細が漏れていない
- [ ] エンティティが永続化等の詳細を持たない
- [ ] 依存は抽象経由で具象への直接依存がない
- [ ] 「詳細」がテストダブルで差し替え可能
- [ ] 構成がドメインを語っている
```
