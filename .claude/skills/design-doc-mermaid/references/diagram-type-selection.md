# 図種の選び方（情報の形から選ぶ）

図種は「ユーザーが使った言葉」ではなく「描こうとしている情報の形」で決める。`flowchart` は他のどの形にも当てはまらなかったときの最後の選択肢であり、既定値ではない。GitHub は下表の全図種を描画する（観測で確認済み）。

対象読者：図を描こうとしているAIが、どの図種を選ぶか決めるとき。

## なぜ言葉ではなく形で選ぶのか

「ワークフローを図にして」と言われて `flowchart` を選ぶのは、語彙のマッチであって設計判断ではない。同じ「ワークフロー」でも、状態が移り変わるなら `stateDiagram-v2`、複数の主体がやり取りするなら `sequenceDiagram`、作業が状態別に並ぶなら `kanban` が正しい。**言葉から選ぶと、ユーザーの語彙のゆらぎがそのまま図種の誤りになる。**

そして `flowchart` はほぼ何でも表現「できてしまう」。できることと適していることは違う。状態遷移を `flowchart` で描くと、状態（ノード）と遷移条件（エッジ）の区別が失われ、読者は「どれが状態でどれが操作か」を本文で確かめる羽目になる。図の目的は本文を読ませないことなので、これは図の失敗である。

## 選択表

「描こうとしている情報の形」の行を上から探し、最初に当てはまったものを使う。

| 情報の形                                         | 図種                 | flowchartで描くとどう劣化するか                                      |
| ------------------------------------------------ | -------------------- | -------------------------------------------------------------------- |
| 複数の主体が時間順にやり取りする                 | `sequenceDiagram`    | 時間軸が消え、誰から誰への呼び出しかを矢印の向きだけで追う羽目になる |
| 1つの主体が状態を移り、遷移に条件がある          | `stateDiagram-v2`    | 状態と遷移条件の区別が失われる                                       |
| 実体と実体の所有・多重度                         | `erDiagram`          | 多重度（1対多・多対多）が表現できず、ただの箱と線になる              |
| 型・継承・インターフェース                       | `classDiagram`       | 継承と参照の区別が消える                                             |
| システム境界と外部アクター                       | `C4Context`          | 「どこまでが自分たちの責任範囲か」の境界が描けない                   |
| 集合の重なり・共通部分・排他                     | `venn-beta`          | 重なりという概念自体を表現できない                                   |
| 階層をそのまま木として見せる                     | `treeView-beta`      | 箱を並べ替えただけになり、階層の深さが読み取りにくい                 |
| 階層＋各要素の量（占有面積で見せる）             | `treemap-beta`       | 量の情報が完全に失われる                                             |
| 数値の比較・推移                                 | `xychart`            | 数値の大小が図から読み取れない                                       |
| 複数軸のスコアを同時に比較する                   | `radar-beta`         | 軸ごとの強弱パターンが一目で比較できない                             |
| 全体に占める割合（単層）                         | `pie`                | 割合が失われる                                                       |
| 流量の分岐・合流（量が保存される）               | `sankey-beta`        | 流れる量の太さが表現できない                                         |
| 日付・期間の軸を持つ                             | `gantt`              | 期間の長短と重なりが消える                                           |
| 出来事を年表として並べる                         | `timeline`           | 時間の間隔が等間隔になってしまう                                     |
| 1つの概念を階層に分解する（発散）                | `mindmap`            | 中心からの発散という構造が直線的な流れに見えてしまう                 |
| 2軸で対象を評価・分類する                        | `quadrantChart`      | 2軸上の位置関係が失われ、ただの分類リストになる                      |
| 結果に対する要因の分解                           | `ishikawa`           | 要因の大分類・小分類の階層が平坦になる                               |
| 作業を状態別のレーンで並べる                     | `kanban`             | どの作業がどの状態にあるかの一覧性が失われる                         |
| ユーザーの体験と感情の起伏                       | `journey`            | 感情のスコアが消える                                                 |
| ブランチ・マージの履歴                           | `gitGraph`           | 分岐と合流の時系列が追えない                                         |
| 要件とその検証の対応関係                         | `requirementDiagram` | 要件・検証という種別の区別が消える                                   |
| インフラ構成要素の接続                           | `architecture-beta`  | クラウドリソースの種別アイコンが使えず、判別に文字を読む必要が出る   |
| メモリ／パケットのビット割り付け                 | `packet-beta`        | ビット位置と幅が表現できない                                         |
| ブロックを格子状に配置する                       | `block-beta`         | 意図した配置が崩れる                                                 |
| **上のどれにも当てはまらない分岐・合流・ループ** | `flowchart`          | — （これが本来の用途）                                               |

## flowchart を選ぶときの条件

`flowchart` / `graph` を選んでよいのは、上表の他の行が**すべて**当てはまらなかったときだけ。選んだ場合は「なぜ他の図種でないか」を一言添える。説明できないなら、選び直す。

```text
（例）flowchart を選んだ理由：状態ではなく処理の分岐であり、主体は1つなので sequence でも state でもない
```

これは形式的な儀式ではない。理由を書こうとすると上表を実際に見ることになり、語彙マッチのショートカットが塞がれる。

## 図を描く前に

図種を選ぶ前に、そもそも図にする価値があるかを [documentation-policy.md](../../../../docs/policy/documentation-policy.md) の逆変換テストで判定する。箇条書きに戻して情報が失われないなら、それは図ではなく箇条書きである。
