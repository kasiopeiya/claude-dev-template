# コードレビュー観点（17項目・51点満点）

各観点を **3点満点** でスコアリングする。

> [!IMPORTANT]
> スコア項目は能動的に探す対象＝レビューの背骨である。`.claude/rules/typescript.md` が定める規約（特に Import 順序）は採点項目として下表に含めること。項目に無い規約違反は素通りしやすいため、スコア化が品質の生命線となる。

> [!TIP]
> **（AI・人間にも）** 人間レビュワーと共通の観点（インターフェース・コメント・名前の妥当性をどう見るか）は [code-review-guide「Part B：レビューの観点」](../../../../docs/guide/code-review-guide.md) を参照。本採点表（17項目）はそのまま用いる。

## 共通観点（17項目）

| #   | 観点                       | チェック内容                                                                                                          | スコア基準                                                                  |
| --- | -------------------------- | --------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| 1   | **型安全性**               | `any` の使用、型アサーション、型推論の活用                                                                            | 3点: `any` なし / 2点: 一部 `any` / 1点: 多数の `any` や型アサーション乱用  |
| 2   | **命名規則・変数名の妥当性** | 変数名・関数名・定数名の明確性・一貫性。利用目的や用途が名前から想像できるか（❌ `mainFunc` ⭕ `accountFunc`）        | 3点: 一貫性があり意図が明確 / 2点: 一部不明瞭 / 1点: 多数の不適切な命名     |
| 3   | **関数の長さ**             | 適切な粒度、単一責任の原則                                                                                            | 3点: 50行以下 / 2点: 50〜100行 / 1点: 100行超                               |
| 4   | **重複コード**             | DRY 原則の遵守                                                                                                        | 3点: 重複なし / 2点: 軽微な重複 / 1点: 顕著な重複                           |
| 5   | **単一責任**               | 関数・モジュールの責務の明確性。**責務を一文で言い切れるか（説明に "and"／「〜と〜」が必要なら責務過多のシグナル＝分割を検討する）**を判定軸にする | 3点: 各単位の責務を一文で言える / 2点: やや複数の責任（一文に "and" が混ざる） / 1点: 明らかに複数の責任 |
| 6   | **引数の数**               | 推奨: 3つ以下                                                                                                         | 3点: 3つ以下 / 2点: 4つ / 1点: 5つ以上                                      |
| 7   | **未使用・不要にoptionalな宣言** | 宣言が実際の使われ方と一致しているか。①参照されない変数・インスタンス変数（`private readonly` など）・関数引数・プロパティ（**optional `?` 付きは未使用が見逃されやすいので必ず対象に含める**）、②常に渡される／必ず存在するのに `?` を付けた**不要な optional** がないか | 3点: いずれもなし / 2点: 軽微にあり / 1点: 明らかな未使用または不要optionalが複数 |
| 8   | **マジックナンバー**       | ハードコード数値の定数化                                                                                              | 3点: ハードコードなし / 2点: 一部ハードコード / 1点: 多数のハードコード     |
| 9   | **コメント**               | WHY コメントの存在、自明なコードへのコメント過多を避ける。**モジュール冒頭の責務コメント（一文）の有無と妥当性（ファイル名の言い換えでなく境界・スコープを語っているか。`.claude/rules/typescript.md`）** | 3点: WHY コメント適切かつ責務コメントあり / 2点: 一部不足または過多 / 1点: コメントなし or 過多 |
| 10  | **エラーハンドリング**     | 適切なエラー処理・エラーメッセージ                                                                                    | 3点: 適切なエラー処理 / 2点: 一部不足 / 1点: エラー処理なし                 |
| 11  | **セキュリティ**           | ハードコード・環境変数の適切な扱い・インジェクション対策                                                              | 3点: リスクなし / 2点: 軽微なリスク / 1点: 重大なリスク                     |
| 12  | **非同期処理**             | Promise・async/await の適切な使用                                                                                     | 3点: 適切 / 2点: 一部改善余地 / 1点: 非同期処理の問題あり                   |
| 13  | **tryブロックの長さ**      | try ブロック内のコード行数。長い try ブロックは例外発生箇所の特定を困難にし、意図しない例外の捕捉につながる           | 3点: try ブロック20行以下 / 2点: 20〜40行 / 1点: 40行超（減点）              |
| 14  | **ネスト深度**             | if / try / for / while / switch などの制御フローの最大ネスト深度。3重以上のネストは可読性・保守性を著しく低下させる。**加えて、ネスト数を見た目だけ減らす小手先の回避（ネスト逃れの `.then`/`.catch` チェーン化、多段三項など）も検出する。ネスト違反は設計改善のシグナルであり、正しい対処は責務分割・早期return。複雑さを温存したまま見た目だけ平坦化したものは違反として扱う（`.claude/rules/typescript.md`「ネスト制限は『設計を見直せ』というシグナル」）** | 3点: 最大2重以下かつ小手先回避なし / 2点: 一部3重ネスト、または軽微な小手先回避あり / **1点: 3重以上のネスト、または明確な小手先回避が1箇所でもある（大減点）** |
| 15  | **不要なexport**           | モジュール外から参照されていない関数・変数・クラスに `export` が付いていないか。外部参照されるものだけに `export` を付けることで、モジュールの公開インターフェースを明確にする | 3点: 不要な export なし / 2点: 軽微な不要 export あり / 1点: 不要な export が複数ある |
| 16  | **不要なpublic公開**       | クラスのメンバ変数・メソッドに `public`（または修飾子なし）が付いているが、クラス外から参照されていないか。外部から使われないメンバは `private` または `protected` にすること | 3点: 不要な public なし / 2点: 軽微な不要 public あり / 1点: 不要な public が複数ある |
| 17  | **Import 順序・配置**      | `.claude/rules/typescript.md` 準拠。①標準 → ②サードパーティ → ③自作の順か、グループ間が空行区切りか。**かつ全 import がファイル冒頭に集約され、途中 import がないか** | 3点: 完全準拠 / 2点: 一部順序違反・グループ未分離 / 1点: 順序が乱れている、または途中 import がある |

## スコア対象外の追加チェック（確認点として報告）

スコアには含めないが、レビュー時に必ず確認し「確認点」セクションとして報告する。

| #   | 観点                   | チェック内容                                                                 |
| --- | ---------------------- | ---------------------------------------------------------------------------- |
| A   | **設定値の妥当性**     | タイムアウト・保存期間・リトライ回数などの設定値が要件に合っているか         |
| B   | **ログレベルの適切さ** | ログの出力レベル（info/warn/error）が状況に合っているか                      |

## 総合レベル判定

| スコア   | レベル                                  |
| -------- | --------------------------------------- |
| 47点以上 | 優秀（Excellent）                       |
| 36〜46点 | 良好（Good）                            |
| 26〜35点 | 要改善（Needs Improvement）             |
| 26点未満 | 要リファクタリング（Needs Refactoring） |

## 種別ごとの重点観点

| 種別             | 重点観点                                     |
| ---------------- | -------------------------------------------- |
| Handler          | セキュリティ・エラーハンドリング・非同期処理 |
| Utility          | 型安全性・単一責任・テスタビリティ           |
| Page / Component | コンポーネント設計・命名規則・単一責任       |
| Context          | 型安全性・状態管理・パフォーマンス           |
| Module           | 汎用的な11観点をバランスよく評価             |
