# 立ち上げで決める基盤技術チェックリスト

`/decide-tech-stack` が対話で潰す項目のリファレンス。判定基準と、決める項目・あえて保留する詳細を定義する。

## 判定基準（決めるか、遅らせるか）

各技術項目を、ただ一つの問いで仕分ける：

> それを決めないと、開発の足場（静的解析・CI・デプロイ）と最初の曳光弾パイプラインが組めないか？

- Yes → **基盤**。今このSkillで決める（下の表）。後から変えるとリポジトリ全体に波及する。
- No（曳光弾ではスタブで代用できる）→ **詳細**。あえて遅らせる（`docs/policy/new-development-policy.md` §3・`docs/policy/application-architecture-policy.md` §1・原則2）。

## 決める：基盤チェックリスト

一項目ずつ判定しながら潰す。要件・組織標準で確定済みのものは問い直さない（原則2の較正：確定済みは遅らせない）。

| 項目 | 例 | なぜ足場に効くか |
|---|---|---|
| 言語 / ランタイム | TypeScript / Node.js | これが無いとコードも静的解析も書けない |
| パッケージマネージャ | npm / pnpm | 依存解決・CI キャッシュの前提 |
| クラウドプロバイダ | AWS / GCP | デプロイ先。IaC・CI/CD・サービス選定を縛る |
| コンピュート / 実行形態 | Lambda / ECS / EC2 / K8s | クラウドを選んでも実行形態が未定だと IaC・デプロイ・CI が組めない。プラットフォームアダプタの構造も左右する |
| IaC ツール | AWS CDK / Terraform | インフラ定義とデプロイの土台 |
| CI/CD プラットフォーム | GitHub Actions | 静的解析・テスト・デプロイの実行基盤 |
| 静的解析 / フォーマッタ | ESLint / Prettier / Biome | 足場そのもの。最初に組む |
| テストランナー | Vitest / Jest | CI で回すテストの土台 |
| リポジトリ / ディレクトリ構成方針 | monorepo / package by feature | 後から変えると全ファイル移動。`docs/guide/directory-structure-guide.md` に従う |

「ほかにも」候補は、判定基準（足場が組めるか）に照らして都度追加・除外する。基盤に昇格する項目はプロジェクトによって変わる。

## あえて保留：詳細（曳光弾でスタブ）

足場を組むのに不要で、最初の曳光弾ではスタブで代用できるもの。決めた気になって前倒し固定しない。

- 具体的な Web / API フレームワーク
- ORM / DB クライアント、DB スキーマ設計
- 具体的な外部サービス連携（決済・通知 等）
- 個別ライブラリの選定

これらは「未決定」ではなく**戦略的な保留**として、初期技術スタックADR に明記する。実際に必要になった縦スライスで、そのとき選ぶ。
