# PRR レポートフォーマット

PRR の結果は次の様式で出力する。判定基準・領域の定義は `readiness-rubric.md` に従う。

```
## PRR レポート — 本番投入レディネス

### 🎯 対象
- **スコープ**: <システム全体 / 指定された対象>
- **実行日**: <YYYY-MM-DD>
- **前提の充足**: <design-hub / requirements の有無など、審査の前提状況>

### 🚦 総合判定
- **判定**: **go / 条件付きgo / no-go / 審査不能（全領域 N-A・前提未整備）**
- **一言サマリ**: <なぜその総合判定か。fail や warn の核心を1〜2文で>
- **最終判断は人間が下す**（本レビューは助言）。

### 📊 領域別評価

| # | 領域 | 判定 | 根拠 | 不足・条件 |
|---|---|---|---|---|
| ① | ドキュメント整備 | pass/warn/fail/N-A | <見た対象と判断理由> | <解消すべき点・N-A の理由> |
| ② | リリース/ロールバック経路 | ... | ... | ... |
| ③ | 監視・アラート | ... | ... | ... |
| ④ | 性能・スケール限界 | ... | ... | ... |
| ⑤ | セキュリティ | ... | ... | ... |
| ⑥ | 回復性/DR | ... | ... | ... |
| ⑦ | 既存レビューの実施状況 | ... | ... | ... |

### ⚠️ 本番投入前に解消すべきこと（no-go / 条件付きgo のとき）

<fail は必須解消、warn は解消 or リスク受容の判断を、優先度順に列挙する。go のときは「なし」。>

1. **<領域・項目>**: <何を・なぜ解消すべきか。具体的なアクション>
2. ...

### 🔍 機械確認の結果
- **CI（静的解析・単体テスト）**: <green / failing / 未確認>（根拠：<確認したコマンド・GitHub 状態>）
- **結合テスト**: <pass / fail / 未確認>
- **PR ラベル**: <期待ラベルの有無>

### 🧑 人間 attest の記録
- **`/code-review` 実施**: <ユーザー回答をそのまま記録>
- **`/cdk-review` 実施**: <同上（CDK 変更がある場合）>
- **`/arch-review` 実施**: <同上（マクロ構造に影響する変更がある場合）>

### 📐 本レビューの限界（静的審査・必須注記）
- 本 PRR はコード・ドキュメント・CI/GitHub 状態の**静的確認**である。ライブ AWS 環境の実デプロイ状態・アラームの実発報・実負荷下の挙動は**検査対象外**（結合テスト・実運用の領域）。「コードにアラーム定義がある」ことは確認できても「本番で実際に発報する」ことは保証しない。

---

**PRR 完了**
```

## 出力の原則

- **根拠を必ず添える。** 「pass」だけ書いて理由を書かない領域を作らない（`readiness-rubric.md` の必須ルール）。
- **`N-A` にも理由を書く。** なぜ該当しないか・どの前提が未整備かを明示する。
- **正直に振る舞う。** 静的審査で確認できないことを「確認済み」と書かない。確認できなければ「未確認」と明記する。
